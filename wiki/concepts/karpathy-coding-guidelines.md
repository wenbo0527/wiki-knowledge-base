## 摘要

Andrej Karpathy 提出的 LLM 编码行为指南，通过四大核心原则减少 LLM 在编码过程中的常见错误，提高代码质量和执行效率。

---

## 核心原则

### 1. Think Before Coding

**不要假设。不要隐藏困惑。呈现权衡。**

在实现之前：
- 明确陈述你的假设。如果不确定，请问。
- 如果存在多种解释，全部呈现——不要沉默地选择其中一个。
- 如果存在更简单的方法，指出它。在合理时反驳。
- 如果有不清楚的地方，停下来。说出哪里不清楚。提问。

### 2. Simplicity First

**最少代码解决问题。不做投机性设计。**

- 不添加超出需求的功能。
- 不为单次使用的代码创建抽象。
- 不添加未请求的"灵活性"或"可配置性"。
- 不为不可能的场景添加错误处理。
- 如果你写了 200 行而它可以是 50 行，重写它。

### 3. Surgical Changes

**精准修改，只改必须改的。只清理自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有坏掉的东西。
- 匹配现有风格，即使你会有不同做法。
- 如果注意到无关的死代码，指出它——不要删除它。

当你的更改造成孤立文件时：
- 移除你的更改造成的未使用的导入/变量/函数。
- 不要删除预先存在的死代码，除非被要求。

**测试标准**：每一行更改都应该直接追溯到用户的请求。

### 4. Goal-Driven Execution

**定义成功标准。循环验证直到完成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写复现它的测试，然后让它通过"
- "重构 X" → "确保测试在重构前后都通过"

对于多步骤任务，简述计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

## 在数据社区项目中的应用

### 代码审查检查清单

- [ ] 是否在实现前明确了假设？
- [ ] 是否有更简单的解决方案？
- [ ] 代码是否只做了必须做的？
- [ ] 更改是否精确、只改必要的部分？
- [ ] 是否有可验证的成功标准？
- [ ] 是否添加了投机性功能？

### 任务执行模板

```
任务：[描述]
假设：[列出假设]
成功标准：[可验证的检查点]
计划：
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
```

---

## 评估标准

**这些指南有效的前提是：**
- diff 中不必要的更改更少
- 因过度复杂而重写更少
- 澄清问题在错误之前而非之后提出
