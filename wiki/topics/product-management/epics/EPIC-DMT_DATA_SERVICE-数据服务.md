## 1. 产品总览

### 1.1 一句话定位
**将企业内部数据资产封装为标准化 API 服务，实现数据能力的快速复用和开放共享。**

### 1.2 产品定位 & 核心价值

|:---|:---|
| **目标用户** | 业务系统开发者、数据工程师、数据管理员、合作伙伴 |
| **核心痛点** | 数据获取依赖人工协调、接口标准不统一、数据安全问题难以保障 |
| **解决方案** | 自助 API 发布 + 标准化接口管理 + 权限和配额控制 + 服务监控 |

## 2. 功能结构（按模块归档）

### 2.1 FEATURE-DMT_SERVICE_PUBLISH（服务发布）

#### 功能入口 & 作用

**入口位置**：数据管理 → 数据服务 → 发布服务

**核心作用**：
- 自助发布数据为 API 服务
- 可视化配置 API 参数
- 自动生成 API 文档
- 支持 SQL 和脚本编写

#### 核心逻辑

```
服务发布流程：

Step 1: 选择数据源
  - 选择数据库或数据表
  - 选择查询类型（单表查询、SQL 查询）
  - 权限和数据范围校验

Step 2: 配置 API
  - 定义请求参数
  - 定义响应结构
  - 配置分页逻辑
  - 配置缓存策略

Step 3: 安全配置
  - 数据脱敏规则
  - 访问权限配置
  - IP 白名单
  - 签名校验

Step 4: 测试验证
  - 在线 API 测试
  - 性能基准测试
  - 数据准确性验证

Step 5: 审批上线
  - 提交服务审批
  - 审核服务配置
  - 配置完成后上线
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---|
| `service.publish_approval` | 是否需要审批 | `true` |
| `service.cache_enabled` | 默认启用缓存 | `false` |
| `service.timeout` | 默认超时时间（ms）| `3000` |
| `service.max_page_size` | 最大分页大小 | `100` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **SQL 注入风险** | 用户输入直接拼接 SQL | 参数化查询、输入校验 |
| **性能问题** | 复杂查询导致超时 | 强制超时、限制复杂度 |
| **数据泄露** | 敏感字段未脱敏 | 默认脱敏、人工审核 |
| **版本管理混乱** | 多版本并存难以管理 | 强制版本号、版本废弃策略 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 元数据中心 | 上游 | 数据源和字段信息 |
| 权限中心 | 依赖 | 权限校验 |
| API 网关 | 依赖 | API 部署 |

### 2.2 FEATURE-DMT_SERVICE_GATEWAY（API 网关）

#### 功能入口 & 作用

**入口位置**：数据管理 → 数据服务 → 网关管理

**核心作用**：
- 统一 API 入口和路由
- 请求鉴权和签名验证
- 限流和熔断保护
- 调用监控和告警

#### 核心逻辑

```
API 网关流程：

Step 1: 请求接入
  - HTTPS 接入
  - 负载均衡
  - SSL 终止

Step 2: 鉴权校验
  - Token 验证
  - 签名校验
  - 权限校验

Step 3: 流量控制
  - 限流（QPS/配额）
  - 熔断降级
  - 排队等待

Step 4: 路由转发
  - 路径匹配
  - 服务发现
  - 负载均衡

Step 5: 监控记录
  - 请求日志
  - 性能指标
  - 告警触发
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `gateway.rate_limit` | 全局限流 QPS | `10000` |
| `gateway.timeout` | 网关超时（ms）| `10000` |
| `gateway.circuit_threshold` | 熔断错误率阈值 | `50%` |
| `gateway.max_queue` | 最大排队数 | `100` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **限流过严** | 正常请求被限流 | 动态调整、差异化限流 |
| **熔断误触发** | 短暂异常导致熔断 | 增加熔断延迟、友好错误提示 |
| **日志量过大** | 高并发下日志存储压力大 | 采样日志、异步写入 |
| **签名验证复杂** | 开发者接入成本高 | 提供 SDK、简化签名方案 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 服务发布 | 上游 | API 服务来源 |
| 权限中心 | 上游 | 鉴权信息 |
| 监控系统 | 下游 | 指标数据 |

## 5. 规则与配置中心

### 5.1 API 安全规则

| 规则类型 | 规则内容 |
|:---|:---|
| **认证方式** | API Key / OAuth 2.0 / JWT |
| **签名算法** | HMAC-SHA256 |
| **传输加密** | HTTPS 强制 |
| **IP 白名单** | 可选配置，默认不限 |
| **调用来源** | Web / App / Server / Partner |

### 5.2 限流规则

```
限流维度：

1. 全局限流
   - 总 QPS 限制
   - 总并发限制

2. 应用限流
   - 单应用 QPS
   - 单应用日调用量

3. 接口限流
   - 单接口 QPS
   - 单接口并发

默认配额：
- 测试环境：无限制
- 生产环境：每应用每天 10000 次
- 超限处理：返回 429 错误
```

### 5.3 熔断规则

```
熔断配置：

触发条件：
- 错误率 > 50%（时间窗口 10s）
- P99 延迟 > 2000ms
- 连续超时 5 次

熔断行为：
- 快速失败，返回友好错误
- 触发告警通知
- 定时探测下游健康状态

恢复条件：
- 探测成功 3 次
- 熔断时间 > 30s
```

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: API 调用返回 401 怎么办？

**排查步骤**：
1. 检查 Access Key 是否正确
2. 检查签名算法是否正确
3. 检查时间戳是否过期

**解决方案**：
- 重新获取 Access Key
- 使用官方 SDK 生成签名
- 检查请求时间是否偏差过大

#### Q: API 调用返回 429 怎么办？

**排查步骤**：
1. 检查是否超过配额限制
2. 查看限流规则配置

**解决方案**：
- 申请提升配额
- 优化调用频率
- 增加缓存减少调用

#### Q: API 响应很慢怎么办？

**排查步骤**：
1. 检查下游数据源响应时间
2. 查看是否触发限流或熔断
3. 检查网络延迟

**解决方案**：
- 增加缓存
- 优化查询逻辑
- 联系管理员排查

## 8. 方法论沉淀

### 8.1 数据服务设计原则

```
服务设计原则：

1. 单一职责
   - 每个服务只做一件事
   - 避免功能耦合

2. 高内聚低耦合
   - 服务间通过 API 交互
   - 避免直接数据库依赖

3. 幂等性
   - 同一请求多次调用结果一致
   - 通过唯一请求 ID 保证

4. 可观测性
   - 请求日志完整记录
   - 关键指标可监控
   - 错误可追溯

5. 版本管理
   - API 版本号管理
   - 废弃策略明确
   - 向后兼容
```

### 8.2 API 最佳实践

**接口设计规范**：
```
RESTful 风格：
- GET：查询
- POST：创建
- PUT：更新
- DELETE：删除

路径规范：
- /users/{id}
- /users/{id}/orders

参数规范：
- 路径参数：用于资源标识
- 查询参数：用于筛选和分页
- 请求体：用于复杂输入

响应规范：
- 统一响应结构
- 明确状态码
- 友好错误信息
```

### 8.3 工具流程

**数据服务生命周期**：
```
设计 → 开发 → 测试 → 上线 → 监控 → 优化 → 下线

工具链：
- 设计：API 设计工具（Swagger）
- 开发：IDE + SDK
- 测试：Postman + 自动测试
- 上线：CI/CD + 容器化
- 监控：Prometheus + Grafana
- 优化：APM 工具
```
