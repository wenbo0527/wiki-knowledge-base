## 核心方法论

### SDD-RIPER框架

SDD-RIPER 是团队级AI编程落地的核心框架：

```
┌─────────────────────────────────────────────────────────────┐
│              SDD-RIPER 框架                                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  SDD = 持久化的大模型上下文与记忆层                         │
│         用来给模型提供长期一致的轨道、边界和注意力聚焦       │
│                                                              │
│  RIPER = 审批驱动的状态机                                   │
│         用来约束模型的执行顺序、推进权限与单步粒度            │
│         确保只有在用户 review/approve 后才进入下一阶段        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### RIPER状态机

```
┌─────────────────────────────────────────────────────────────┐
│                    RIPER 状态流转                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│    Request → Plan → Execute → Perform → Review              │
│       ↑                                              │       │
│       └──────────────────────────────────────────────┘       │
│                                                              │
│  核心规则: 未经Plan Approved不得改代码                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### SDD核心组件

```
┌─────────────────────────────────────────────────────────────┐
│                    SDD 三层上下文                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Layer 1: Spec (任务级真相源)                              │
│  - 这次要做什么、边界是什么、风险是什么、验证方式           │
│                                                              │
│  Layer 2: CodeMap (单项目索引)                             │
│  - 项目内部核心模块、关键链路、入口文件、依赖关系            │
│                                                              │
│  Layer 3: ProjectMap (多项目索引)                          │
│  - 多个项目之间的关系                                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 四大工程痛点

### 痛点一：上下文腐烂
- **问题**: 对话越长，AI越容易遗忘前文约束，悄悄破坏已有逻辑
- **本质**: 模型架构的固有局限，无法靠"更强的模型"解决
- **案例**: AI第5轮改了一个函数，第15轮又改回去了

### 痛点二：审查瘫痪
- **问题**: AI秒生成500行代码，人根本Review不过来
- **本质**: 传统逐行Code Review在AI编程时代彻底失效
- **后果**: 产出速度远超审查速度，质量失控

### 痛点三：维护断层
- **问题**: 全是AI生成的陌生代码，两周后不敢动，改一行崩三处
- **本质**: 没有人知道"AI当时为什么这么写"，代码变成黑盒

### 痛点四：代码不信任
- **问题**: 不知道AI为什么这么写，不敢上线、不敢重构
- **本质**: 信任缺失是最大的推广障碍

| 研发人力释放 | - | 显著 | ⬆️ |

## 三大核心场景

### 场景一：研发提效
- **新业务快速交付**: 需求周期压缩
- **老业务随时可回**: Spec + CodeMap让上下文可持久化
- **效果**: 半年没碰的老项目，20分钟就能重新进入状态

### 场景二：人力解耦
- **核心研发释放**: 老业务沉淀Spec + CodeMap + 约束文档
- **低经验同学稳定交付**: 在RIPER流程约束下可控交付
- **Bug率不升反降**: 18%-37%

### 场景三：知识不随人走
- **交接变简单**: 读Spec → 按图施工
- **人员轮换无忧**: 上下文不丢失，质量不滑坡
- **适用范围**: 一人负责多个项目、跨团队借调

## Spec首先是给人的

> 不要只盯着Spec对大模型的影响。Spec首先是给人看的工程文档。

| 受益者 | 没有Spec的代价 | 有Spec的价值 |
|--------|---------------|-------------|
| 开发者自己 | 2天翻代码猜逻辑 | 20分钟读Spec进入状态 |
| 接手同事 | 口头交接风险高 | 接入周期周→天 |
| TL/主管 | 靠口头汇报风险暴露 | 打开Spec知进度/决策/风险 |
| 组织 | 核心人力离职=黑盒 | 知识沉淀，组织资产 |
| 大模型 | 上下文腐烂 | 注意力聚焦，防遗忘 |

## 复杂项目适配

### 常见误解
> "我们业务太复杂、项目太多，这套方法很难落地"

### 正确认知
**项目越复杂，越需要SDD-RIPER**：
- 小项目 → 靠个人经验、临时对话就能推进
- 复杂业务 → 单靠聊天记录、口头同步反而最容易失控

### 真正的难点
不是"代码太多"，而是三件事没被工程化：
1. **没有稳定边界** — 不知道任务该改到哪里为止
2. **没有统一索引** — 每次都要重新扫仓、认路、猜链路
3. **没有协作真相源** — 人和模型都在依赖临时上下文

## 治理抓手

### 核心规则（先建立这一条就够了）
> **未经Plan Approved不得改代码**

### 扩展规则
- 每次修改都有决策留痕
- 有Plan审批
- 有Review闭环
- 质量可追溯
