## 🎯 核心方法论：四层架构

```
┌─────────────────────────────────────────────────────────┐
│  第零层：业务需求入口层                                   │
│  - 业务方原始需求收集                                     │
│  - 初步分类与优先级判断                                   │
│  - 进入 AI 拆解队列                                       │
├─────────────────────────────────────────────────────────┤
│  第一层：AI 需求智能拆解层                                │
│  - 自然语言理解                                           │
│  - 自动拆解为 Epic/Feature/Story                          │
│  - 验收标准自动生成                                       │
├─────────────────────────────────────────────────────────┤
│  第二层：本体论语义层                                     │
│  - 统一术语定义                                           │
│  - 实体类型与关系定义                                     │
│  - 约束规则与推理规则                                     │
├─────────────────────────────────────────────────────────┤
│  第三层：图数据库 + 向量数据库层                          │
│  - Neo4j 图数据库：存储产品层级结构                       │
│  - 向量数据库：语义检索与相似度匹配                       │
│  - RAG 增强问答                                           │
├─────────────────────────────────────────────────────────┤
│  第四层：元数据与数据源层                                 │
│  - 飞书多维表格：需求状态同步                             │
│  - 代码仓库：技术实现追踪                                 │
│  - 产品文档：PRD 与设计稿                                 │
└─────────────────────────────────────────────────────────┘
```

## 📋 需求分层标准

### Epic（史诗）

**定义**：一个完整的业务模块或产品域

**特征**：
- 可以独立交付的业务价值单元
- 通常对应一个产品域或子系统
- 包含多个 Feature
- 有明确的业务目标和成功指标

**命名规范**：`EPIC-{DOMAIN}_{MODULE}`

**示例**：
- `EPIC-DFD_DATA_ASSET` - 数据资产字典
- `EPIC-MKT_CROWD_CENTER` - 客群中心
- `EPIC-RISK_EXT_DATA_LIFECYCLE` - 外部数据生命周期管理

### Feature（功能）

**定义**：Epic 下的具体功能模块

**特征**：
- 功能内聚的模块
- 可以独立完成开发和测试
- 包含多个 Story
- 有明确的业务逻辑边界

**命名规范**：`FEATURE-{EPIC}_{FUNCTION}`

**示例**：
- `FEATURE-DFD_ASSET_OPERATE_TOOL` - 数据资产运营工具
- `FEATURE-MKT_CROWD_CREATE` - 客群创建

### Story（用户故事）

**定义**：用户可感知的完整功能点

**特征**：
- 用户视角描述（"作为...我想要...以便..."）
- 可在一个迭代内完成
- 有明确的验收标准
- 可独立演示和验收

**命名规范**：`STORY-{FEATURE}_{ACTION}`

**示例**：
- `STORY-ASSET_OPERATE_CREATE_PAGE` - 资产创建页面
- `STORY-CROWD_CREATE_WIZARD` - 客群创建向导

## 🔄 Epic 拆解流程

```
┌─────────────────────────────────────────────────────────┐
│ Step 1: 范围确认                                          │
│ - 阅读产品 PRD 和需求文档                                 │
│ - 查看相关前端代码目录                                    │
│ - 确认 Epic 范围和边界                                    │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│ Step 2: Feature 结构重建                                  │
│ - 清理 Neo4j 中现有 Feature/Story                         │
│ - 基于 PRD 和代码重新拆解 Feature 结构                    │
│ - 创建 Feature 节点                                       │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│ Step 3: Story 拆解                                        │
│ - 每个 Feature 拆解为 Story                               │
│ - 为每个 Story 生成验收标准                               │
│ - 创建 Story 节点                                         │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│ Step 4: 飞书表格同步                                      │
│ - 创建飞书多维表格                                        │
│ - 写入 Story 数据                                         │
│ - 生成表格链接                                            │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│ Step 5: 验证与复盘                                        │
│ - 对比 PRD 确认完整性                                     │
│ - 对比代码确认准确性                                      │
│ - 更新 HEARTBEAT.md                                       │
└─────────────────────────────────────────────────────────┘
```

## ✅ Story 验收标准规范

每个 Story 验收标准包含 4 项：

| 标准 | 说明 | 示例 |
|------|------|------|
| **功能可正常使用** | 核心功能完整 | 数据资产列表页可正常加载并显示数据 |
| **相关操作正常** | CRUD 操作无误 | 新增/编辑/删除/查询资产操作正常 |
| **数据准确完整** | 数据流转正确 | 资产字段、状态、权限显示准确 |
| **交互符合预期** | UI/UX 符合设计 | 页面布局、样式、交互效果符合 PRD |

## 🎨 工作原则

### 1. 聪明高效

- 用最聪明的方式解决问题
- 自动化重复工作
- 工具链最大化利用

### 2. 结果导向

- 关注结果，过程可以灵活
- 交付可衡量的业务价值
- 数据驱动决策

### 3. 快速迭代

- 先做出来，再优化
- 小步快跑，频繁验证
- 持续交付，持续改进

### 4. 数据驱动 + 直觉

- 用数据说话，但也要有直觉
- 定量和定性结合
- 经验和创新并重

## 📝 文件命名规范

```
格式: YYYY-MM-DD-[类型]-[标题]-[版本].md

示例:
- 2026-03-12-PRD-营销画布优化-v1.0.md
- 2026-03-12-周报-W09-v1.0.md
- 2026-03-12-需求-用户中心重构-v0.5-draft.md
```

### 版本管理

| 版本标识 | 含义 | 存储位置 |
|----------|------|----------|
| v0.x-draft | 草稿版本 | 仅本地过程文件 |
| v0.x-wip | 进行中 | 本地 + 飞书（工作文件夹）|
| v1.0+ | 正式发布 | 本地 + 飞书（正式文件夹）|
