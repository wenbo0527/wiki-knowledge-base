## 1. 产品总览

### 1.1 一句话定位
**企业级权益资产统一管理平台，支持权益的创建、配置、发放和核销，为营销活动提供灵活的权益支撑。**

### 1.2 产品定位 & 核心价值

| 维度 | 描述 |
|:---|:---|
| **目标用户** | 营销运营人员、CRM 管理员、业务人员 |
| **核心痛点** | 权益分散在不同系统难以统一管理、发放渠道不统一、核销数据不透明 |
| **解决方案** | 权益中心统一管理 + 多渠道发放 + 全链路核销追踪 |
| **核心价值** | 权益管理效率提升 80%，核销率提升 20%，运营成本降低 15% |

### 1.3 核心业务流程

```

│  权益创建                                                        │
│  - 选择权益类型（优惠券、积分、实物、虚拟权益）                  │

│  - 设置权益库存和发放上限                                        │
└─────────────────────────────────────────────────────────────────┘
                              │

┌─────────────────────────────────────────────────────────────────┐
│  权益发放                                                        │
│  - 手动发放（运营人员指定用户发放）                              │
```

### 1.4 核心角色 & 使用人群

| 角色 | 核心职责 | 使用场景 | 高频功能 |
|:---|:---|:---|:---|
| **营销运营** | 创建和配置权益、追踪核销效果 | 活动前配置权益、活动后分析效果 | 权益创建、发放记录查看 |
| **CRM 管理员** | 管理权益规则和库存 | 维护权益资产库 | 权益上下架、库存调整 |

### 1.5 关键约束

| 约束类型 | 具体约束 | 应对策略 |
|:---|:---|:---|
| **数据一致性** | 权益库存必须强一致，防止超发 | 分布式锁 + 库存预占机制 |
| **发放性能** | 单次批量发放可达 10 万级别，30 分钟内完成 | 消息队列异步处理、分批执行 |
| **核销时效** | 核销响应 < 200ms，不影响主交易流程 | 本地缓存 + 异步写入 |
| **合规性** | 权益发放需符合监管要求（个人信息保护）| 脱敏处理、知情同意校验 |
| **渠道兼容性** | 支持多端（App/H5/小程序/PC）核销 | 统一权益码 + 渠道适配层 |

### 1.6 术语词典（本体论级别）

| 术语 | 英文 | 定义 | 示例 |
|:---|:---|:---|:---|

| **权益中心** | Benefit Center | 权益的创建、发放、核销、分析的统一管理平台 | 运营在权益中心配置满减券 |

## 2. 功能结构（按模块归档）

### 2.1 FEATURE-MKT_BENEFIT_ASSET_MGMT（权益资产管理）

#### 功能入口 & 作用

**入口位置**：
- 权益中心 → 权益资产管理
- 营销工具箱 → 权益管理

**核心作用**：
- 统一管理所有类型的权益资产
- 支持权益的增删改查、上下架
- 配置权益的发放规则和使用条件
- 管理权益批次和库存

#### 核心逻辑

```
权益资产管理流程：

Step 1: 权益类型定义
  - 选择权益模板（优惠券/积分/实物/虚拟权益）
  - 配置权益基本信息（名称、描述、图标）
  - 设置权益价值（面值、兑换比例）

Step 2: 权益规则配置
  - 发放规则：谁可以领、领多少、怎么发
  - 使用规则：满多少可用、限什么商品、叠加规则
  - 有效期规则：固定日期、领取后 N 天、滚动有效期

Step 3: 批次管理
  - 创建权益批次，生成唯一批次号
  - 设置批次库存和发放上限
  - 批次状态管理（待生效/生效中/已结束/已作废）

Step 4: 库存管理
  - 实时库存监控和预警
  - 库存锁定（预占）和释放
  - 库存调拨（跨批次）
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `benefit.default_validity_days` | 默认有效期天数 | `30` |
| `benefit.max_issue_per_user` | 单用户累计发放上限 | `10` |
| `benefit.stock_warning_threshold` | 库存预警阈值 | `10%` |
| `benefit.auto_shelf.enabled` | 库存为 0 自动下架 | `true` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **超发问题** | 并发情况下库存扣减不准确，导致超发 | 使用 Redis 原子操作或数据库乐观锁 |
| **批次状态混乱** | 多批次权益叠加使用导致优惠叠加过度 | 互斥规则引擎、叠加优先级配置 |
| **库存数据不一致** | 多系统读取库存不一致 | 统一库存服务、事件驱动同步 |
| **权益模板复用困难** | 每次创建权益都要重新配置规则 | 权益模板市场，复制已有模板快速创建 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 营销活动管理 | 下游 | 活动引用权益中心的权益 |
| 触达系统 | 下游 | 发放通知通过触达系统推送 |
| 订单系统 | 上游 | 核销时校验订单信息 |
| 用户中心 | 上游 | 获取用户信息和权益发放记录 |

### 2.2 FEATURE-MKT_BENEFIT_DISTRIBUTION（权益发放）

#### 功能入口 & 作用

**入口位置**：权益中心 → 权益发放

**核心作用**：
- 支持多种方式的权益发放（手动/自动/批量/渠道）
- 灵活配置发放触发条件
- 监控发放进度和处理异常
- 记录完整的发放日志

#### 核心逻辑

```
权益发放流程：

Step 1: 发放配置
  - 选择发放对象（指定用户/用户群/全量用户）
  - 选择权益类型和批次
  - 配置发放数量和每人限领数量

Step 2: 发放执行
  - 单笔发放：实时处理，同步返回结果
  - 批量发放：消息队列异步处理，分批执行
  - 触发发放：用户行为事件触发，自动发放

Step 3: 发放确认
  - 库存预占（锁定）
  - 权益生成（权益码生成）
  - 发放记录写入
  - 库存实际扣减

Step 4: 通知触达
  - 发送发放成功通知（App 推送/短信/公众号）
  - 附带权益使用入口链接
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `distribution.batch_size` | 批量发放每批数量 | `1000` |
| `distribution.async_enabled` | 超过 N 条自动走异步 | `100` |
| `distribution.notification.enabled` | 发放后自动通知 | `true` |
| `distribution.deduplication_window` | 去重时间窗口（秒）| `300` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **重复发放** | 同一用户触发多次发放条件导致重复领券 | 发放前检查用户已领记录、去重键控制 |
| **发放延迟** | 大批量发放时排队时间长，用户感知差 | 分批异步处理、进度实时展示 |
| **发放失败重试** | 部分用户发放失败后没有重试机制 | 失败重试队列、最多重试 3 次 |
| **渠道下发失败** | 触达渠道（短信/推送）发送失败 | 降级处理、备用渠道重试 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| FEATURE-MKT_BENEFIT_ASSET_MGMT | 上游 | 权益资产和批次数据 |
| 触达系统 | 下游 | 发放通知通道 |
| 消息队列 | 依赖 | 批量发放异步处理 |
| 用户中心 | 上游 | 获取用户信息和联系方式 |

### 2.3 FEATURE-MKT_BENEFIT_REDEMPTION（权益核销）

#### 功能入口 & 作用

**入口位置**：权益中心 → 核销管理 | 收银台 / 订单确认页

**核心作用**：
- 提供权益核销的校验能力
- 支持线上线下多场景核销
- 实时返回核销结果
- 记录完整核销流水

#### 核心逻辑

```
权益核销流程：

Step 1: 核销请求
  - 用户提交权益码或选择已有权益
  - 系统接收核销请求

Step 2: 权益校验
  - 权益有效性（未使用、未过期、未撤销）
  - 适用条件（订单金额、商品范围、用户范围）
  - 互斥规则（不可与某些优惠叠加）
  - 库存校验（实物权益需检查库存）

Step 3: 核销执行
  - 库存扣减（实物）
  - 权益状态变更（已使用）
  - 生成核销流水号
  - 更新用户权益列表

Step 4: 结果回执
  - 返回核销成功/失败及原因
  - 通知用户核销结果
  - 关联订单或交易记录
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `redemption.timeout_ms` | 核销接口超时时间 | `200` |
| `redemption.max_retry` | 核销失败最大重试次数 | `3` |
| `redemption.rollback_enabled` | 失败是否回滚库存 | `true` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **核销幂等性** | 网络超时导致客户端重试，造成重复核销 | 核销流水号幂等控制、状态机校验 |
| **库存超卖** | 实物权益并发核销时超卖 | 分布式锁 + 库存预占 |
| **核销失败用户体验差** | 失败原因不明确，用户不知道怎么处理 | 细化错误码，提供明确引导 |
| **跨门店核销不一致** | 不同门店核销数据不同步 | 统一核销服务、实时数据同步 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 订单系统 | 上游 | 订单金额和商品信息用于核销校验 |
| 支付系统 | 下游 | 核销后调用支付优惠 |
| 门店系统 | 上游 | 线下门店核销场景 |
| 会员系统 | 上游 | 会员等级影响权益适用性 |

## 4. 数据相关

### 4.1 核心指标口径

| 指标 | 定义 | 计算逻辑 | 目标值 |
|:---|:---|:---|:---:|
| **发放量** | 成功发放的权益总数 | `count(issue_success)` | - |
| **领取量** | 用户实际领取的权益数 | `count(claim_success)` | - |
| **核销量** | 实际完成核销的权益数 | `count(redemption_success)` | - |
| **核销率** | 核销量 / 发放量的比例 | `redemption_count / issue_count * 100%` | > 25% |
| **到店核销率** | 线下核销量 / 发放量的比例 | `offline_redemption / issue_count * 100%` | > 15% |
| **权益使用客单价** | 使用权益的订单平均金额 | `sum(order_amount) / redemption_count` | - |
| **权益 ROI** | 带来的 GMV / 权益成本 | `attributed_gmv / benefit_cost` | > 5x |

### 4.2 数据来源 & 计算逻辑

| 数据 | 来源 | 计算逻辑 |
|:---|:---|:---|
| 发放记录 | 权益发放服务 | 实时写入 MySQL + 同步到 ClickHouse |
| 核销记录 | 核销服务 | 实时写入 MySQL + 事件到 Kafka |
| 订单数据 | 订单系统 | 通过 order_id 关联核销记录 |
| 用户行为 | 前端埋点 | 领取、浏览、点击等行为数据 |

### 4.3 埋点事件字典

| 事件名 | 触发时机 | 携带参数 |
|:---|:---|:---|
| `benefit_issue` | 权益发放成功 | benefit_id, batch_id, user_id, channel |
| `benefit_claim` | 用户领取权益 | benefit_id, user_id, claim_time |
| `benefit_view` | 用户浏览权益详情 | benefit_id, user_id, entry_point |
| `benefit_use` | 权益核销成功 | benefit_id, order_id, discount_amount |
| `benefit_expire` | 权益过期 | benefit_id, user_id, expire_time |

### 4.4 报表 / 看板地址

| 报表 | 地址 | 说明 |
|:---|:---|:---|

## 5. 规则与配置中心

### 5.1 权益类型配置

| 权益类型 | 描述 | 适用场景 | 特殊配置 |
|:---|:---|:---|:---|
| **优惠券** | 满减券、折扣券、兑换券 | 促销活动 | 面值、满减门槛、可用商品范围 |
| **积分** | 消费积分、奖励积分 | 会员运营 | 积分比例、有效期、兑换规则 |
| **实物权益** | 赠品、实物兑换 | 新客礼包 | 库存管理、发货规则 |
| **虚拟权益** | 会员时长、等级加速 | 会员运营 | 开通时长、权限范围 |

### 5.2 权益叠加规则

```
叠加规则优先级（高 → 低）：
1. 不可叠加类（与任何权益互斥）
2. 品牌/品类券（与通用券可叠加）
3. 通用满减券
4. 折扣券
5. 积分（独立使用）

互斥规则：
- 同一批次优惠券不可叠加使用
- 限时折扣与满减券不可叠加
- 会员专属折扣与普通优惠互斥
```

### 5.3 权益有效期规则

| 有效期类型 | 规则 | 示例 |
|:---|:---|:---|
| 固定日期 | 指定起止日期 | 2026-03-01 至 2026-03-31 |
| 领取后生效 | 领取后 N 天内有效 | 领取后 7 天有效 |
| 滚动有效期 | 每月 / 每年固定天数 | 每月 1-7 日可用 |

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: 用户反馈领不到券怎么办？

**排查步骤**：
1. 检查权益库存是否充足
2. 检查用户是否已达到领取上限
3. 检查权益是否在有效期
4. 检查发放渠道是否开启

**解决方案**：
- 联系运营补充库存
- 核实用户领取记录
- 重新配置发放规则

#### Q: 核销失败怎么排查？

**排查步骤**：
1. 确认权益码是否正确
2. 检查权益状态（已使用/已过期/已撤销）
3. 检查适用条件（订单金额、商品是否在范围内）
4. 检查系统核销日志

**解决方案**：
- 重新获取有效权益码
- 调整订单满足适用条件
- 联系客服处理异常

#### Q: 库存对不上怎么查？

**排查步骤**：
1. 对比 MySQL 库存和 Redis 库存
2. 检查是否有未完成的核销事务
3. 检查发放和核销记录是否有遗漏
4. 检查库存回滚是否正常

**解决方案**：
- 执行库存对账脚本
- 手动修正库存差异
- 排查代码中的库存操作漏洞

## 8. 方法论沉淀

### 8.1 权益设计方法论

**权益价值评估**：
```
权益边际成本 = 权益面值 × 核销率
权益边际收益 = 客单价提升 × 核销率 × 毛利
权益 ROI = 权益边际收益 / 权益边际成本
```

**权益设计原则**：
1. 价值感知清晰：用户一眼看懂能省多少钱
2. 使用门槛合理：踮脚能够到的门槛最能刺激消费
3. 有效期适度：太长降低紧迫感，太短降低使用率
4. 稀缺性营造：限量和限时制造紧迫感

### 8.2 权益运营策略

**发放策略**：
- 新客：首单优惠，高抵扣券，提升转化
- 老客：分层运营，高价值客户发大额券
- 沉默用户：召回券，较低门槛激活
- 大促节点：叠加多档次券，提升客单价

**核销提升策略**：
- 临期提醒：过期前 3 天触达提醒使用
- 使用引导：核销页面展示适用范围和案例
- 关联推荐：展示凑单商品，引导提升客单
