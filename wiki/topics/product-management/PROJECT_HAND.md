# 1. 产品总览

## 1.1 产品定位 & 核心价值

**一句话定位**：构建企业级产品需求管理体系，实现业务需求的智能拆解、结构化管理和全流程追踪

**核心价值**：
- **需求可追踪**：从业务需求 → Epic → Feature → Story 的完整链路
- **协作可视化**：飞书多维表格实时同步，团队协同透明高效
- **知识可沉淀**：Neo4j 图数据库存储，产品知识资产化
- **决策智能化**：AI 辅助分析，数据驱动决策

# 2. 功能结构（按模块归档）

## 2.1 模块：Epic 拆解流程

### 功能入口 & 作用
- **入口**：产品 PRD 文档 → Neo4j 录入 → 飞书表格同步
- **作用**：将业务需求系统化拆解为可执行的技术任务

### 核心逻辑

```
Step 1: 范围确认
  - 阅读产品 PRD 和需求文档
  - 查看相关前端代码目录
  - 确认 Epic 范围和边界
  
Step 2: Feature 结构重建
  - 清理 Neo4j 中现有 Feature/Story
  - 基于 PRD 和代码重新拆解 Feature 结构
  - 创建 Feature 节点
  
Step 3: Story 拆解
  - 每个 Feature 拆解为 Story
  - 为每个 Story 生成验收标准（4项标准）
  - 创建 Story 节点
  
Step 4: 飞书表格同步
  - 创建飞书多维表格
  - 写入 Story 数据
  - 生成表格链接
  
Step 5: 验证与复盘
  - 对比 PRD 确认完整性
  - 对比代码确认准确性
  - 更新 HEARTBEAT.md
```

### 常见坑 & 注意点

| 坑点 | 说明 | 解决方案 |
|:---|:---|:---|
| **范围蔓延** | 拆解过程中不断发现新需求，范围越来越大 | 严格按 PRD 范围执行，新需求另起 Epic |
| **粒度不均** | 有的 Feature 只有 1 个 Story，有的有 20 个 | Feature 重新划分，确保粒度均匀（3-10 Story/Feature） |
| **遗漏依赖** | Story 之间有依赖关系，但拆解时未识别 | 拆解时标注依赖关系，排期时考虑 |
| **验收标准模糊** | 验收标准描述不清，测试时产生歧义 | 使用 4 项标准模板，描述要具体可验证 |

### 关联页面 / 依赖
- **前置依赖**：产品 PRD 文档、前端代码目录
- **后置依赖**：飞书表格、技术排期、测试计划
- **关联工具**：Neo4j、飞书多维表格、AI 问答服务

## 2.2 模块：Neo4j 图数据管理

### 功能入口 & 作用
- **入口**：Python 脚本 / Cypher 查询 / AI 问答服务
- **作用**：存储和管理产品层级结构（Domain → Epic → Feature → Story）

### 核心逻辑

**数据模型**：
```
(ProductDomain)-[:HAS_EPIC]->(Epic)-[:HAS_FEATURE]->(Feature)-[:HAS_STORY]->(Story)
```

**核心 Cypher 查询**：
```cypher
// 查询 Epic 的完整层级结构
MATCH (e:Epic {id: 'EPIC-DFD_DATA_ASSET'})-[:HAS_FEATURE]->(f:Feature)-[:HAS_STORY]->(s:Story)
RETURN f.label as Feature, s.label as Story, s.status as Status

// 统计 Epic 的完成度
MATCH (e:Epic)-[:HAS_FEATURE]->(f:Feature)-[:HAS_STORY]->(s:Story)
WITH e, count(s) as total, count(CASE WHEN s.status = "已完成" THEN 1 END) as completed
RETURN e.label as Epic, round(100.0 * completed / total, 2) as completionRate
```

### 关键配置项

| 配置项 | 配置值 |
|:---|:---|
| 地址 | `bolt://localhost:7687` |
| 用户名 | `neo4j` |
| 密码 | `password123` |
| 数据库 | `neo4j` |

### 常见坑 & 注意点
- **性能**：大量数据导入时使用批量操作，避免单条创建
- **索引**：为常用查询字段（id, label, status）创建索引
- **备份**：定期备份数据，防止意外丢失

## 2.3 模块：飞书表格同步

### 功能入口 & 作用
- **入口**：Python 脚本调用飞书 API
- **作用**：将 Neo4j 中的 Story 数据同步到飞书多维表格，便于团队协作

### 核心逻辑

**同步流程**：
```
Neo4j 读取 Story 数据 → 创建飞书表格 → 批量写入数据 → 生成分享链接
```

**表格字段**：
| 字段 | 类型 | 说明 |
|:---|:---|:---|
| Story编号 | 文本 | 唯一标识 |
| Story名称 | 文本 | 简短描述 |
| 所属Feature | 文本 | 关联的Feature |
| 所属Epic | 文本 | 关联的Epic |
| 需求描述 | 文本 | 详细描述 |
| 验收标准 | 多行文本 | 4项标准 |
| 状态 | 单选 | 待办/进行中/已完成 |
| 优先级 | 单选 | P0/P1/P2/P3 |
| 负责人 | 人员 | 开发负责人 |
| 故事点 | 数字 | 工作量估算 |

### 关键配置项
- **App Token**: `ANhxbU3MDabWsysyeI8c4t0mnfe`
- **权限**: 需要 `bitable:record` 等权限

### 常见坑 & 注意点
- **API 限流**：飞书 API 有调用频率限制，注意批量操作的间隔
- **字段映射**：确保 Neo4j 字段和飞书表格字段一一对应
- **权限**：确保 Bot 有权限访问和编辑表格

# 5. 规则与配置中心

## 5.1 Story 验收标准模板（强制）

**每个 Story 必须包含以下 4 项验收标准**：

| 标准 | 说明 | 示例 |
|:---|:---|:---|
| **功能可正常使用** | 核心功能完整 | 页面可正常加载，数据可正常展示 |
| **相关操作正常** | CRUD 操作无误 | 新增/编辑/删除/查询操作正常 |
| **数据准确完整** | 数据流转正确 | 字段显示准确，数据计算正确 |
| **交互符合预期** | UI/UX 符合设计 | 布局、样式、交互符合 PRD |

## 5.2 Epic 拆解流程标准（SOP）

**必须按以下 5 步执行**：

1. **范围确认** - 阅读 PRD + 查看代码 + 确认边界
2. **Feature 重建** - 清理旧数据 + 重建结构 + 创建节点
3. **Story 拆解** - 拆解 Story + 生成验收标准 + 创建节点
4. **飞书同步** - 创建表格 + 写入数据 + 生成链接
5. **验证复盘** - 对比 PRD + 对比代码 + 更新文档

## 5.3 Feature 粒度标准

| 标准 | 值 | 说明 |
|:---|:---:|:---|
| 最小 Story 数 | 3 | 避免过度拆分 |
| 最大 Story 数 | 10 | 避免粒度过大 |
| 最佳 Story 数 | 5-7 | 推荐范围 |

## 5.4 优先级定义

| 优先级 | 定义 | 处理时效 |
|:---|:---|:---:|
| **P0** | 阻塞后续工作，必须立即处理 | 24小时内 |
| **P1** | 重要需求，正常迭代完成 | 当前 Sprint |
| **P2** | 一般需求，有空时处理 | 下个 Sprint |
| **P3** | 优化需求，长期规划 | 按需排期 |

# 7. 问题库 & FAQ

## 7.1 常见排查问题

### Q: 飞书表格同步失败怎么办？

**排查步骤**：
1. 检查 App Token 是否有效
2. 检查 Bot 是否有表格编辑权限
3. 检查 API 调用频率是否超限
4. 查看错误日志定位具体原因

**解决方案**：
- 重新授权 Bot 权限
- 分批写入数据，避免超限
- 检查数据格式是否符合要求

### Q: Neo4j 查询性能慢怎么办？

**排查步骤**：
1. 检查是否为常用查询字段创建索引
2. 检查查询是否返回大量数据
3. 检查是否存在全表扫描

**解决方案**：
- 为 id, label, status 字段创建索引
- 使用 LIMIT 限制返回数量
- 优化 Cypher 查询语句

### Q: Story 验收标准不明确怎么办？

**解决方案**：
- 强制使用 4 项标准模板
- 与产品经理确认具体业务场景
- 参考类似 Story 的验收标准

# 8. 方法论沉淀

## 8.1 需求拆解方法论

**四层拆解法**：
```
产品域 (Domain) → Epic → Feature → Story
```

**拆解原则**：
1. **自顶向下**：先理解整体，再拆解细节
2. **业务驱动**：以用户价值为导向
3. **可交付性**：每个层级都能独立交付价值
4. **可测试性**：每个 Story 都有明确的验收标准

## 8.2 产品设计原则

- **简单性**：能用简单方案不用复杂方案
- **一致性**：保持术语、流程、标准的一致性
- **可追踪性**：每个需求都有完整的追踪链路
- **数据驱动**：用数据说话，但也要有直觉
- **快速迭代**：先做出来，再优化

## 8.3 本体论使用范式

**定义原则**：
- 一个名词只有一个含义
- 层级关系清晰（Domain → Epic → Feature → Story）
- 属性定义标准化（id, label, description, status, priority）

**使用场景**：
- Neo4j 节点类型定义
- 飞书表格字段定义
- 团队术语统一

## 8.4 工具流程

**日常流程**：
1. 收到产品 PRD
2. Neo4j 拆解 Epic/Feature/Story
3. 飞书表格同步
4. 技术排期
5. 开发实现
6. 验收确认
7. 更新状态

**工具链**：
- Neo4j：产品知识存储
- 飞书表格：协作同步
- OpenClaw：Agent 自动化
- Python：脚本工具
