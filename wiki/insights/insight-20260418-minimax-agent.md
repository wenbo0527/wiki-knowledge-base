## 执行摘要

MiniMax分享了公司发展与大模型业务布局，重点介绍了大模型参与自迭代的开发模式，以及AI Agent在企业内部的落地实践与未来企业级Agent的发展方向。

**核心要点**：
1. AI参与迭代的分工模式让人工从重复性工作中解放

## 三、AI参与迭代的分工机制 ⭐

### 3.1 分层协作模式

| 分工层级 | 说明 | 占比 |
|----------|------|------|
| **蓝色（纯人工）** | 定义下一代大模型方向、下发任务、关键决策 | 极低 |
| **绿色（纯AI）** | 运行脚本、确定实验指标、校验实验结果、任务分析总结 | 大部分 |
| **黄色（人机协作）** | AI给出初步结论（如Loss变化、数据标注结果），人工负责结论校验与最终决策 | 适中 |

### 3.2 核心价值

> 该模式将人工从重复性工作中解放，仅负责核心节点把控，大幅加快了大模型迭代速度，是MiniMax迭代周期大幅缩短的核心原因。

## 四、MiniMax Agent产品迭代路径

### 4.1 产品发展历程

```
┌─────────────────────────────────────────────────────────────┐
│  V1.0: 初始版本                                            │
│  → 支持生成完整前后端网站、客户端等开发任务                   │
├─────────────────────────────────────────────────────────────┤
│  V2.0: 技能化升级                                          │
│  → 引入Experts技能体系                                     │
│  → 用户通过自然语言描述创建自定义工作流（Skill）              │
│  → Skill本质为markdown文件，使用门槛极低                    │
├─────────────────────────────────────────────────────────────┤
│  V3.0: 桌面端迭代                                          │
│  → 推出桌面端Workspace                                      │
│  → 满足本地文档整理等桌面操作场景需求                        │
├─────────────────────────────────────────────────────────────┤
│  V4.0: 生态接入                                            │
│  → 结合OpenCloud、Hermes推出MaxCloud、MaxHermes            │
│  → 接入微信、飞书等常用软件                                  │
│  → 用户可在熟悉工具中直接调用Agent能力                        │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 Skill的本质

- 本质就是一个**markdown文件**
- 不需要复杂开发
- 甚至有专门辅助生成Skill的Skill
- 使用门槛极低

**示例**：用户可以将自己公众号的历史文章和写作风格输入，自动生成匹配个人风格的写作Skill

### 4.3 OpenCloud适配优化

MiniMax在2026年1月和OpenCloud团队深度合作：
- 模型已针对OpenCloud、Hermes等Agent场景做专门迭代优化
- 被OpenCloud官方在社区推荐
- **优化后**：速度更快、效果更好

## 五、AI Native公司内部落地实践 ⭐

### 5.1 AI Native的文化定位

**核心理念**：全员遇到任何问题时，**优先考虑让AI解决**，而非直接求助其他同事

**基础建设**：
- 为全员提供不限量token
- 支持在各类工具中使用AI能力

### 5.2 内部Agent实习生系统

**上线时间**：2025年8月

**核心能力**：
- 打通公司内部所有系统
- 集成到飞书
- 可自动调用内部系统完成用户指令

**覆盖范围**：全公司所有岗位
- 技术、HR、行政、法务、财务、设计等非技术岗位
- **降低了非技术岗位的使用门槛**

### 5.3 系统架构：通用底座+细分领域子Agent

```
┌─────────────────────────────────────────────────────────────┐
│                    通用底座                                 │
│           (复用MiniMax Agent基础框架)                        │
│                    ↓ ↑                                     │
│         ┌─────────┼─────────┐                              │
│         ↓         ↓         ↓                              │
│    ┌────────┐ ┌────────┐ ┌────────┐                        │
│    │HR子Agent│ │算法子Agent│ │前端子Agent│ ...              │
│    └────────┘ └────────┘ └────────┘                        │
│                                                             │
│  无法路由到细分Agent的任务 → 通用底座处理                    │
│  各业务部门可自行开发对应场景的子Agent                       │
└─────────────────────────────────────────────────────────────┘
```

### 5.4 典型落地案例

#### 案例1：HR招聘场景
- AI自动登录招聘平台
- 根据要求筛选候选人
- 针对每个候选人生成个性化打招呼信息
- 自动完成简历收集

#### 案例2：算法研发场景
- 数据清洗全流程引入AI
- 自动化实验
- 故障排查

#### 案例3：技术支持场景
- 所有错误日志先由Agent实习生分析排查
- 无法解决时再转人工处理

#### 案例4：设计场景
- 设计师交付内容从单纯设计稿
- 升级为**带交互的可运行页面**
- 部分前端编码工作由AI辅助完成
- 原来对代码恐惧的设计同学也可以熟练使用

### 5.5 落地效果总结

> AI Agent重塑了企业工作方式，模糊了岗位之间的职责边界，消除了非技术岗位对代码和AI的恐惧，大幅提升了全公司整体工作效率。

## 六、企业级Agent的未来发展方向 ⭐

### 6.1 核心需求点

| 需求 | 说明 |
|------|------|
| **安全环境隔离** | 不同部门需要独立可控的Workspace，避免数据交叉 |
| **企业级技能沉淀** | 内部工作流技能化是核心需求，内部技能是企业核心资产 |
| **多层级记忆体系** | 需要支持个人记忆、团队记忆、公司记忆三个层级 |
| **全时段全场景服务支撑** | 满足企业各类业务场景的连续服务需求 |

### 6.2 三层记忆体系详解

| 层级 | 范围 | 记忆重要性定义 |
|------|------|----------------|
| **个人记忆** | 单个用户 | 个人偏好、工作习惯 |
| **团队记忆** | 团队/部门 | 团队知识、协作流程 |
| **公司记忆** | 公司整体 | 企业知识资产 |

## 七、关键洞察

### 洞察1：AI自迭代是核心能力

> 不仅是模型适配，更重要的是让大模型自身参与迭代优化的模式

### 洞察2：Skill降低AI使用门槛

- Skill本质是markdown文件
- 可通过自然语言生成
- 用户甚至不需要了解技术细节

### 洞察3：企业落地需要文化+工具双轮驱动

- 文化：优先用AI解决问题
- 工具：为全员提供不限量AI能力
- 产品：为非技术岗位做封装，消除恐惧

### 洞察4：Agent模糊岗位边界

AI Agent重塑了企业工作方式：
- 设计师可以做前端工作
- HR可以做数据工作
- 非技术岗位也能使用AI能力
