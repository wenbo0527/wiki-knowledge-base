## 执行摘要

Memos是由记忆张量公司开发的企业级大模型记忆框架，参考人脑设计，在算力短缺背景下降低大模型训练成本。本次分享围绕Memos的技术架构、OpenCloud插件实现、以及Agent时代记忆框架的演化方向展开。

**核心观点**：
1. 记忆框架需要结合模型驱动与应用驱动，模型决定上限、应用决定下限
2. Agent时代需要面向全生命周期的记忆管理，实现"边运行边学习"
3. Memory to Skill是核心能力：从执行记录中自动沉淀可复用技能

---

## 一、背景：为什么需要记忆框架

### 1.1 大模型记忆的发展逻辑

从ChatGPT开始，大模型厂商持续强调记忆能力：
- OpenAI的GPT系列迭代过程中反复强化记忆功能
- Agent时代到来后，长程任务对记忆的依赖度大幅提升

### 1.2 传统记忆方案分类

| 方案 | 原理 | 效果 | 现状 |
|------|------|------|------|
| **模型层增强** | 直接在大模型基座中增加记忆能力 | 效果不如RAG | 应用较少 |
| **外部记忆增强** | 基于应用层的记忆框架 |效果好、实现简单 | **当前主流路线** |

### 1.3 Memos的定位

- 结合模型驱动与应用驱动两种路线
- 模型驱动决定记忆能力**上限**，应用驱动决定记忆能力**下限**
- 通过内外协同实现深度与广度兼顾
- 支持分层记忆与多触点应用

> **核心设计原则**：从系统层面进行整体设计，而非局限于单一维度

## 二、记忆模块核心功能

完整记忆系统需要具备五大核心功能：

### 2.1 记忆抽取
- 处理输入到记忆系统的内容
- 将非结构化信息转化为可存储的记忆单元

### 2.2 记忆组织
- 对抽取后的记忆进行结构化存储
- 可引入图存储组织实体关系
- **适用场景**：复杂推理场景下有性能增益
- **注意**：简单任务效果提升不明显

### 2.3 记忆检索
- 实现"存"与"用"的衔接
- **是记忆系统的核心功能**
- 需要平衡精确性和召回率

### 2.4 记忆更新
- 处理记忆增量与演化
- 支持记忆的修改、删除、版本管理

### 2.5 记忆共享 ⭐
- 支持团队协作与多Agent协作
- **实现群体智能演化**
- 解决企业信息流失问题（员工离职导致的知识流失）

> ⚠️ **常见问题**：记忆抽取、组织过程中容易产生幻觉，Memos框架针对该问题做了专门优化

## 三、Memos分层技术架构

### 3.1 整体架构图

```
┌─────────────────────────────────────────────────────────────┐
│                    记忆编码与应用层                          │
│         （API / MCP / 本地化服务 / 记忆反馈更新）             │
├─────────────────────────────────────────────────────────────┤
│                    记忆调度层                                │
│    （工作记忆管理 / K Cache复用 / 激活记忆复用推理成本优化）  │
├─────────────────────────────────────────────────────────────┤
│                    记忆治理层                               │
│      （幻觉控制 / 水印 / 权限控制 / 隐私保护 / 生命周期）    │
├─────────────────────────────────────────────────────────────┤
│                    记忆存储层                               │
│           （记忆立方体 Memory Cube × N）                     │
│    ┌─────────────────┐     ┌─────────────────┐             │
│    │   向量存储       │     │   图数据库       │             │
│    │  (基础检索)      │     │ (复杂实体关系)   │             │
│    └─────────────────┘     └─────────────────┘             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 记忆存储层详解

**核心组织单元：记忆立方体 (Memory Cube)**

每个Cube对应一个独立的存储空间：
- 用户级Cube
- 场景级Cube
- 知识库级Cube

**两层存储设计**：

| 存储层 | 作用 | 适用场景 |
|--------|------|----------|
| **向量存储** | 基础检索存储 | 快速相似度匹配 |
| **图数据库(GraphDB)** | 存储复杂实体关系与因果关系 | 复杂推理、思维链前置 |

> **图数据库价值**：可实现思维链前置，让复杂推理直接通过检索获得结果，提升复杂问题得分
> 
> **注意事项**：图数据库存储和查询资源消耗大，高QPS场景需要谨慎使用，简单场景增益有限

**记忆分享与继承**：
- 存储架构天然支持记忆分享
- 可实现**用户记忆继承**
- **解决企业员工离职导致的知识流失问题**

### 3.3 记忆治理层详解

提供记忆全生命周期管理：

| 功能 | 说明 |
|------|------|
| **幻觉控制** | 检测并纠正记忆生成中的幻觉问题 |
| **水印** | 记忆来源追溯 |
| **权限控制** | 保证记忆在权限范围内流通 |
| **隐私保护** | 避免不同场景记忆串扰 |

### 3.4 记忆调度层详解 ⭐

**核心功能：工作记忆(Working Memory)管理**

- 缓存高频使用记忆
- 实现快速响应

**K Cache复用实现推理成本优化**：

```
问题：
大模型推理中大量请求依赖重复知识
传统检索排序不满足严格前缀匹配
→ K Cache无法复用 → 推理成本高

解决方案：
Memos通过对检索知识的有序存储实现K Cache复用
→ 大幅降低调用API的成本
```

### 3.5 记忆编码与应用层

- 负责记忆加工与重组
- 对外提供多种接口：
  - API
  - MCP (Model Context Protocol)
  - 本地化服务
- 支持记忆反馈更新

## 四、Memos OpenCloud插件设计

### 4.1 原生OpenCloud记忆方案痛点

| 痛点 | 具体表现 |
|------|----------|
| **非技术用户不友好** | 无法对记忆进行增删改查，不支持可视化管理 |
| **启动时间长** | 依赖Agent自身能力实现记忆压缩，可达数十秒 |
| **Token消耗大** | 复杂任务可达数百万甚至上千万级 |
| **触发机制不完善** | 容易重复询问用户，额外消耗Token |
| **多Agent隔离** | Agent之间记忆完全隔离，不支持协同共享 |

### 4.2 Memos插件四大核心优化

#### 优化1：可视化记忆管理界面
- 支持查看记忆创建时间、创建Agent、隔离状态
- 支持原有记忆导入

#### 优化2：原生框架能力引入
- 引入Memos原生框架的存储、检索、继承能力
- 替代原生文件系统

#### 优化3：Agent进化能力 ⭐
- 通过与环境交互不断积累经验
- 规避过往踩过的坑
- **让Agent越用越聪明**
- 类似私人秘书的成长过程
- 所有经验积累过程可在可视化面板展示

> "老板雇了一个秘书，要么一开始就觉得不够聪明换掉，要么就是这个秘书越用越觉得好用，因为他越来越了解你。培养智能体是一个道理。"

#### 优化4：多Agent记忆共享与协同进化 ⭐
- 解决原生OpenCloud多Agent记忆隔离问题
- **实现群体智能能力提升**

### 4.3 版本选择

| 版本 | 特点 | 适用场景 |
|------|------|----------|
| **云上版** | 记忆处理开销由Memos云服务承担，支持高并发 | 50QPS下500ms内响应，对时延敏感 |
| **本地版** | 数据保留在本地，用户自行承担处理开销 | 对数据隐私有要求，目前快速迭代中 |

### 4.4 核心增强能力

#### 记忆沉淀技能(Memory to Skill) ⭐⭐
这是插件**最重要的功能**：

```
机制：
1. 对对话/执行记录做语义切分
2. 自动识别完整任务边界
3. 提取任务关键点、操作脚本、用户反馈
4. 自动沉淀为可复用的技能

价值：
- 支持团队共享
- 多个用户针对同一任务的经验可以联合迭代
- 整合为更完善的技能体系
```

**示例**：不同系统下安装软件的经验可以整合为完整的安装技能

#### 局域网团队共享
- 可搭建本地服务端
- 局域网内其他用户可通过插件加入团队
- 实现记忆、任务、技能的共享
- 在共享过程中可以合并多个技能，生成更强大的技能

### 4.5 Hook机制

当前插件基于OpenCloud的Hook机制实现：
- 通过6个Hook完成无侵入记忆增强
- 未来规划：标准化Agent记忆标准

> 规划：未来一个月内会发布标准化Agent记忆标准，从主动集成改为被集成，推动记忆生态标准化

## 五、Agent时代的记忆框架演化方向

### 5.1 记忆需求的变化

| 传统对话场景 | Agent执行场景 |
|-------------|---------------|
| 用户提问 → 模型回答 | 用户请求 → 组建上下文 → 推理生成指令 → 工具执行 → 结果反思 → 循环迭代 |
| 简单流程 | 复杂流程，包含工具调用、脚本执行、中间结果等非文本内容 |

### 5.2 新记忆框架的核心价值

```
Agent执行过程中会产生大量中间轨迹
↓
这些信息对后续相似任务有极高借鉴价值
↓
帮助Agent快速定位问题、减少重复探索
↓
提升执行效率、降低Token消耗
↓
实现Agent边运行边学习的自我演化
```

### 5.3 核心设计思路

#### 思路1：分层反馈量化
整合执行过程反馈与用户反馈，对记忆价值做全面量化，支撑框架自我演化

#### 思路2：分层检索机制
同时召回三类信息：
- 沉淀的技能
- 执行轨迹
- 环境场景认知

让规划和工具调用更精准

#### 思路3：五层记忆加工流程

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 任务案例构建                                      │
│  → 将执行过程记录为可复用的任务案例                          │
├─────────────────────────────────────────────────────────────┤
│  Step 2: 执行反馈价值回填                                  │
│  → 基于执行结果和用户反馈做价值评估                          │
├─────────────────────────────────────────────────────────────┤
│  Step 3: 跨任务归纳总结                                    │
│  → 合并相似任务记忆                                        │
├─────────────────────────────────────────────────────────────┤
│  Step 4: 规律抽象构建认知                                  │
│  → 抽象规律与约束，构建环境认知                            │
│  → 例如：Agent探索后生成对当前文件系统的结构化认知          │
├─────────────────────────────────────────────────────────────┤
│  Step 5: 知识沉淀                                          │
│  → 最终沉淀为四类知识：                                     │
│    1. 原始记忆                                            │
│    2. 环境管理地图                                        │
│    3. 操作手册                                            │
│    4. 可复用执行步骤                                       │
└─────────────────────────────────────────────────────────────┘
```

### 5.4 新任务执行流程

```
新任务到来
    ↓
检索匹配相关知识（技能/轨迹/认知）
    ↓
辅助规划、调度、修正
    ↓
生成执行结果
    ↓
实现任务级、子任务级的知识复用
    ↓
达到边运行边学习优化的效果
```

## 六、关键洞察与金句

### 核心洞察

1. **模型驱动决定上限，应用驱动决定下限**
   - 需要从系统的层面去考虑整个记忆框架的设计

2. **Agent进化能力是核心**
   - 通过交互积累经验，让Agent越用越聪明
   - 类似私人秘书的成长过程

3. **记忆共享解决企业知识流失**
   - 员工离职导致的知识流失是企业面临的核心问题
   - 记忆继承是解决方案

4. **K Cache复用降低推理成本**
   - 大模型推理成本优化的重要方向

### 经典金句

> "模型驱动决定上限，应用驱动决定下限，我们需要从系统的层面去考虑整个记忆框架的设计。"

> "核心的本质就是说他能够根据环境的交互不断的去提升自己的能力，这也就是我们希望我们的Agent以后能够实现私人个人助理一个重要的观念。"

> "老板雇了一个秘书，要么就开始就换掉，觉得这个秘书不够聪明，要么就是说这个秘书越用越不会再换，因为这个秘书会越来越了解他，培养智能体是一个道理。"
