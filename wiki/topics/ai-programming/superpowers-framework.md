# Superpowers 工作流框架

> 来源：Andrej Karpathy 推崇的 AI 辅助开发工程纪律框架
> 整理时间：2026-04-13
> 适用场景：Vibe Coding、AI 协作开发、产品需求拆解

---

## 一句话本质

Superpowers 是一套**给 AI 助手上"工程纪律"**的工作流框架，把 Vibe Coding 从"凭感觉写代码"升级为**可流程化、可审查、可交付**的工程化协作模式。

**核心解决**：
- AI 乱做、跑偏、幻觉
- 代码无测试、无设计、无审查
- 上下文越长越乱
- 交付不可靠、不可回溯

---

## 核心 6 大原则

| # | 原则 | 说明 |
|:--:|:---|:---|
| 1 | **先设计，再实现** | 不批准不干活 |
| 2 | **任务必须拆小** | 2–5 分钟可完成 |
| 3 | **测试先行 TDD** | 先写失败测试，再写代码 |
| 4 | **子代理隔离** | 一个任务一个干净上下文 |
| 5 | **证据驱动** | 不猜、不蒙、不随机调试 |
| 6 | **必须审查收尾** | 不合规不合并 |

---

## 标准工作流（固定套路）

```
需求 → 澄清&设计 Brainstorming → 批准
→ 建独立工作区 Worktree
→ 拆细粒度任务 Writing Plans
→ 子代理执行 Subagent
→ 测试驱动 TDD
→ 代码审查 Code Review
→ 交付合并 Finish
```

---

## 每一步强制要求（门控清单）

### 4.1 Brainstorming（需求&设计）

**要求**：
- 必须先提问澄清，不直接开做
- 给出 2–3 方案 + 优缺点
- 必须等你**明确批准**再进入实现
- 输出：一份可执行的 Spec

**Spec 模板**：

```markdown
## 功能规格

**名称**：[功能名称]
**描述**：[一句话描述]

### 输入
- 参数1：[类型] [说明]
- 参数2：[类型] [说明]

### 输出
- 返回值：[类型] [说明]

### 边界条件
- 空输入处理
- 最大值处理
- 异常输入处理

### 验收标准
- [ ] 标准1
- [ ] 标准2
```

### 4.2 Writing Plans（任务拆分）

**要求**：
- 任务粒度：2–5 分钟能做完
- 每个任务必须写清：
  - 改哪个文件
  - 实现什么
  - 怎么验证
- 无歧义、新人也能看懂

**任务模板**：

```markdown
### 任务 N：[任务名称]

- **文件**：`src/xxx.py`
- **实现**：完成 XXX 功能
- **验证**：运行 `pytest tests/test_xxx.py -v`
- **预期**：测试通过
```

### 4.3 Subagent 子代理

**要求**：
- 每个任务开**新上下文**
- 只传最小必要信息
- 避免历史污染、逻辑漂移

**执行规则**：
```markdown
1. 新任务 → 新代理会话
2. 只传递：任务描述 + 相关文件 + 验收标准
3. 禁止：传递完整项目历史
```

### 4.4 TDD 测试驱动（最硬规则）

**强制执行顺序**：

```
1. 先写测试 → 跑失败（RED）
2. 写最少代码让测试过（GREEN）
3. 重构但测试必须继续过（REFACTOR）
```

**禁止**：先写代码再补测试

**红/绿/重构循环**：

```python
# ========== 红阶段 ==========
def test_function_with_valid_input():
    """RED: 测试应该失败，因为功能未实现"""
    result = my_function(valid_input)
    assert result == expected  # 这会失败

# ========== 绿阶段 ==========
def my_function(input):
    """GREEN: 最简单实现，让测试通过"""
    return expected  # 直接返回预期值

# ========== 重构阶段 ==========
def my_function(input):
    """REFACTOR: 优化实现，但测试继续通过"""
    # 改进代码质量
    validated = validate(input)
    processed = process(validated)
    return format(processed)
```

### 4.5 Systematic Debugging（科学调试）

**强制步骤**：

```
1. 稳定复现（确保问题可重复）
2. 收集证据（日志/堆栈/数据）
3. 提出唯一根因假设
4. 验证修复
```

**禁止**：乱试、乱改、凭感觉修

**调试日志模板**：

```markdown
## Bug 调试记录

**问题描述**：[清晰描述]
**复现步骤**：
1. 步骤1
2. 步骤2
3. 步骤3

**错误信息**：
```
[粘贴错误日志]
```

**根因假设**：[一句话假设]
**验证修复**：[如何验证]
**修复结果**：[PASS/FAIL]
```

### 4.6 Code Review + Finish

**检查清单**：
- [ ] 规范遵循
- [ ] 逻辑正确
- [ ] 安全检查
- [ ] 测试覆盖
- [ ] 文档完整

**输出**：可直接上线的成果

## Superpowers 对产品经理的真正价值

不是让你写代码，而是**把 AI 变成听话、可控、可交付的协作伙伴**。

### 适用场景

| 场景 | 应用方式 |
|------|---------|
| **需求拆解** | 用 Brainstorming 澄清需求，Writing Plans 拆任务 |
| **PRD 结构化** | AI 按模板生成结构化 PRD |
| **流程图/架构图生成** | AI 按规范输出 Mermaid/PlantUML |
| **数据口径梳理** | AI 帮助整理计算逻辑和边界条件 |
| **规则/频控/模板逻辑梳理** | AI 提取和结构化业务规则 |
| **个人知识库结构化** | AI 整理和关联碎片知识 |
| **快速 Demo 生成** | Vibe Coding 可控版 |

## 极简版「Superpowers 指令」

直接发给 AI 使用：

```
请使用 Superpowers 工作流完成任务：
1. 先澄清需求，给出方案并等我批准
2. 拆成 2-5 分钟可完成的小任务
3. 每个任务使用干净子代理上下文
4. 采用 TDD：先写验证方式，再实现
5. 证据驱动，不猜测
6. 完成后做审查并给出最终交付物
```
