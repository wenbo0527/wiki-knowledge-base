## 1. 产品总览

### 1.1 一句话定位
**建立和维护企业级数据标准规范，确保数据定义统一、格式一致、命名规范，实现数据资产的标准化管理。**

### 1.2 产品定位 & 核心价值

| 维度 | 描述 |
|:---|:---|
| **目标用户** | 数据工程师、数据管理员、数据架构师、业务人员 |
| **核心痛点** | 数据命名混乱、格式不统一、质量参差不齐、系统间数据难以互通 |
| **解决方案** | 数据标准库 + 标准检查 + 标准发布 + 合规审计 |
| **核心价值** | 数据标准化覆盖率 95%+，数据互通效率提升 70% |

## 2. 功能结构（按模块归档）

### 2.1 FEATURE-DMT_STD_LIBRARY（标准库管理）

#### 功能入口 & 作用

**入口位置**：数据管理 → 数据标准 → 标准库

**核心作用**：
- 维护数据标准分类体系
- 管理标准条目和定义
- 版本控制和变更记录
- 标准查询和浏览

#### 核心逻辑

```
标准库结构：

数据标准体系
├── 技术标准
│   ├── 命名规范
│   │   ├── 表命名规范
│   │   ├── 字段命名规范
│   │   └── 任务命名规范
│   ├── 数据类型规范
│   │   ├── 基础数据类型映射
│   │   └── 业务数据类型定义
│   └── 编码规范
│       ├── 状态码标准
│       └── 枚举值标准
├── 业务标准
│   ├── 业务术语标准
│   ├── 指标口径标准
│   └── 维度定义标准
└── 管理标准
    ├── 数据安全等级
    ├── 数据质量等级
    └── 生命周期标准
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---|
| `std.approval_required` | 新增标准是否需要审批 | `true` |
| `std.version_enabled` | 版本控制开关 | `true` |
| `std.max_violation_rate` | 允许的最大违规率 | `5%` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **标准过于严格** | 标准要求太高导致落地困难 | 分级标准、渐进式推进 |
| **标准覆盖不全** | 新业务场景无标准可用 | 建立标准快速制定通道 |
| **标准更新滞后** | 业务变化后标准未及时更新 | 定期标准评审、版本联动 |
| **多标准冲突** | 同一对象有多个标准要求 | 建立标准优先级和仲裁机制 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 业务概念管理 | 依赖 | 业务术语标准来源 |
| 元数据中心 | 依赖 | 元数据关联 |

### 2.2 FEATURE-DMT_STD_CHECK（标准检查）

#### 功能入口 & 作用

**入口位置**：数据管理 → 数据标准 → 标准检查

**核心作用**：
- 新建数据标准检查（前置卡点）
- ETL 开发标准校验
- 数据模型评审
- 标准违规告警

#### 核心逻辑

```
标准检查流程：

1. 表/模型创建检查
   - 表名是否符合命名规范
   - 字段命名是否符合规范
   - 数据类型是否符合标准
   - 是否包含必选标准字段

2. ETL 开发检查
   - 源表和目标表是否符合标准
   - 字段映射是否符合标准
   - 转换逻辑是否遵循标准
   - 异常处理是否符合规范

3. 周期性巡检
   - 全量表标准合规检查
   - 生成违规报表
   - 发送告警通知
   - 跟踪整改情况
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `check.block_on_error` | 违规是否阻断流程 | `true` |
| `check.warn_on_low` | 低级别违规是否警告 | `true` |
| `check.auto_fix` | 是否支持自动修复 | `false` |
| `check.schedule` | 周期性巡检周期 | `DAILY` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **检查耗时长** | 全量检查影响开发效率 | 分层检查、增量检查、异步检查 |
| **误报率高** | 合规数据被误判为违规 | 优化检查规则、人工确认反馈 |
| **修复成本高** | 历史违规修复工作量大 | 提供批量修复工具、分阶段治理 |
| **检查规则僵化** | 无法适应特殊业务场景 | 支持规则豁免和自定义规则 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 标准库 | 依赖 | 检查规则来源 |
| ETL 开发平台 | 集成 | 检查触发入口 |
| 告警中心 | 依赖 | 违规通知 |

#### 核心逻辑

```
标准治理流程：

1. 覆盖率分析
   - 按标准分类统计覆盖率
   - 按数据源统计覆盖率
   - 按部门统计覆盖率
   - 生成覆盖率热力图

2. 违规分析
   - 违规类型分布
   - 违规趋势分析
   - 高频违规识别
   - 违规责任人分析

3. 合规审计
   - 合规率统计
   - 重大违规追踪
   - 整改情况跟踪
   - 审计报告生成

4. 优化建议
   - 标准完善建议
   - 检查规则优化
   - 流程改进建议

## 5. 规则与配置中心

### 5.1 命名规范标准

```
表命名规范：

格式：{层级}_{业务域}_{主题}_{更新频率}_{版本}
示例：
- ods_user_info_d （ODS 层用户信息表，每日更新）
- dwd_order_detail_d （DWD 层订单明细表，每日更新）
- dws_sales_summary_m （DWS 层销售汇总表，每月更新）

字段命名规范：

通用规则：
- 使用小写下划线分隔
- 不超过 32 个字符
- 禁止使用拼音缩写

时间字段：
- dt：分区字段（格式 YYYYMMDD）
- create_time：创建时间
- update_time：更新时间

ID 字段：
- xxx_id：主键 ID
- xxx_key：业务键

状态字段：
- xxx_status：状态
- xxx_type：类型
```

### 5.2 数据类型标准

```
基础数据类型映射：

| 业务含义 | MySQL | Hive | ClickHouse | 说明 |
|:---|:---|:---|:---|:---|
| 整数 | INT | INT | Int32 | 4 字节整型 |
| 长整数 | BIGINT | BIGINT | Int64 | 8 字节整型 |
| 小数 | DECIMAL(18,4) | DECIMAL(18,4) | Decimal(18,4) | 金额等精确计算 |
| 字符串 | VARCHAR(255) | STRING | String | 普通字符串 |
| 长文本 | TEXT | STRING | String | 文本内容 |
| 日期 | DATE | DATE | Date | 日期 |
| 时间戳 | TIMESTAMP | TIMESTAMP | DateTime | 精确时间 |
```

### 5.3 合规检查规则

```
检查级别：

严重（阻断）：
- 表名不符合规范
- 缺少必选字段
- 数据类型严重不匹配

警告（提醒）：
- 字段命名不规范
- 注释缺失
- 缺少分区字段

提示（建议）：

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: 表名不符合规范被拦截怎么办？

**排查步骤**：
1. 查看具体的命名规范要求
2. 确认表所属的业务域和层级
3. 对照规范重新命名

**解决方案**：
- 按规范重新命名表
- 如有特殊情况可申请豁免
- 提交改名申请

#### Q: 标准检查误报怎么办？

**排查步骤**：
1. 查看检查详情，确认误报原因
2. 收集证据证明合规
3. 提交误报反馈

**解决方案**：
- 反馈给标准管理员
- 提供例外场景说明
- 等待规则优化

#### Q: 如何申请新增标准？

**排查步骤**：
1. 确认该标准是否已存在
2. 准备标准草案
3. 提交标准申请

**解决方案**：
- 联系数据架构师评审
- 通过后进入标准库
- 标准发布后执行

## 8. 方法论沉淀

### 8.1 数据标准建设方法论

```
标准建设成熟度模型：

L1 初始级
- 无统一标准
- 各团队自行其是
- 标准意识薄弱

L2 基础级
- 有基本命名规范
- 部分系统执行
- 标准文档存在

L3 规范级
- 完整标准体系
- 检查机制建立
- 大部分数据合规

L4 量化级
- 标准全覆盖
- 自动化检查
- 持续优化机制

L5 智能级
- AI 辅助标准制定
- 智能检查和修复
- 预测性合规
```

### 8.2 标准执行策略

```
执行策略选择：

1. 强执行（推荐）
   - 检查作为上线卡点
   - 违规阻断流程
   - 适用于核心数据

2. 弱执行
   - 检查但不阻断
   - 警告提醒
   - 适用于历史数据整改

3. 引导执行
   - 提供建议但不强制
   - 激励合规行为
   - 适用于推广阶段

执行节奏：
- 新增数据：强制检查，100% 合规
- 存量数据：分批整改，逐步达标
```

### 8.3 工具流程

**数据标准工具链**：
```
制定 → 发布 → 执行 → 检查 → 整改 → 优化

工具：
- 制定：标准管理平台
- 发布：标准库 + 消息通知
- 执行：IDE 插件 + CI/CD 集成
- 检查：规则引擎 + 定时任务
- 整改：整改工单系统
- 优化：分析工具 + 评审机制
```
