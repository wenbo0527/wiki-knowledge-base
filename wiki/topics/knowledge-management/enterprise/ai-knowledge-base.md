## 传统知识库的局限

| 局限 | 说明 | 影响 |
|------|------|------|
| **搜索依赖关键词** | 无法理解语义 | 找不到相关内容 |
| **需要人工维护** | 更新不及时 | 知识过时 |
| **被动检索** | 人找知识 | 效率低 |
| **孤立存储** | 不知道知识之间的关系 | 无法推理 |

## 技术架构：RAG

### 什么是RAG？

**RAG (Retrieval-Augmented Generation)** = 检索 + 生成

```
用户问题
    │
    ▼
┌─────────────────┐
│  检索 (Retrieval) │
│                   │
│  1. 理解问题      │
│  2. 召回相关文档  │
│  3. 重排序       │
└────────┬──────────┘
         │
         ▼
┌─────────────────┐
│  增强 (Augment)  │
│                   │
│  拼接Context     │
│  添加参考        │
│  格式化输出      │
└────────┬──────────┘
         │
         ▼
┌─────────────────┐
│  生成 (Generate) │
│                   │
│  LLM基于Context   │
│  生成答案        │
└─────────────────┘
```

### RAG组件

| 组件 | 技术 | 说明 |
|------|------|------|
| **Embedding模型** | OpenAI, BGE, Jina | 把文本变成向量 |
| **向量数据库** | Pinecone, Milvus, Qdrant | 存储和检索向量 |
| **搜索引擎** | Elasticsearch, ES | 全文检索 |
| **LLM** | GPT-4, Claude, Qwen | 生成答案 |
| **编排框架** | LangChain, LlamaIndex | 组装各组件 |

### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                 企业级RAG架构                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  数据源层                                                     │
│  ├── 飞书文档 / Confluence                                   │
│  ├── 数据库（结构化数据）                                    │
│  ├── 邮件和聊天记录                                          │
│  └── 业务系统API                                            │
│                                                              │
│  数据处理层                                                   │
│  ├── 文档解析（PDF、Word、HTML）                             │
│  ├── 文本分块（Chunking）                                    │
│  ├── 向量化（Embedding）                                     │
│  └── 元数据提取                                             │
│                                                              │
│  存储层                                                       │
│  ├── 向量数据库（语义检索）                                  │
│  ├── 全文索引（关键词检索）                                  │
│  └── 知识图谱（关系检索）                                    │
│                                                              │
│  应用层                                                       │
│  ├── 智能问答机器人                                          │
│  ├── 知识推荐系统                                            │
│  ├── 文档助手                                               │
│  └── 客服辅助                                               │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 关键配置

| 配置项 | 说明 | 推荐值 |
|--------|------|--------|
| **Chunk大小** | 单个知识块的大小 | 500-1000 tokens |
| **Chunk重叠** | 块之间的重叠 | 50-100 tokens |
| **检索数量** | 召回的相关文档数 | Top 5-10 |
| **重排序** | 对召回结果重新排序 | BGE-Ranker |

## 知识图谱增强RAG

### 为什么需要知识图谱？

| 能力 | 纯RAG | 知识图谱+RAG |
|------|-------|--------------|
| **实体关系理解** | ❌ | ✅ |
| **多跳推理** | ❌ | ✅ |
| **可解释性** | 中 | 高 |
| **复杂查询** | 弱 | 强 |

### 实现方式

```
问题：谁是基于Transformer架构的模型提供商？

纯RAG流程：
1. 检索包含"Transformer"、"模型"的文章
2. 可能找到混乱的结果

知识图谱+RAG流程：
1. 知识图谱查询：
   - 找到"Transformer"这个架构
   - 找到使用该架构的"模型"实体
   - 找到这些模型的"提供商"实体
2. RAG检索这些实体的详细文档
3. 综合生成答案
```

## 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **答案不准确** | 检索召回不相关 | 优化Embedding、添加重排序 |
| **幻觉严重** | LLM自由发挥 | 增加Context、限制输出格式 |
| **响应慢** | 链路太长 | 缓存、热数据优化 |
| **知识不更新** | 同步不及时 | 增量更新、事件触发 |
| **私有化部署** | 数据安全要求 | 开源模型+私有部署 |

## 最佳实践

### DO

1. **从简单开始** — 先用纯RAG，效果不好再加知识图谱
2. **重视数据质量** — 垃圾进垃圾出
3. **做好评估** — 建立评测集，持续优化
4. **混合检索** — 向量+关键词+知识图谱组合
5. **控制Context** — 太多不相关的Context会影响效果

### DON'T

1. **不要盲目追求大模型** — 小模型+好检索也能有好效果
2. **不要忽视数据治理** — 先清洗数据再上RAG
3. **不要忽视用户体验** — 答案要有出处，方便验证
4. **不要一次性做太多** — 选一个场景先做深
5. **不要忘记监控** — 上线后要持续看效果数据
