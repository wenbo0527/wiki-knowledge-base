## 1. 产品总览

### 1.1 一句话定位
**企业数据资源的统一注册、发现和使用平台，让数据资源像商品一样可被搜索、申请和使用。**

### 1.2 产品定位 & 核心价值

| 维度 | 描述 |
|:---|:---|
| **目标用户** | 数据分析师、数据工程师、业务人员、数据管理员 |
| **核心痛点** | 不知道企业有哪些数据、找不到需要的数据、申请数据流程长 |
| **解决方案** | 统一数据资源目录 + 智能搜索推荐 + 自助申请流程 |
| **核心价值** | 数据发现效率提升 10 倍，数据使用门槛大幅降低 |

### 1.3 核心业务流程

```
┌─────────────────────────────────────────────────────────────────┐
│  数据资源提供方                                                  │
│  - 数据库表、API、文件等资源注册                                  │
│  - 资源元信息填写（名称、描述、字段、更新频率）                  │
│  - 资源访问权限配置                                             │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  数据资源目录中心                                                │
│  - 统一资源注册和管理                                           │
│  - 资源分类和标签体系                                           │
│  - 资源质量评分和热度排行                                       │
│  - 智能搜索和推荐                                               │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  数据资源消费方                                                  │
│  - 搜索发现需要的资源                                           │
│  - 查看资源详情和样例数据                                       │
│  - 提交资源使用申请                                             │
│  - 审批通过后获取数据                                           │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 功能结构（按模块归档）

### 2.1 FEATURE-DFD_RESOURCE_REGISTER（数据资源注册）

#### 功能入口 & 作用

**入口位置**：数据资源中心 → 资源管理 → 注册资源

**核心作用**：
- 提供数据资源的标准化注册流程
- 收集资源的元信息（名称、描述、字段、更新策略等）
- 建立资源的分类和标签体系
- 配置资源的访问权限和审批流程

#### 核心逻辑

```
资源注册流程：

Step 1: 选择资源类型
  - 数据库表（MySQL、Hive、ClickHouse 等）
  - API 接口（REST、GraphQL）
  - 文件资源（CSV、JSON、Parquet 等）
  - 消息队列（Kafka、RabbitMQ）

Step 2: 填写基础信息
  - 资源名称（唯一标识）
  - 资源描述（业务含义）
  - 所属业务域（销售、用户、商品等）
  - 数据负责人
  - 更新频率（实时、小时、天、周）

Step 3: 配置技术信息
  - 数据库表：连接信息、表名、Schema
  - API：URL、Method、认证方式、参数定义
  - 文件：存储路径、格式、分隔符
  - 消息队列：Topic、分区、消费组

Step 4: 定义字段信息
  - 字段名、数据类型、长度、精度
  - 业务含义、示例值
  - 是否敏感、脱敏规则
  - 是否主键、是否可空

Step 5: 配置分类标签
  - 一级分类（业务域）
  - 二级分类（业务模块）
  - 三级分类（业务场景）
  - 自定义标签

Step 6: 设置权限和审批
  - 可见范围（公开、指定部门、指定人员）
  - 申请审批流程（自动通过、人工审批）
  - 审批人设置
  - 使用期限限制

Step 7: 预览和提交
  - 预览注册信息
  - 保存草稿或提交审核
  - 审核通过后正式发布
```

#### 关键配置项

| 配置项 | 说明 | 示例 |
|:---|:---|:---|
| `resource.name` | 资源名称（唯一） | `user_order_detail` |
| `resource.type` | 资源类型 | `DATABASE_TABLE` |
| `resource.domain` | 所属业务域 | `SALES_DOMAIN` |
| `resource.owner` | 数据负责人 | `张三` |
| `resource.update_frequency` | 更新频率 | `DAILY` |
| `resource.visibility` | 可见范围 | `DEPARTMENT` |
| `resource.approval_flow` | 审批流程 | `MANUAL_APPROVAL` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **资源名称冲突** | 不同业务域可能存在同名资源 | 使用命名规范：`{业务域}_{资源名}` |
| **字段类型映射** | 不同数据库类型映射不一致 | 建立统一类型映射表 |
| **敏感字段识别** | 敏感字段容易遗漏 | 建立敏感字段关键词库，自动识别提示 |
| **审批流程复杂** | 不同资源需要不同审批流程 | 支持审批流程模板配置 |
| **资源变更同步** | 源数据结构变更后元信息不同步 | 建立变更检测和提醒机制 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| EPIC-DFD_DATA_ASSET | 依赖 | 数据资产字典提供基础资产信息 |
| 数据质量中心 | 调用 | 注册时进行基础质量检查 |
| 权限管理中心 | 调用 | 资源的权限配置和校验 |
| 审批流程引擎 | 调用 | 资源注册的审批流程 |

### 2.2 FEATURE-DFD_RESOURCE_CATALOG（数据资源目录）

#### 功能入口 & 作用

**入口位置**：数据资源中心 → 资源目录

**核心作用**：
- 提供统一的数据资源分类浏览和检索
- 支持多维度的资源发现和推荐
- 建立资源的热度和质量评价体系
- 支持资源的收藏和关注

#### 核心逻辑

```
目录浏览逻辑：

Step 1: 选择浏览维度
  - 按业务域（销售、用户、商品、订单等）
  - 按资源类型（数据库表、API、文件等）
  - 按使用场景（分析、开发、测试等）
  - 按标签（自定义标签体系）

Step 2: 目录树导航
  - 展开/收起目录层级
  - 显示每个目录下的资源数量
  - 显示目录下的子目录

Step 3: 资源列表展示
  - 卡片视图：缩略信息、热度指标、质量评分
  - 列表视图：详细字段、排序筛选
  - 快捷操作：申请、收藏、分享

Step 4: 资源详情查看
  - 基础信息：名称、描述、负责人
  - 技术信息：类型、位置、更新频率
  - 元信息：字段定义、样本数据
  - 使用信息：申请记录、使用统计
  - 评价信息：评分、评论
```

#### 关键配置项

| 配置项 | 说明 | 示例 |
|:---|:---|:---|
| `catalog.hierarchy` | 目录层级配置 | `业务域 > 业务模块 > 业务场景` |
| `catalog.display_mode` | 默认展示模式 | `CARD` |
| `catalog.page_size` | 分页大小 | `20` |
| `catalog.sort_default` | 默认排序 | `HEAT_DESC` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **目录层级过深** | 用户浏览路径过长 | 限制最大层级为 4 级，提供面包屑导航 |
| **资源分类争议** | 同一资源可能属于多个分类 | 支持多分类标签，主分类+辅助分类 |
| **搜索结果为空** | 用户搜索不到想要的资源 | 提供相似推荐、热门搜索、搜索建议 |
| **资源信息不完整** | 部分资源缺少描述和标签 | 建立资源完整性评分，提示补充信息 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| FEATURE-DFD_RESOURCE_SEARCH | 关联 | 目录浏览与搜索功能互补 |
| FEATURE-DFD_RESOURCE_APPLY | 下游 | 目录中找到资源后进行申请 |
| 推荐系统 | 调用 | 个性化资源推荐 |

### 2.3 FEATURE-DFD_RESOURCE_SEARCH（数据资源搜索）

#### 功能入口 & 作用

**入口位置**：全局搜索栏 / 数据资源中心 → 搜索

**核心作用**：
- 提供强大的资源搜索能力，支持多维度检索
- 支持自然语言搜索和语义理解
- 提供搜索建议和热门搜索
- 支持搜索历史和个人化推荐

#### 核心逻辑

```
搜索处理流程：

Step 1: 搜索意图识别
  - 关键词提取
  - 实体识别（资源名、字段名、标签等）
  - 意图分类（精确查找、模糊探索、范围筛选）

Step 2: 搜索执行
  - 全文检索（名称、描述、字段）
  - 标签匹配
  - 分类筛选
  - 语义匹配（同义词、近义词）

Step 3: 结果排序
  - 相关性评分
  - 热度加权（浏览量、申请量）
  - 质量加权（完整性、准确性）
  - 个性化加权（历史行为、角色）

Step 4: 结果展示
  - 高亮显示匹配内容
  - 显示资源关键信息卡片
  - 支持快速筛选和排序
  - 提供相关搜索建议
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `search.min_keyword_length` | 最小关键词长度 | `2` |
| `search.max_results` | 最大返回结果数 | `100` |
| `search.highlight_enabled` | 是否高亮显示 | `true` |
| `search.suggestion_enabled` | 是否启用搜索建议 | `true` |
| `search.history_enabled` | 是否记录搜索历史 | `true` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **搜索无结果** | 用户输入的关键词匹配不到内容 | 提供相似词建议、自动纠错、热门搜索推荐 |
| **搜索结果过多** | 关键词过于宽泛返回大量结果 | 引导用户增加筛选条件、提供高级搜索 |
| **搜索结果排序不合理** | 用户想要的资源不在前面 | 优化排序算法、增加个性化权重 |
| **搜索性能慢** | 大数据量下搜索响应慢 | 建立搜索索引、使用搜索引擎（Elasticsearch） |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| FEATURE-DFD_RESOURCE_CATALOG | 关联 | 搜索与目录浏览互补 |
| 搜索引擎 | 依赖 | 需要 Elasticsearch 等搜索引擎支持 |
| 推荐系统 | 调用 | 个性化搜索结果排序 |

## 4. 数据相关

### 4.1 核心指标口径

| 指标 | 定义 | 计算逻辑 | 目标值 |
|:---|:---|:---|:---:|
| **资源注册数** | 已注册的数据资源总数 | `count(registered_resources)` | > 1000 |
| **资源覆盖率** | 已注册资源 / 实际资源总数 | `registered / total * 100%` | > 80% |
| **搜索成功率** | 有结果的搜索 / 总搜索次数 | `searches_with_results / total_searches` | > 90% |
| **平均搜索响应时间** | 搜索请求到结果返回的平均时间 | `avg(response_time)` | < 500ms |
| **资源申请转化率** | 申请次数 / 浏览次数 | `applications / views * 100%` | > 20% |
| **权限审批时效** | 申请提交到审批完成的平均时间 | `avg(approval_duration)` | < 4h |

### 5.3 审批流程配置

```
标准审批流程：

申请人提交申请
    │
    ▼
┌─────────────────┐
│ 自动审批检查    │ ──→ 符合条件的自动通过
│ (规则引擎)      │
└─────────────────┘
    │
    ▼ (需要人工审批)
┌─────────────────┐
│ 一级审批        │ ──→ 数据负责人审批
│ (数据负责人)    │
└─────────────────┘
    │
    ▼ (敏感资源)
┌─────────────────┐
│ 二级审批        │ ──→ 安全/合规负责人审批
│ (安全负责人)    │
└─────────────────┘
    │
    ▼
审批通过 → 发放访问凭证 → 申请人使用
```

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: 资源搜索不到怎么办？

**排查步骤**：
1. 检查关键词是否正确
2. 尝试使用分类浏览
3. 检查是否有权限限制
4. 联系数据管理员确认资源是否存在

**解决方案**：
- 使用模糊搜索
- 通过分类目录浏览
- 申请资源权限

#### Q: 资源申请被拒绝了怎么办？

**排查步骤**：
1. 查看拒绝原因
2. 检查申请信息是否完整
3. 确认是否有权限申请该资源

**解决方案**：
- 补充申请理由
- 联系数据负责人沟通
- 申请更低权限的数据

#### Q: 资源元信息不准确怎么办？

**排查步骤**：
1. 检查元信息填写是否完整
2. 对比实际数据验证
3. 查看是否有更新记录

**解决方案**：
- 提交元信息修改申请
- 联系数据负责人更新
- 建立元信息定期审核机制

## 8. 方法论沉淀

### 8.1 数据资源管理方法论

**资源生命周期管理**：
```
资源规划 → 资源注册 → 资源审核 → 资源发布 → 资源使用 → 资源监控 → 资源下线
```

**资源价值评估模型**：
```
资源价值 = 使用频次 × 数据质量 × 业务重要度 × 稀缺性
```

**资源分类体系设计原则**：
1. **MECE 原则**：相互独立，完全穷尽
2. **业务导向**：按业务域划分，贴近用户认知
3. **可扩展性**：预留分类空间，支持未来扩展
4. **稳定性**：分类体系一旦确定，不宜频繁变更

### 8.2 资源目录设计最佳实践

**目录层级设计**：
- 建议 3-4 层，不超过 5 层
- 每层节点数建议 5-15 个
- 避免过深（用户迷失）和过宽（选择困难）

**目录命名规范**：
- 使用业务语言，避免技术术语
- 名称简洁明了，2-6 个字
- 同一层级命名风格统一

### 8.3 资源搜索优化策略

**搜索质量提升**：
1. **同义词库**：建立业务同义词映射
2. **拼写纠错**：支持模糊匹配和纠错建议
3. **智能提示**：搜索框自动补全和热门推荐
4. **个性化排序**：基于用户历史行为调整排序

**搜索性能优化**：
1. **搜索引擎**：使用 Elasticsearch 等专业搜索引擎
2. **索引优化**：合理设计索引结构，避免过多字段索引
3. **缓存策略**：热点搜索缓存，减少重复计算
4. **分页加载**：大数据量分页，避免一次性返回过多结果

### 8.4 工具流程

**资源管理工具链**：
- **元数据管理**：Apache Atlas、DataHub
- **搜索引擎**：Elasticsearch、Apache Solr
- **流程引擎**：Camunda、Activiti
- **图数据库**：Neo4j（血缘关系）

**自动化工具**：
- **资源发现**：自动扫描数据库、API 网关
- **元数据采集**：自动采集表结构、字段信息
- **质量检查**：自动化数据质量规则检查
- **影响分析**：自动分析资源变更影响范围
