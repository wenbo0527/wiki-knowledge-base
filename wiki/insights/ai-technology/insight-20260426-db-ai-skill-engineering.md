# 数据库AI Skill工程化：从Demo到可用的距离

## 📌 执行摘要

**核心观点**："一个Skill能跑起来，不等于它已经可用。"

Demo证明方向能跑，可用要求结果能被信任。从Demo到工程化，中间隔着**粒度边界、权限、安全、沙箱、审批、灰度、输入输出、建议边界和失败策略**这些看不见的工程化约束。

---

## 🔄 Demo vs 可用

| 维度 | Demo | 可用 |
|------|------|------|
| **目标** | 证明方向能跑 | 结果能被信任 |
| **数据** | 不知道表结构、索引、执行计划 | 完整上下文 |
| **输出** | 方向性建议 | 可进入决策链路的结论 |
| **风险** | 无保护 | 全链路安全可控 |

---

## 🏗️ 分层架构：SOP ≠ Skill

### 五层架构

```
现象/告警 → 根因分类 → SOPTemplate → HealingPlan → HealingStep → Action → Skill/Executor
```

### 层级对照

| 层级 | 作用 | 慢查询场景例子 |
|------|------|----------------|
| **SOPTemplate** | 处理模板 | 慢查询分析流程 |
| **HealingPlan** | 本次实例化计划 | 针对这条SQL的分析计划 |
| **HealingStep** | 分步动作 | 查结构、查索引、看执行计划 |
| **Action** | 标准动作定义 | parse_sql、read_schema、read_indexes |
| **Skill/Executor** | 动作执行实现 | 解析SQL、读取元数据、分析执行计划 |

### 一句话概括

> **SOP是剧本，Plan是本次安排，Step是分镜头，Action是标准动作，Skill是动作背后的执行能力。**

---

## ⚙️ 工程化9大要素

从Demo到可用，还需要考虑：

| 要素 | 说明 |
|------|------|
| **粒度边界** | Skill应该包多大？不能包整个SOP |
| **权限** | 谁可以调用？权限必须前移至数据源层 |
| **安全** | 执行前后的安全检查 |
| **沙箱** | 执行环境的隔离与保护 |
| **审批** | 高风险操作的人工审批节点 |
| **灰度** | 逐步放量，而非全量发布 |
| **输入输出** | 标准化接口设计 |
| **建议边界** | AI建议的适用范围与限制 |
| **失败策略** | 出错时的降级与恢复机制 |

---

## 💡 关键洞察

### Skill不是SOP

- **SOP** = 剧本，包含判断、分支、前置检查、执行步骤、验证步骤和回滚步骤
- **Action** = 其中某一步允许调用的标准动作
- **Skill/Executor** = Action背后的具体执行能力

### 平台口径 vs 工程口径

| 口径 | Skill指什么 |
|------|-------------|
| **平台口径** | 一个包含说明、工具和资源的能力目录 |
| **工程口径** | 某个标准Action背后的可复用执行能力 |
