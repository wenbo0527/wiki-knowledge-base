## 🆕 Harness Engineering（2026年新范式）

> 2026年AI工程圈最热话题 - "模型是CPU，Harness是操作系统"

### 三层进化框架

| 阶段 | 时间 | 核心关注 |
|------|------|----------|
| **Prompt Engineering** | 2022-2024 | 精心构造单次指令 |
| **Context Engineering** | 2025 | 为每个决策点动态构建上下文 |
| **Harness Engineering** | 2026年2月起 | 设计完整的控制系统 |

### 核心类比：OS for AI Agent

```
计算机组件     →     Agent对应物     →     角色
─────────────────────────────────────────────────
CPU           →     模型            →     原始算力
内存           →     上下文窗口      →     有限、易失的工作记忆
操作系统       →     Agent Harness  →     管理上下文、处理启动、提供标准驱动
应用程序       →     Agent          →     跑在操作系统上的用户逻辑
```

**核心洞察**：模型就是CPU，算力再强，没有操作系统也跑不起来。

### 实战数据

| 案例 | 数据 | 来源 |
|------|------|------|
| **OpenAI Codex** | 100万行代码，5个月，1500 PR，人类一行没写 | OpenAI博文 |
| **Stripe Minions** | 每周1300+ PR，无人值守Agent完成 | Stripe博客 |
| **Cursor** | 每小时1000个commit，一周1亿次工具调用 | Cursor研究 |
| **Nate B Jones** | 换Harness后，同一模型编程成功率从42%→78% | 研究论文 |


### 七根杠杆

| # | 杠杆 | 关键实践 |
|---|------|----------|
| 1 | **AGENTS.md** | 控制在60行以内，写"目录"别写"百科全书" |
| 2 | **确定性约束** | linter + 结构化测试 + pre-commit hooks |
| 3 | **工具精简** | 工具越多越不准，Vercel从15砍到2准确率反而100% |
| 4 | **Sub-Agent隔离** | 防止上下文噪声累积，不同子任务用不同模型 |
| 5 | **反馈循环** | Agent自己验证产出，自己查日志、看截图 |
| 6 | **CI限速** | 最多两轮，第二轮失败直接转交人类 |
| 7 | **垃圾回收** | 定期扫描技术债、过时文档、架构漂移 |

### 关键语录

> **「Agent不难，Harness才难。」**
> — Ryan Lopopolo, OpenAI Codex团队

> **「每当你发现Agent犯了一个错误，你就花时间去工程化一个解决方案，让它再也不会犯同样的错。」**
> — Mitchell Hashimoto

> **「约束解空间反而让Agent更有生产力。」**
> — Cursor Self-Driving研究

### 行业实践

| 公司 | 做法 |
|------|------|
| **OpenAI** | 仓库是Agent唯一知识来源，AGENTS.md做目录，强制架构分层 |
| **Stripe** | Blueprint编排（确定性+Agentic节点混合），500工具但Agent只看到子集 |
| **Cursor** | 递归Planner-Worker模型，每小时1000 commit |
| **Vercel** | 工具从15砍到2，准确率80%→100% |

### Martin Fowler 三类 Harness

| 类型 | 目标 | 当前成熟度 | 关键实践 |
|------|------|------------|----------|
| **Maintainability Harness** | 代码质量、可维护性 | ✅ 最成熟 | 测试、linter、类型检查、结构分析 |
| **Architecture Fitness Harness** | 架构约束、边界检查 | 🟡 中等 | 模块边界、依赖规则、分层验证 |
| **Behaviour Harness** | 功能行为正确性 | 🔴 不成熟 | 功能测试、集成测试、验收标准 |

**核心洞察**：
- 计算型控制（linter/测试）便宜可靠，推理型控制（AI代码审查）昂贵但语义丰富
- **Keep Quality Left**：越早发现问题，修复成本越低

### 常见失败模式

| 失败模式 | 原因 | 解决方案 |
|----------|------|----------|
| **Agent重复犯同样错误** | 只有反馈，没有前馈 | 每次错误都要工程化解决方案 |
| **上下文噪声累积** | 长时间运行 | Sub-Agent隔离 + 定期清理 |
| **锁竞争** | 多Agent共享状态 | 锁机制优化或无锁设计 |
| **Agent过载** | 角色/任务过多 | 拆分职责，递归Planner-Worker |
| **指令漂移** | 长期任务失去焦点 | Freshness机制 + 定期重置 |

## 🆕 上下文压缩方法论（2026-03-31）

> 核心问题：上下文不是越长越好，有限注意力预算内让模型看到最关键信息

### 五家方案对比

| 产品 | 核心方案 | 关键创新 |
|------|----------|----------|
| **Manus** | Reduce/Offload/Isolate三板斧 | Compact双版本，可恢复压缩 |
| **Claude Code** | 三层防线 | Tool Result Clearing→Compaction→子Agent |
| **OpenClaw** | ContextEngine可插拔接口 | head+tail截断，30%窗口上限 |
| **Cursor** | Dynamic Context Discovery | 按需加载，token消耗减少46.9% |
| **LangChain** | 自主压缩 | Agent判断触发时机 |

### 管理优先级

```
从轻到重：
1. 管入口 ─── 少放不必要信息
2. 管工具 ─── 工具结果最大token源
3. 对话压缩 ─── Compaction / Summarization
4. 拆Agent ─── 上下文隔离
5. 文件系统 ─── 外部记忆兜底
```

### 关键认知

| 认知 | 说明 |
|------|------|
| **压缩本质** | 是注意力管理，不是数据压缩 |
| **KV Cache** | 被严重低估的优化维度，命中/未命中差10倍 |
| **可恢复原则** | 压缩≠删除，只是暂时移出视野 |

### OpenClaw关键参数

| 参数 | 值 |
|------|-----|
| 单工具结果上限 | 上下文窗口30% |
| 安全边距 | 1.2 (20%) |
| 摘要触发点 | 体量180K左右 |
| 最近对话 | 保留3轮不压缩 |
| 标识符保护 | UUID/路径原文保留 |

## 🆕 Agentic UI协议（2026-04-22）

> 来源：淘宝技术团队 | 卡片式对话的协议方案探索

### 核心问题

1. **卡片如何嵌入对话流？** - Markdown扩展方案
2. **卡片数据从何而来？** - 数据获取模式演进
3. **多团队协作怎么不乱？** - 四层统一协议体系

### 三种嵌入方案对比

| 方案 | 实现 | 优势 | 劣势 |
|------|------|------|------|
| **代码块扩展** | ```ProductCard {...}``` | 改造成本低 | JSON格式要求严 |
| **占位符替换** | [(ProductCard)] | 轻量 | 流式体验差 |
| **自定义标签** | `<boltArtifact>` | 表达力最强 | 需独立parser |

### 四层协议体系

```
Layer 1: Markdown标记协议 ── 卡片怎么写进文本流
Layer 2: 消息传输协议   ── 前后端传什么格式
Layer 3: UI渲染协议     ── 卡片长什么样(A2UI)
Layer 4: 事件通信协议   ── 用户点了我怎么办
```

### Agentic协议生态

| 协议 | 厂商 | 定位 |
|------|------|------|
| **MCP** | Anthropic | Agent↔外部工具 |
| **MCP Apps** | Anthropic | Tool返回数据+UI描述 |
| **A2UI** | Google | 声明式UI规范 |
| **A2A** | Google | Agent↔Agent协调 |
| **AG-UI** | Agent Protocol | Agent↔用户交互 |

### 角色重定义

| 传统 | → | Agent时代 |
|------|---|----------|
| 模型=文本生成器 | → | 模型=界面规划师 |
| 后端=数据提供者 | → | 后端=协议协调者 |
| 前端=页面渲染器 | → | 前端=协议执行引擎 |
