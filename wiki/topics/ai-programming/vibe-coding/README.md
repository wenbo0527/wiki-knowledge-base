## 一、核心定义

**Vibe Coding** = 规划驱动 + 上下文固定 + AI 结对编程，让「从想法到可维护代码」变成一条可审计的流水线。

> 核心理念：*规划就是一切。* 谨慎让 AI 自主规划，否则你的代码库会变成一团无法管理的乱麻。

**一句话本质**：与 AI 结对编程的终极工作流程，旨在帮助开发者将想法丝滑地变为现实。

---

## 二、🧭 道（核心理念）

| 原则 | 说明 |
|------|------|
| AI能做的，就不人工做 | 最大化AI作用 |
| 一切问题问AI | 把AI当伙伴而非工具 |
| 上下文是第一性要素 | 垃圾进，垃圾出 |
| 先结构，后代码 | 规划好框架，避免技术债 |
| 奥卡姆剃刀 | 如无必要，勿增代码 |
| 帕累托法则 | 关注重要的20% |
| 一次只做一件事 | 极致专注击穿代码 |
| 逆向思考 | 先明确需求，从需求逆向构建代码 |
| 重复多试 | 实在不行重新开个窗口 |

---

## 三、🧩 法（方法论）

| 方法 | 说明 |
|------|------|
| 一句话目标 + 非目标 | 明确边界 |
| 正交性 | 功能不要太重复 |
| 能抄不写 | 不重复造轮子，先问AI有没有合适的仓库 |
| 一定要看官方文档 | 先把官方文档喂给AI |
| 按职责拆模块 | 接口先行，实现后补 |
| 一次只改一个模块 | 降低复杂度 |
| 文档即上下文 | 不是事后补 |
| 代码一多就切会话 | 避免上下文膨胀 |

## 四、🛠️ 术（具体技巧）

### 1. 提示词编写
- **明确写清**：能改什么，不能改什么
- **Debug技巧**：只给「预期 vs 实际 + 最小复现」
- **测试**：交给AI，断言人审

### 2. 实施计划模板
```
## 实施计划

### 步骤1：[具体任务]
- **文件**：`src/xxx.py`
- **实现**：完成XXX功能
- **验证**：运行测试命令

### 步骤2：...
```

### 3. 工作流程
```
需求文档 → 技术栈选择 → CLAUDE.md/Agents.md → 实施计划 → 分步执行 → 验证 → 进度记录
```

### 4. 记忆库结构（Memory Bank）
```
memory-bank/
├── game-design-document.md   # 游戏设计文档（PRD）
├── tech-stack.md            # 技术栈
├── implementation-plan.md   # 实施计划（分步指令）
├── progress.md             # 进度记录
└── architecture.md         # 架构说明
```

## 五、📋 实践案例

### 案例1：Linus Torvalds的项目
- Python可视化部分基本是「vibe coding」写的
- Linus自己对模拟滤波器的了解都比Python多，但项目仍然成功

### 案例2：2万+行代码合并
- AI深度参与的商业项目
- 需要有技术背景的人来提出正确的问题

### 案例3：出海项目实践（2025年）
- 使用vibe coding + AI agent (coding agent)
- 经验：vibe coding是实践性极强的手艺，一切技巧全靠自己踩坑

### 案例4：AI写诗小玩具
- 从"解决午饭吃什么"的想法出发
- 最终跑偏到"AI写诗"的小玩具
- 体现了vibe coding的快速原型能力

## 六、⚠️ 常见陷阱与规避

| 陷阱 | 规避方法 |
|------|----------|
| AI自主规划导致代码混乱 | 先写文档，规划驱动 |
| 上下文膨胀 | 代码一多就切会话 |
| 忽略测试 | 测试交给AI，断言人审 |
| 过度设计 | 先基础功能，细节后面加 |
| 技术债积累 | 先结构后代码 |
| 功能重复 | 注意正交性设计 |
| 需求模糊 | 先提问澄清，不直接开做 |
