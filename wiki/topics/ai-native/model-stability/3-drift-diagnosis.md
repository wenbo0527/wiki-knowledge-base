## 漂移类型与根因分类
### 1. 数据源漂移（最常见，占60%以上）
**特征表现**：特征PSI大幅上升，但模型效果可能还没明显下降
**根因**：
- 数据源接口变更，字段统计口径变化
- 数据源服务器故障，数据延迟或丢失
- 第三方数据（如征信、运营商）服务不稳定
- 数据清洗逻辑变更
**处理**：优先排查数据源，确认变化原因，修复后重新计算特征
### 2. 客群漂移（次常见，占25%）
**特征表现**：整体PSI上升，特征分布整体偏移，但特征重要性排序不变
**根因**：
- 推广渠道变化，新客群和训练集差异大
- 营销活动带来大量新用户
- 季节性因素（节假日、假期）
- 宏观经济变化（疫情、经济下行）
**处理**：评估新客群和训练集的差异，决定是否需要用新客群数据重新训练模型
### 3. 特征逻辑漂移（占10%）
**特征表现**：某个特征的PSI突然大幅上升，其他特征稳定
**根因**：
- 特征计算逻辑错误（上线时没发现的bug）
- 特征依赖的原始数据发生了变化
- 特征计算的ETL任务失败或延迟
**处理**：检查特征计算逻辑，确认后修复，重新上线
### 4. 标签漂移（最严重，占5%）
**特征表现**：模型效果（KS、AUC）明显下降，但PSI可能稳定
**根因**：
- 逾期定义发生变化（比如宽限期调整）
- 催收策略变化，回收率变化
- 业务定义变化（比如vintage定义调整）
- 数据延迟，好坏用户标签不准确
**处理**：确认标签定义是否一致，修复后重新标注数据，重新训练模型

## 漂移诊断标准流程
### Step 1：确认漂移
- 检查模型输出PSI是否>0.25
- 检查KS、AUC是否明显下降
- 检查是否有P1/P0告警
**确认标准**：PSI>0.25或者KS下降>15%，确认模型已发生漂移，启动诊断流程
### Step 2：定位根因
```
诊断方向检查清单：
□ 1. 检查数据源
  - 数据源接口是否有变更公告
  - 数据源字段分布是否有变化
  - 数据源服务是否稳定
  - 特征计算任务的ETL日志是否有异常
□ 2. 检查客群
  - 最近是否有推广活动、新渠道
  - 新增用户的特征分布和历史用户差异
  - 是否有季节性因素
□ 3. 检查特征逻辑
  - 特征计算逻辑是否和训练时一致
  - 特征依赖的表是否有变更
□ 4. 检查标签
  - 逾期定义是否变化
  - 催收策略是否变化
  - 标签数据是否有延迟
```
### Step 3：特征级诊断
找到PSI最高的特征，逐一排查：
```
特征PSI排名分析：
1. 近7天交易金额 PSI=0.38
   - 检查数据源：支付宝接口近期有变更，统计口径从"支付成功"变为"发起支付"
   - 结论：数据源变更导致，需要修复特征计算逻辑
2. 信用卡使用率 PSI=0.22
   - 检查数据源：数据源稳定，无异常
   - 检查客群：新客群使用率略高
   - 结论：轻度漂移，可能是客群变化，继续观察
3. 逾期次数 PSI=0.08
   - 结论：稳定，无问题
```
### Step 4：制定处置方案
根据诊断结果，选择合适的处置方案：
| 漂移类型 | 处置方案 | 处置时间 |
|----------|----------|----------|
| 数据源漂移 | 修复数据源/特征计算逻辑 | 1-4小时 |
| 客群漂移 | 用新客群数据重新训练模型 | 1-3天 |
| 特征逻辑漂移 | 修复特征逻辑，重新上线 | 2-8小时 |
| 标签漂移 | 修复标签定义，重新标注，重新训练 | 3-7天 |

## 应急预案（当模型已无法使用时）
### 触发条件
- 输出PSI>0.5，或者KS<0.2，模型已严重失效
- 需要立即切换备用模型或者规则引擎
### 应急切换流程
```
1. 确认切换（5分钟内）
   - 确认模型已无法使用，PSI严重超标
   - 确认备用模型/规则引擎可用
   - 通知业务方即将切换

2. 执行切换（10分钟内）
   - 关闭失效模型，切换到备用模型/规则
   - 验证备用模型/规则输出正常
   - 确认业务不受影响

3. 监控过渡（切换后24小时）
   - 密切监控备用模型/规则的效果
   - 每小时检查一次PSI、KS等核心指标
   - 准备再次切换（如备用模型也有问题）

4. 问题复盘（切换后3天内）
   - 分析模型失效的根本原因
   - 制定修复方案
   - 编写复盘报告
   - 安排模型重新训练
```
### 备用模型要求
- 必须有至少一个备用模型，效果比当前模型略差但稳定可用
- 备用模型必须定期验证效果，不能放着不管
- 备用模型的切换必须经过测试，不能直接全量切换

## 修复后验证流程
模型修复或者重新训练后，必须验证才能重新上线：
### 灰度验证（1-2天）
- 先用10%的流量灰度新模型
- 监控PSI、KS等指标是否稳定
- 对比新旧模型的效果差异
### 全量上线（灰度验证通过后）
- 灰度验证通过后，全量切换到新模型
- 继续监控24小时，确认稳定
- 确认业务指标（坏账率）正常
### 上线后跟踪（上线后1周）
- 每天监控PSI、KS等核心指标
- 确认新模型效果稳定，没有再次漂移
- 输出上线报告

## 常见问题快速排查
| 问题现象 | 可能原因 | 快速排查方法 | 解决方向 |
|----------|----------|--------------|----------|
| PSI突然>0.5 | 数据源接口变更 | 检查数据源变更记录 | 修复特征计算或回滚数据源 |
| 特征PSI高，其他稳定 | 单个特征计算逻辑问题 | 检查该特征的计算逻辑 | 修复特征逻辑 |
| 所有特征PSI都高 | 客群变化 | 检查最近用户来源分布 | 用新客群数据重新训练 |
| PSI稳定但KS下降 | 标签定义变化 | 检查逾期定义是否变化 | 修复标签，重新训练 |
| 白天PSI高，晚上低 | 实时特征计算延迟 | 检查ETL任务延迟 | 优化特征计算管道 |
