## 模型评估流程
```
1. 数据集划分 → 训练集、验证集、测试集，保证分布一致
2. 选择评估指标 → 和业务目标对齐
3. 交叉验证 → 更稳定的评估结果
4. 基线对比 → 和简单模型、现有线上模型对比
5. 效果分析 → 坏案例分析，找到模型不足
6. 上线评估 → 线上A/B测试，最终验证效果
```

## 1. 数据集划分
### 1.1 划分方法
| 方法 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **随机划分** | 独立同分布数据，没有时间依赖 | 简单 | 不适合时序数据 |
| **时间划分** | 有时序依赖的数据（如风控、推荐） | 更接近真实线上场景 | 需要保证时间顺序，不能用未来数据 |
| **分层划分** | 不平衡数据集（如正负样本比例1:9） | 保证各个子集的类别比例和整体一致 | 实现略复杂 |

### 1.2 比例建议
| 数据集规模 | 训练集 | 验证集 | 测试集 |
|------------|--------|--------|--------|
| 小数据集（万级以下） | 60% | 20% | 20% |
| 中等数据集（十万/百万级） | 70% | 15% | 15% |
| 大数据集（千万级以上） | 90% | 5% | 5% |

### 1.3 避坑要点
- ✅ 测试集必须保持"干净"，只能用于最终评估，不能用于调参
- ✅ 时序数据必须按时间划分，不能随机打乱
- ✅ 划分后要检查三个集合的特征分布、目标分布是否一致
- ❌ 不能把测试集的数据混入训练集/验证集，会导致过拟合
- ❌ 不能用测试集的统计值做特征工程（如均值填充、标准化）

## 2. 评估指标选择
### 核心原则：指标必须和业务目标对齐，不要盲目追求AUC/准确率
### 2.1 分类任务
| 指标 | 公式 | 适用业务场景 |
|------|------|--------------|
| **准确率 (Accuracy)** | 预测正确数/总数 | 平衡数据集，误判漏判成本相同 |
| **精确率 (Precision)** | TP/(TP+FP) | 重视误判成本：如风控（误判好人为坏人会损失客户）、推荐（误推荐不感兴趣的内容会影响用户体验） |
| **召回率 (Recall)** | TP/(TP+FN) | 重视漏判成本：如反欺诈（漏判欺诈会损失钱）、癌症筛查（漏判癌症会出人命） |
| **F1 Score** | 2*P*R/(P+R) | P和R同等重要，需要均衡 |
| **AUC** | ROC曲线下面积 | 排序能力：如推荐系统、风控评分卡，不关心绝对阈值，只关心相对顺序 |
| **KS值** | max(TPR - FPR) | 风控领域常用，衡量正负样本的区分度，越大越好 |
| **业务指标** | 如：模型带来的营收提升、坏账率下降 | 最终看业务价值，算法指标是手段不是目标 |

### 2.2 回归任务
| 指标 | 适用场景 |
|------|----------|
| MAE | 对异常值鲁棒，关注平均误差 |
| MSE/RMSE | 对大误差惩罚大，关注极端误差 |
| R² | 模型的解释度，最大值1，越大越好 |

### 2.3 排序任务（推荐、搜索）
| 指标 | 适用场景 |
|------|----------|
| NDCG | 考虑排序位置的权重，位置越靠前权重越高 |
| MAP | 平均准确率，关注相关结果的排序 |
| 点击率/转化率 | 线上真实业务指标，最核心 |

## 3. 交叉验证 (Cross Validation)
### 3.1 常用方法
| 方法 | 适用场景 | 说明 |
|------|----------|------|
| **K折交叉验证** | 独立同分布数据 | 将数据分为K份，每次用K-1份训练，1份验证，重复K次，取平均结果 |
| **分层K折** | 不平衡分类任务 | 保证每折的类别比例和整体一致 |
| **时间序列交叉验证** | 时序数据 | 每次用历史数据训练，未来数据验证，不能用未来数据 |

### 3.2 最佳实践
- K值常用5或10，K越大评估结果越稳定，但训练时间越长
- 小数据集用K折，大数据集用简单的划分即可（因为数据多，划分的方差小）
- 最终模型用全量训练集训练，用交叉验证的平均结果作为评估

## 4. 调参方法
### 4.1 调参优先级
> 先调影响大的参数，再调影响小的参数，大部分参数用默认值即可
#### 以XGBoost/LightGBM为例，参数优先级从高到低：
1. **树结构参数**：max_depth（树深度，3-8）、num_leaves（叶子数，2^max_depth以内）
2. **学习率和迭代次数**：learning_rate（0.01-0.3，越小越好但迭代次数越多）、n_estimators（100-1000）
3. **采样参数**：subsample（样本采样比例，0.7-0.9）、colsample_bytree（特征采样比例，0.7-0.9）
4. **正则化参数**：reg_alpha（L1正则，0-1）、reg_lambda（L2正则，1-10）
5. **其他参数**：默认即可

### 4.2 调参方法
| 方法 | 说明 | 适用场景 |
|------|------|----------|
| **网格搜索 (Grid Search)** | 遍历所有参数组合 | 参数少、参数范围小的场景 |
| **随机搜索 (Random Search)** | 随机采样参数组合 | 参数多、参数范围大的场景，效果比网格搜索好 |
| **贝叶斯优化** | 用贝叶斯方法根据历史结果选择下一个参数组合 | 调参成本高的场景（如深度学习训练时间长），效率更高 |

### 4.3 最佳实践
1. **先粗后细**：先用大的参数范围快速找到最优区间，再在小区间内精细调参
2. **固定其他参数，每次只调一个参数**：更容易看到参数的影响
3. **不要过度调参**：在验证集上调参太多会导致过拟合验证集，线上效果下降
4. **参数调整的效果要显著**：如果调参只提升0.1个百分点AUC，不如花时间做特征工程
5. **优先调学习率和迭代次数**：learning_rate设小一点，n_estimators设大一点，用早停防止过拟合

### 4.4 早停 (Early Stopping)
**核心思想**：训练过程中监控验证集的效果，当效果连续N轮不再提升时停止训练，防止过拟合
**参数**：
- eval_metric：验证集评估指标（如AUC、logloss）
- early_stopping_rounds：连续多少轮不提升就停止（常用10-50）
**优点**：
- 自动控制迭代次数，不需要手动调n_estimators
- 防止过拟合，提升泛化能力
- 节省训练时间

## 5. 效果分析与迭代
### 5.1 坏案例分析
模型训练完之后，一定要分析预测错误的样本：
```
分析步骤：
1. 找出所有预测错误的样本（FP：假阳性、FN：假阴性）
2. 给这些样本打标签，分类分析错误原因：
   - 特征缺失：样本的关键特征缺失，导致预测错误
   - 特征不足：现有特征无法区分这类样本，需要加新特征
   - 标注错误：样本的标签本身就是错的
   - 模型缺陷：模型能力不足，需要调整模型/参数
3. 根据错误原因针对性优化
```
**例子**：风控模型中，FN（坏人被预测为好人）占比高，分析发现这些用户都是新用户，没有历史行为特征，那么解决方案就是新增新用户相关特征。

### 5.2 分维度评估
不要只看整体指标，要分维度评估模型效果：
- 用户维度：新用户/老用户、不同城市、不同年龄段
- 时间维度：不同月份、工作日/周末、节假日
- 业务场景：不同产品线、不同活动
**目的**：找到模型效果差的细分场景，针对性优化

### 5.3 基线对比
模型效果必须和基线对比才有意义：
| 基线类型 | 说明 |
|----------|------|
| 简单模型基线 | 逻辑回归、XGBoost默认参数，复杂模型效果必须显著优于简单模型 |
| 规则基线 | 现有业务规则的效果，模型效果必须优于规则 |
| 线上现有模型基线 | 新模型效果必须优于线上正在运行的模型，才有上线价值 |

## 6. 上线评估与监控
### 6.1 线上A/B测试
- 线下评估效果好的模型，必须经过线上A/B测试才能全量上线
- 分流要随机，保证对照组和实验组的用户特征一致
- 重点观测业务指标（如营收、转化率、坏账率），而不是只看算法指标
- 测试周期要足够长，覆盖不同时间段的用户行为

### 6.2 上线后监控
| 监控类型 | 监控指标 | 说明 |
|----------|----------|------|
| 特征监控 | 特征缺失率、分布变化、相关性变化 | 特征是模型效果的基础，特征变了模型效果肯定变 |
| 模型监控 | 预测值分布、AUC、KS、精确率、召回率 | 模型效果下降时及时报警 |
| 业务监控 | 核心业务指标变化 | 最终看业务价值 |

### 6.3 模型迭代
- 定期重新训练模型（如每月/每季度），适应数据分布变化
- 建立反馈闭环，线上的bad case回流到训练集，持续优化模型

---

## 避坑指南
1. **不要过度相信线下指标**：线下指标好只是必要条件，线上效果才是充分条件
2. **不要盲目追求高AUC/准确率**：指标高但业务价值低的模型没有意义，必须和业务对齐
3. **不要调参调过头**：大部分时候特征工程带来的提升远大于调参
4. **不要忽略样本权重**：不同样本的重要性不同，比如欺诈样本的重要性远高于正常样本，可以给欺诈样本更高的权重
5. **不要用准确率评估不平衡数据集**：正负样本比例1:99的情况下，全部预测为负样本准确率也有99%，但模型完全没用
