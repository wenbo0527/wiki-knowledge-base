## 定义

RAG (Retrieval-Augmented Generation) 是一种将**外部知识检索**与**LLM生成**结合的技术架构。2020年由Facebook提出（论文：Lewis et al., 2020）。

## 工作流程

```
用户查询 → 检索 → Top-K相关文档 → 上下文增强 → LLM生成 → 回答
```

## 核心组件

| 组件 | 说明 |
|------|------|
| **Retriever** | 检索器，从知识库找到相关文档 |
| **Vector Store** | 向量数据库，存储文档Embedding |
| **Generator** | LLM，基于上下文生成回答 |

## 向量数据库对比

| 数据库 | 特点 | 适用场景 |
|--------|------|----------|
| **Pinecone** | 云原生托管 | 快速上线 |
| **Weaviate** | 开源、灵活 | 私有部署 |
| **Milvus** | 大规模、高性能 | 大规模数据 |
| **Chroma** | 轻量、简单 | 原型/POC |
| **Qdrant** | 高性能 | 生产环境 |

## RAG的局限性

> 来自Karpathy的观点

Karpathy指出传统RAG的**核心问题**：

| 问题 | 说明 |
|------|------|
| **无积累** | 每次查询重新检索，没有知识积累 |
| **交叉引用缺失** | 无法建立实体间的关联 |
| **矛盾未标注** | 不同来源的矛盾信息未被处理 |
| **效率低下** | 相同事实每次都要重新发现 |

## RAG的演进

```
基础RAG → 高级RAG → 模块化RAG → Agent化RAG
    │          │          │          │
    ▼          ▼          ▼          ▼
  原始检索   优化检索   模块可插拔  动态决策
```

| 阶段 | 特征 | 适用场景 |
|------|------|----------|
| **基础RAG** | 直接检索+生成 | 简单问答 |
| **高级RAG** | 分块优化、rerank | 复杂文档 |
| **模块化RAG** | 模块可插拔 | 定制化需求 |
| **Agent化RAG** | Agent自主决策检索时机 | 开放任务 |

## RAG vs 微调 vs 提示工程

Karpathy的观点：选择取决于你的需求

```
需要注入新知识？
    ├─ 是 → RAG（知识更新频繁）
    │
    └─ 否 → 需要改变模型行为？
        ├─ 是 → 微调
        └─ 否 → 提示工程
```

| 方法 | 适用场景 | 成本 | 效果 |
|------|----------|------|------|
| **提示工程** | 通用任务、快速迭代 | 低 | 中 |
| **RAG** | 知识密集、需要实时信息 | 中 | 高 |
| **微调** | 特定行为、垂直领域 | 高 | 最高 |

## RAG + Agent = ?

Karpathy提出的**LLM Wiki模式**是RAG的进化：

| 维度 | 传统RAG | LLM Wiki |
|------|---------|----------|
| **知识状态** | 动态检索 | 持久积累 |
| **交叉引用** | 无 | 已建立 |
| **维护** | 无 | LLM持续维护 |
| **查询** | 每次重新综合 | 基于Wiki快速回答 |

## 最佳实践

| 实践 | 说明 |
|------|------|
| **分块策略** | 固定大小 / 语义 / 递归 |
| **Embedding选择** | BGE / OpenAI / Cohere |
| **检索优化** | hybrid search / rerank |
| **上下文窗口** | 控制token数量 |
