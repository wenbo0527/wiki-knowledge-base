## 什么是知识图谱？

### 定义

**知识图谱**是由Google在2012年提出的一种用图结构表示知识的方法。它由**实体（Nodes）**和**关系（Edges）**组成，可以表达"谁是什么"、"谁属于谁"、"谁做了什么"等复杂关系。

### 简单例子

```
知识图谱示例
    ┌──────────┐        创立        ┌──────────┐
    │  马斯克   │─────────────────▶│  Tesla   │
    └──────────┘                  └──────────┘
          │                              ▲
          │ CEO                         │ 投资
          ▼                             │
    ┌──────────┐        联合创始人    │
    │ JB Straus │──────────────────────┘
    └──────────┘
```

## 企业知识图谱的价值

### 核心价值

| 价值 | 说明 | 例子 |
|------|------|------|
| **知识关联** | 发现隐藏的关系 | "这个客户和那个供应商有关联" |
| **智能推理** | 从已知推未知 | "购买A产品的用户也会买B" |
| **搜索增强** | 理解语义而非关键词 | "找和产品X类似但更便宜的" |
| **异常检测** | 发现异常关系 | "这笔交易金额异常偏高" |

## 构建知识图谱

### 步骤

```
Step 1: 定义本体（Ontology）
    └── 确定要表示哪些实体和关系

Step 2: 数据采集
    ├── 结构化数据（数据库）
    ├── 半结构化数据（JSON/XML）
    └── 非结构化数据（文本）

Step 3: 实体抽取
    ├── 命名实体识别（NER）
    ├── 关系抽取
    └── 属性抽取

Step 4: 图谱构建
    └── 存入图数据库（Neo4j等）

Step 5: 应用开发
    ├── 知识检索
    ├── 推理计算
    └── 可视化展示
```

### 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| **存储** | Neo4j, Amazon Neptune, Tencent Graph | 图数据库 |
| **抽取** | LLM, SpaCy, Stanford NLP | NLP工具 |
| **推理** | RDF2Rules, Prolog | 逻辑推理 |
| **可视化** | D3.js, Cytoscape, Gephi | 图可视化 |
| **查询** | Cypher, Gremlin, SPARQL | 图查询语言 |

## 金融风控知识图谱案例

### 场景

```
                ┌─────────────────────────────────────┐
                │         金融风控知识图谱              │
                └─────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
   ┌─────────┐          ┌─────────┐          ┌─────────┐
   │  企业   │          │  人物   │          │  交易   │
   │ 实体    │          │  实体   │          │  实体   │
   └────┬────┘          └────┬────┘          └────┬────┘
        │                    │                    │
        │  控股关系          │  任职关系          │ 转账关系
        ▼                    ▼                    ▼
   ┌─────────┐          ┌─────────┐          ┌─────────┐
   │  企业   │          │  企业   │          │  交易   │
   │  (B)    │          │  (C)    │          │  (D)    │
   └─────────┘          └─────────┘          └─────────┘
```

### 识别风险

| 风险类型 | 图谱如何识别 |
|---------|--------------|
| **关联担保** | 发现企业A和企业B互为担保人 |
| **资金闭环** | 发现资金在多个账户间循环流动 |
| **虚假交易** | 发现多个企业之间交易价格异常 |
| **实际控制人** | 通过股权关系追溯最终控制人 |

## 知识图谱 + RAG

### 传统RAG的局限

| 局限 | 说明 |
|------|------|
| **语义鸿沟** | 检索结果可能不精准 |
| **关系缺失** | 不知道实体之间的关系 |
| **推理能力弱** | 无法进行多跳推理 |

### 知识图谱增强RAG

```
传统RAG
    用户问题
        │
        ▼
    Embedding检索
        │
        ▼
    拼接Context
        │
        ▼
    LLM生成答案

知识图谱增强RAG
    用户问题
        │
        ├──▶ 关键词提取
        │         │
        │         ▼
        │    图谱查询（实体关系）
        │
        ▼
    知识图谱 + 向量检索 合并
        │
        ▼
    拼接Context（包含关系信息）
        │
        ▼
    LLM生成答案（可解释）
```

## 最佳实践

### DO

1. **从业务问题出发** — 先明确要解决什么问题
2. **小步快跑** — 先构建核心知识图谱，逐步扩展
3. **数据质量优先** — 脏数据不如没有数据
4. **持续更新** — 知识图谱需要维护
5. **可视化验证** — 用可视化工具检查图谱质量

### DON'T

1. **不要追求一步到位** — 先有再优
2. **不要忽视数据治理** — 数据质量是基础
3. **不要过度设计本体** — 够用就好
4. **不要忘记更新机制** — 静态图谱会过时
5. **不要孤立运营** — 要和业务系统结合
