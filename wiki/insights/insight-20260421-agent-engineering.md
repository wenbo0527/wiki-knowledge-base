## 执行摘要

**核心洞察**: 构建生产级Agent系统的关键不是选择哪个模型或框架，而是**工程化能力**——包括工具设计、记忆架构、安全边界和评估体系。

**为什么重要**: 90%的Agent原型失败不是因为模型不够聪明，而是因为缺乏正确的Harness（工程脚手架）。

---

## 一、Agent系统的核心挑战

### 1.1 原型 vs 生产环境

| 维度 | 原型阶段 | 生产环境 |
|------|----------|----------|
| **调用频率** | 几次/天 | 千次/小时 |
| **错误处理** | 手动修复 | 自动降级 |
| **上下文** | 简单 | 复杂多轮 |
| **安全性** | 忽略 | 层层把关 |
| **可观测性** | 无 | 完整监控 |

### 1.2 Agent失败的主要原因

根据行业实践，Agent系统生产失败的前三大原因：

1. **工具设计缺陷** — 接口不符合模型认知习惯，导致调用失败
2. **上下文管理失控** — 内存泄漏、上下文膨胀
3. **缺乏安全边界** — 越权操作、数据泄露

### 2.1 工具调用 (Tool Use)

**核心原则**: 工具设计要符合模型的认知架构，而不是人类的编程习惯。

**ACI（Agent-计算机接口）设计准则**:

| 准则 | 说明 | 反例 |
|------|------|------|
| **自解释** | 返回结果包含足够上下文 | 只返回ID需要再查询 |
| **幂等性** | 同一调用多次结果一致 | 每次返回不同ID |
| **错误自愈** | 失败时能自动重试/降级 | 直接抛异常 |
| **最小依赖** | 工具间无隐式依赖 | A需要B先执行 |

### 2.2 记忆系统

**三层记忆架构**:

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 记忆系统                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  即时记忆 (Working Memory)                                   │
│  └── 当前对话窗口内的上下文                                 │
│      容量：模型上下文窗口限制                               │
│                                                              │
│  工作记忆 (Session Memory)                                   │
│  └── 当前会话内的重要信息                                   │
│      容量：需要主动压缩/摘要                                 │
│                                                              │
│  长期记忆 (Long-term Memory)                                │
│  └── 跨会话积累的知识和经验                                 │
│      存储：向量数据库/知识图谱                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 规划能力

**ReAct模式** (Reasoning + Acting):

```
问题 → 思考 → 行动 → 观察 → (循环直到完成)
```

**关键设计**: Agent需要有"知道自己不知道"的能力，遇到无法解决的问题时主动求助而非硬撑。

### 2.4 评估反馈

**四层评估体系**:

| 层级 | 内容 | 触发时机 |
|------|------|----------|
| **L1** | 语法/格式检查 | 每次输出 |
| **L2** | 业务逻辑验证 | 关键步骤 |
| **L3** | 安全/合规审核 | 敏感操作 |
| **L4** | 人工抽检 | 定期/异常 |

## 三、框架选型指南

### 主流框架对比

| 框架 | 定位 | 核心理念 | 最佳场景 |
|------|------|----------|----------|
| **LangChain** | 全能型 | 生态丰富、组件多 | 快速原型 |
| **LangGraph** | 可控型 | 精细流程控制 | 生产系统 |
| **AutoGen** | 多Agent | 对话驱动协作 | 复杂对话 |
| **CrewAI** | 多Agent | 角色+目标驱动 | 团队协作 |
| **MetaGPT** | 多Agent | SOP驱动 | 软件开发 |

### 选型决策树

```
需要构建Agent？
    │
    ├── 单Agent？
    │         ├── 快速验证 → LangChain
    │         └── 生产系统 → LangGraph
    │
    └── 多Agent？
              ├── 简单协作 → CrewAI
              ├── 复杂对话 → AutoGen
              └── 代码相关 → MetaGPT
```

### 框架选择建议

| 场景 | 推荐 | 原因 |
|------|------|------|
| **学习/原型** | LangChain | 文档丰富、上手快 |
| **生产单Agent** | LangGraph | 可控性强、调试方便 |
| **多Agent团队** | CrewAI | 概念清晰、角色分明 |
| **企业级复杂系统** | AutoGen + LangGraph | 灵活+可控 |

## 四、Harness Engineering

### 4.1 什么是Harness？

**定义**: Harness是Agent运行的完整工程环境，包括工具、格式、历史管理、错误处理等。

**类比**: 模型是发动机，Harness是底盘、悬挂、传动系统——决定了你能不能安全上路。

### 4.2 Harness的五个要素

| 要素 | 说明 | 常见问题 |
|------|------|----------|
| **工具（Tools）** | Agent可调用的功能 | 设计不符合模型习惯 |
| **信息格式（Formats）** | 输入输出的结构化方式 | 格式复杂导致解析失败 |
| **历史记录（History）** | 对话和操作的持久化 | 上下文膨胀失控 |
| **错误处理（Error Handling）** | 错误捕获与恢复 | 缺乏降级策略 |
| **脚手架（Scaffolding）** | 辅助开发的工具和模板 | 过度工程化 |

### 4.3 The Harness is the Dataset

**核心观点**: 在AI应用领域，竞争优势的关键不是模型本身，而是**捕获的轨迹数据**——也就是Harness运行时产生的数据。

这与互联网时代"数据是护城河"的观点一脉相承。

## 五、多Agent架构

### 5.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    分层Agent架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐                                            │
│  │  Manager   │ ← 任务分解、结果汇聚、质量把控              │
│  └──────┬──────┘                                            │
│         │                                                    │
│    ┌────┴────┐                                             │
│    ▼         ▼         ▼                                    │
│ ┌──────┐ ┌──────┐ ┌──────┐                                 │
│ │Worker │ │Worker │ │Worker │ ← 各司其职、专业分工          │
│ │搜索   │ │分析   │ │写作   │                               │
│ └──────┘ └──────┘ └──────┘                                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 协作模式

| 模式 | 复杂度 | 效果 | 适用场景 |
|------|--------|------|----------|
| **顺序** | 低 | 中 | 流程固定、依赖清晰 |
| **分层** | 中 | 高 | 管理+执行分离 |
| **辩论** | 高 | 最高 | 复杂决策、需要共识 |

### 5.3 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **群聊死锁** | Agent相互等待 | 超时机制 + 强制输出 |
| **信息丢失** | 传递不完整 | 结构化消息格式 |
| **状态不一致** | 视角不同 | 共享状态存储 |

## 九、关键结论

1. **Harness > Model**: 生产级Agent的瓶颈在工程化，不在模型能力
2. **工具设计是关键**: 工具接口要符合模型的认知习惯
3. **记忆需要管理**: 三层记忆架构 + 主动压缩
4. **安全不能事后补**: 安全边界要从设计阶段考虑
5. **评估驱动优化**: 没有度量就没有改进
