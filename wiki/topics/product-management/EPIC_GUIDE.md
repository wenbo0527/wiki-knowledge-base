# 1. 产品总览

## 1.1 一句话定位

**构建企业级产品需求管理体系，实现从业务需求到技术实现的智能拆解、结构化管理和全流程追踪。**

## 1.2 产品定位 & 核心价值

| 维度 | 描述 |
|:---|:---|
| **目标用户** | 产品经理、技术负责人、项目经理、开发人员、测试人员 |
| **核心痛点** | 需求拆解不系统、协作不透明、知识难沉淀、进度难追踪 |
| **解决方案** | 四层架构（业务入口→AI拆解→本体语义→图向量层）+ 双存储（Neo4j+飞书） |
| **核心价值** | 需求可追踪、协作可视化、知识可沉淀、决策智能化 |

## 1.4 核心角色 & 使用人群

| 角色 | 核心职责 | 使用场景 | 高频功能 |
|:---|:---|:---|:---|
| **产品经理** | 需求提出、Epic定义、验收确认 | 拆解业务需求、追踪Story状态、确认验收 | Epic创建、Story查看、状态更新 |
| **技术负责人** | 技术评估、Story拆解、进度把控 | 拆解Feature为Story、评估工作量、跟踪进度 | Story创建、工作量评估、进度看板 |

| **项目经理** | 全局把控、风险预警、资源协调 | 查看整体进度、识别延期风险、协调资源 | 进度看板、风险预警、资源分配 |

## 1.5 关键约束

| 约束类型 | 具体约束 | 应对策略 | 责任方 |
|:---|:---|:---|:---:|
| **数据安全** | Epic/Story 涉及业务敏感信息 | 飞书表格权限控制、Neo4j 本地部署、敏感数据脱敏 | 技术负责人 |
| **性能要求** | 飞书 API 调用频率限制（100次/秒） | 批量同步、本地缓存、增量更新、异步处理 | 开发团队 |
| **数据一致性** | Neo4j ↔ 飞书表格双向同步 | 以 Neo4j 为准、定时同步、冲突检测、人工介入 | 产品经理 |
| **可扩展性** | 支持多产品域、多 Epic 并行（目标 100+ Epic） | 模块化设计、标准化流程、模板复用、自动化工具 | 架构师 |
| **合规要求** | 数据血缘、审计追踪、变更日志 | 变更日志、操作审计、版本管理、定期审计 | 项目经理 |
| **用户体验** | 飞书表格响应 < 2s、Neo4j 查询 < 500ms | 索引优化、缓存策略、分页加载、异步渲染 | 开发团队 |

## 1.6 术语词典（本体论级别）

| 术语 | 英文 | 定义 | 示例 | 层级 |
|:---|:---|:---|:---|:---:|
| **产品域** | Product Domain | 产品的高层级分类，代表一个完整的业务领域，包含多个 Epic | PD-DFD（数据发现）、PD-MKT（数字营销） | L1 |
| **Epic** | Epic | 产品域下的业务模块，代表一个可独立交付的价值单元，包含多个 Feature | EPIC-DFD_DATA_ASSET（数据资产字典） | L2 |
| **Feature** | Feature | Epic 下的功能模块，代表一个内聚的功能单元，包含多个 Story | FEATURE-DFD_ASSET_OPERATE_TOOL（资产运营工具） | L3 |
| **Story** | User Story | 用户可感知的完整功能点，代表一个可独立交付的用户价值，可在一个迭代内完成 | STORY-ASSET_LIST_PAGE（资产列表页） | L4 |
| **故事点** | Story Point | 相对估算单位，表示完成 Story 所需的工作量，用于评估团队产能 | 1, 2, 3, 5, 8, 13 | - |
| **验收标准** | Acceptance Criteria | Story 完成必须满足的条件，用于测试验证，通常包含功能、操作、数据、交互 4 项 | 功能正常、操作正常、数据准确、交互符合预期 | - |
| **飞书表格** | Feishu Bitable | 飞书多维表格，用于需求状态同步和团队协作，支持多视图、权限控制、自动化 | Epic拆解确认-数据资产字典 | - |
| **四层架构** | Four-Layer Architecture | 产品管理体系的架构分层：业务需求入口 → AI 需求拆解 → 本体论语义 → 图与向量层 | - | - |
| **双存储** | Dual Storage | Neo4j（核心数据）+ 飞书表格（协作同步）的存储架构 | - | - |

# 2. 功能结构（按模块归档 - Epic 粒度）

## 2.1 PD-DFD 数据发现（Data Discovery）

### 2.1.1 模块总览

| 属性 | 描述 |
|:---|:---|
| **模块名称** | 数据发现（PD-DFD） |
| **一句话描述** | 企业级数据资产管理与运营平台，实现数据资产的"可见、可管、可用、可运营" |
| **核心目标** | 构建数据资产的核心枢纽，让数据价值最大化 |
| **覆盖范围** | 数据资产字典、数据要素字典、数据资源管理、统一搜索 |

### 2.1.2 Epic 清单

| Epic ID | Epic 名称 | 核心价值 | 主要功能 | 依赖关系 |
|:---|:---|:---|:---|:---|
| EPIC-DFD_DATA_ASSET | 数据资产字典 | 统一管理数据资产的元信息和生命周期 | 资产元数据管理、资产分类体系、资产血缘追踪 | 无 |
| EPIC-DFD_DATA_ELEMENT | 数据要素字典 | 数据要素的全生命周期管控 | 数据要素定义、数据标准管理、数据质量管理 | 依赖 DATA_ASSET |
| EPIC-DFD_DATA_RESOURCE | 数据资源管理 | 企业数据资源的统一注册、发现和使用 | 数据资源注册、资源目录、资源申请、资源使用 | 无 |
| EPIC-DFD_UNIFIED_SEARCH | 统一搜索 | 跨域数据资产的统一智能搜索入口 | 智能搜索引擎、个性化推荐、搜索分析与优化 | 依赖 DATA_ASSET |

### 2.1.3 关键术语

| 术语 | 定义 | 使用场景 |
|:---|:---|:---|
| 数据资产 | 企业拥有的、可带来价值的数据资源，包括数据本身及其元信息 | 资产目录、资产盘点 |
| 数据要素 | 可参与社会化生产、具有价值的数据资源，强调流通和交易属性 | 要素管理、标准制定 |
| 数据资源 | 企业可用的数据集合，包括内部数据和外部数据 | 资源注册、资源申请 |
| 元数据 | 描述数据的数据，包括技术元数据、业务元数据、管理元数据 | 元数据管理、血缘追踪 |

### 2.1.4 常见坑 & 注意点

| 坑点 | 影响 | 解决方案 |
|:---|:---|:---|
| 数据资产 vs 数据要素边界不清 | 功能重叠、用户混淆 | 明确边界：资产侧重管理，要素侧重流通 |
| 血缘关系维护成本高 | 血缘数据不完整、更新滞后 | 自动化采集 + 人工确认相结合 |
| 数据资源申请流程复杂 | 用户申请意愿低、使用率低 | 简化流程、提供自助服务、优化体验 |

### 2.1.5 关联系统

| 系统 | 关系 | 说明 |
|:---|:---:|:---|
| PD-MKT 数字营销 | 下游 | 数据资产为客群中心提供数据支持 |
| PD-RISK 数字风险 | 下游 | 数据资产为风控模型提供数据支持 |
| PD-DEX 数据探索 | 下游 | 数据资产为自助分析提供数据支持 |

## 2.2 PD-MKT 数字营销（Digital Marketing）

### 2.2.1 模块总览

| 属性 | 描述 |
|:---|:---|
| **模块名称** | 数字营销（PD-MKT） |
| **一句话描述** | 全渠道数字营销平台，实现精准用户触达和智能运营，驱动业务增长 |
| **核心目标** | 实现"千人千面"的个性化营销，成为企业数字化营销的核心引擎 |
| **覆盖范围** | 客群中心、营销活动管理、内容管理、触点管理、增长分析 |

### 2.2.2 Epic 清单

| Epic ID | Epic 名称 | 核心价值 | 主要功能 | 依赖关系 |
|:---|:---|:---|:---|:---|
| EPIC-MKT_CROWD_CENTER | 客群中心 | 精准人群定向和客群运营 | 客群创建与管理、客群洞察分析、客群分发与同步、客群效果追踪 | 依赖 PD-DFD 数据资产 |
| EPIC-MKT_CAMPAIGN_MGMT | 营销活动管理 | 全渠道营销活动的策划、执行和分析 | 活动策划、活动配置、活动执行、活动分析 | 依赖 CROWD_CENTER |
| EPIC-MKT_CONTENT_MGMT | 内容管理中心 | 营销内容的创作、管理和优化 | 内容库管理、内容创作、内容审核、内容效果 | 无 |
| EPIC-MKT_TOUCHPOINT_MGMT | 触点管理中心 | 多渠道触点的统一管理和智能路由 | 触点管理、消息推送、触达策略、触达分析 | 依赖 CAMPAIGN_MGMT |
| EPIC-MKT_GROWTH_ANALYSIS | 增长分析中心 | 数据驱动的增长运营和智能优化 | 增长模型、归因分析、LTV预测、增长实验 | 依赖所有上游 Epic |

### 2.2.3 关键术语

| 术语 | 定义 | 使用场景 |
|:---|:---|:---|
| 客群 | 基于特定规则筛选出的用户集合，用于精准营销 | 客群创建、客群分析、客群营销 |
| 标签 | 描述用户特征的标识，包括人口属性、行为特征、偏好等 | 标签体系、标签管理、标签应用 |
| 营销活动 | 有计划、有目标的营销动作，包括活动策划、执行、分析 | 活动管理、活动分析 |
| 触点 | 与用户交互的渠道和方式，包括短信、Push、微信、邮件等 | 触点管理、触达策略 |
| 增长模型 | 描述用户增长路径和关键指标的模型，如 AARRR | 增长分析、策略优化 |

### 2.2.4 常见坑 & 注意点

| 坑点 | 影响 | 解决方案 |
|:---|:---|:---|
| 客群规则过于复杂 | 计算性能差、用户等待时间长 | 优化规则引擎、预计算、分层缓存 |
| 多渠道触达冲突 | 用户体验差、过度打扰 | 全局频控、优先级策略、智能路由 |
| 营销活动效果难以归因 | 无法评估真实效果、资源浪费 | 多触点归因模型、A/B测试、数据追踪 |
| 内容审核不及时 | 合规风险、品牌损害 | 自动化审核 + 人工复核、快速响应机制 |

### 2.2.5 关联系统

### 2.3.4 常见坑 & 注意点

| 坑点 | 影响 | 解决方案 |
|:---|:---|:---|
| 模型效果波动大 | 误杀/漏杀率高、业务损失 | 模型监控告警、快速回滚、A/B测试 |
| 实时决策延迟高 | 用户体验差、交易失败 | 优化算法、扩容资源、缓存预热 |
| 合规规则更新频繁 | 系统改造成本高、上线延迟 | 规则引擎化、配置化、热更新 |
| 风险调查证据不足 | 无法定性、处置困难 | 全链路追踪、证据固化、审计日志 |

| EPIC-COM_MSG_CENTER | 消息通知中心 | 多渠道消息的统一接入和管理 | 消息聚合、消息分类、消息推送、消息管理 | 依赖 SEARCH_CENTER |
| EPIC-COM_APP_MARKET | 应用市场 | 开放的应用生态，支持内外部应用接入 | 应用商店、应用管理、应用集成、应用评价 | 依赖 MSG_CENTER |

### 2.4.3 关键术语

| 术语 | 定义 | 使用场景 |
|:---|:---|:---|
| 门户 | 企业统一的入口页面，集成各种应用和信息 | 门户设计、门户运营 |
| 个性化 | 基于用户特征和行为的定制化展示 | 个性化推荐、用户体验 |
| 企业搜索 | 跨系统、跨数据源的统一搜索能力 | 搜索中心、信息检索 |
| 应用市场 | 企业应用的发布、发现、安装平台 | 应用管理、生态建设 |
| 权限模型 | 定义用户访问资源的规则和策略 | 权限设计、安全控制 |

### 2.4.4 常见坑 & 注意点

| 坑点 | 影响 | 解决方案 |
|:---|:---|:---|
| 门户页面加载慢 | 用户体验差、跳出率高 | 性能优化、懒加载、CDN加速 |
| 个性化推荐不准确 | 用户满意度低、转化率差 | 算法优化、数据完善、A/B测试 |
| 搜索结果质量差 | 用户找不到所需信息 | 搜索算法优化、语义理解、人工标注 |
| 应用接入标准不统一 | 应用质量参差不齐 | 制定标准、审核机制、开发者文档 |
