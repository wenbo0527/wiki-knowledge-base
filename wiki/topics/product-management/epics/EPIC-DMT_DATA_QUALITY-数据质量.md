## 2. 功能结构（按模块归档）

### 2.1 FEATURE-DMT_QUALITY_RULES（质量规则引擎）

#### 功能入口 & 作用

**入口位置**：数据管理 → 数据质量 → 规则管理

**核心作用**：
- 管理数据质量检测规则
- 提供预置规则模板
- 支持自定义规则配置
- 规则执行和调度管理

#### 核心逻辑

```
质量规则配置流程：

Step 1: 选择检测对象
  - 选择数据库和表
  - 选择检测字段
  - 配置抽样比例（全量/抽样）

Step 2: 选择规则类型
  - 准确性：值域检查、格式检查、跨表一致性
  - 完整性：非空检查、唯一性检查
  - 一致性：同义词一致性、格式一致性
  - 时效性：数据延迟检查、更新频率检查
  - 唯一性：主键唯一性、重复记录检查

Step 3: 配置规则参数
  - 阈值设置（告警阈值、阻断阈值）
  - 告警方式配置
  - 执行计划配置

Step 4: 规则测试
  - 用历史数据测试规则
  - 验证规则准确性
  - 调整参数优化
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `rule.sample_rate` | 默认抽样比例 | `10%` |
| `rule.warn_threshold` | 告警阈值 | `95%` |
| `rule.block_threshold` | 阻断阈值 | `90%` |
| `rule.max_violations` | 最大记录问题数 | `10000` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **规则过于严格** | 轻微问题也触发告警 | 调整阈值、增加豁免机制 |
| **检测性能差** | 全量检测导致系统压力 | 抽样检测、增量检测 |
| **误报率高** | 正常数据被判定为问题 | 优化规则条件、人工确认 |
| **规则冲突** | 多规则对同一数据判定矛盾 | 规则优先级设置、互斥配置 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 元数据中心 | 上游 | 获取表和字段信息 |
| 调度中心 | 依赖 | 规则执行调度 |
| 告警中心 | 下游 | 质量告警通知 |

- 问题认领和分配
- 处理流程跟踪
- 根因分析和改进

#### 核心逻辑

```
质量问题处理流程：

Step 1: 问题发现
  - 自动检测发现
  - 用户手动上报
  - 周期性巡检

Step 2: 问题登记
  - 自动创建问题工单
  - 关联检测任务和规则
  - 初步影响评估

Step 3: 问题分派
  - 自动识别责任人
  - 手动调整分配
  - 升级机制

Step 4: 问题处理
  - 根因分析
  - 制定修复方案
  - 执行修复
  - 结果验证

Step 5: 问题关闭
  - 验证数据质量
  - 更新规则和阈值
```

## 4. 数据相关

### 4.1 核心指标口径

| 指标 | 定义 | 计算逻辑 | 目标值 |
|:---|:---|:---|:---:|
| **整体质量分** | 综合质量评分 | `Σ(维度分 × 权重) / Σ(权重)` | > 95 |
| **完整性** | 非空字段比例 | `非空数 / 总数 * 100%` | > 98% |
| **准确性** | 符合规则数据比例 | `符合数 / 检测数 * 100%` | > 95% |
| **一致性** | 跨表一致数据比例 | `一致数 / 总数 * 100%` | > 99% |
| **问题发现率** | 发现问题与实际问题比例 | `发现 / 实际 * 100%` | > 90% |
| **问题解决率** | 解决问题与发现问题比例 | `解决 / 发现 * 100%` | > 85% |
| **平均解决时长** | 问题从发现到解决的平均时间 | `avg(resolve_time)` | < 24h |

## 5. 规则与配置中心

### 5.1 质量维度定义

| 维度 | 定义 | 衡量方式 | 权重 |
|:---|:---|:---|:---|
| **准确性** | 数据与实际业务一致 | 准确性检测通过率 | 30% |
| **完整性** | 数据完整无缺失 | 非空率、主键完整性 | 25% |
| **一致性** | 跨系统数据一致 | 一致性检测通过率 | 20% |
| **时效性** | 数据及时更新 | 更新延迟、超时比例 | 15% |
| **唯一性** | 数据无重复 | 重复记录比例 | 10% |

### 5.2 质量规则模板

```
预置质量规则：

1. 基础规则
   - 非空检查：字段不能为空
   - 唯一性检查：字段值唯一
   - 格式检查：符合指定格式
   - 值域检查：在指定范围内

2. 高级规则
   - 跨表一致性：多表关联一致性
   - 业务规则：自定义业务逻辑
   - 历史对比：与历史数据对比
   - 趋势异常：数据趋势异常检测

3. 模板库
   - 用户表质量模板
   - 订单表质量模板
   - 财务表质量模板
```

### 5.3 告警规则

```
告警级别：

P0 阻断：
- 质量分 < 60
- 数据完全不可用
- 立即处理

P1 严重：
- 质量分 60-80
- 核心指标受影响
- 4 小时内处理

P2 警告：
- 质量分 80-90
- 轻微问题
- 24 小时内处理

P3 提示：
- 质量分 90-95
- 观察级别
- 工作日内处理
```

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: 质量分数突然下降怎么办？

**排查步骤**：
1. 查看质量大盘定位问题表
2. 查看具体是哪个维度出问题
3. 查看检测任务详情

**解决方案**：
- 定位问题后提交工单
- 联系责任人修复
- 调整检测规则

#### Q: 告警误报怎么办？

**排查步骤**：
1. 查看告警详情，确认误报
2. 分析误报原因
3. 调整规则阈值

**解决方案**：
- 调整告警阈值
- 优化规则条件
- 反馈误报情况

#### Q: 质量问题反复出现怎么办？

**排查步骤**：
1. 进行根因分析
2. 确认是否 ETL 逻辑问题
3. 制定长期解决方案

**解决方案**：
- 修改 ETL 逻辑
- 增加上游数据校验
- 建立预防机制

## 8. 方法论沉淀

### 8.1 数据质量评估模型

```
质量评估公式：

综合质量分 = Accuracy × 0.3 
           + Completeness × 0.25 
           + Consistency × 0.2 
           + Timeliness × 0.15 
           + Uniqueness × 0.1

等级划分：
- A（95-100）：优秀，数据完全可信
- B（85-94）：良好，数据基本可用
- C（70-84）：一般，数据需谨慎使用
- D（60-69）：较差，数据不可靠
- F（<60）：不合格，数据不可用
```

### 8.2 质量问题分析方法

```
5-Why 分析法：

问题：订单表中"下单时间"字段有 10% 为空

Why1：为什么为空？
    → 部分订单创建时时间未记录

Why2：为什么时间未记录？
    → 系统异常时未做兜底处理

Why3：为什么系统异常时未做兜底？
    → ETL 未考虑异常情况

Why4：为什么 ETL 未考虑？
    → 开发人员未做异常处理培训

Why5：为什么未培训？
    → 缺乏开发规范和流程

根本原因：缺乏 ETL 开发规范
解决方案：制定 ETL 开发规范，增加异常处理要求
```

### 8.3 工具流程

**数据质量管理工具链**：
```

工具：

- 问题发现：质量大盘 + 告警
- 问题处理：工单系统

```
