## 🕵️ LLM 执行指令（Schema）

> 这是给尼克·弗瑞的执行指令。严格按照以下规程执行。

### S1. 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                    Wiki 维护三原则                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1️⃣ 有价值才入库   → 不堆量，只存高价值内容                   │
│                                                              │
│  2️⃣ 有来源可追溯   → 每条知识必须标注来源                     │
│                                                              │
│  3️⃣ 持续保鲜      → 新来源触发增量更新，不全量重写           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### S2. Ingest 操作规程

> 当收到新的来源（RSS、最佳实践、手动提供）时执行。

#### S2.1 处理流程

```
┌─────────────────────────────────────────────────────────────┐
│                    Ingest 执行流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 1: 价值评估                                           │
│  └── 评估价值 ≥ ⭐⭐⭐ 才入库                                 │
│                                                              │
│  Step 2: 类型判断                                           │
│  ├── Insight → 写入 insights/                               │
│  ├── Topic → 写入 topics/ 或更新现有 Topic                   │
│  └── Entity → 写入 entities/ 或更新现有 Entity               │
│                                                              │
│  Step 3: 内容提取                                            │
│  ├── 提取核心信息（不是全盘复制）                            │
│  ├── 识别与现有知识的关联                                    │
│  ├── 检查是否有矛盾                                          │
│  └── 产出至少 1 条洞察                                        │
│                                                              │
│  Step 4: 写入 Wiki                                           │
│  ├── 创建/更新对应文件                                       │
│  ├── 添加双向链接                                            │
│  └── 更新元数据（updated, status）                           │
│                                                              │
│  Step 5: 记录日志                                            │
│  └── 在 log.md 添加条目（格式见 S6.1）                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### S2.2 矛盾检测

当新来源与现有知识冲突时：

```
┌─────────────────────────────────────────────────────────────┐
│                    矛盾处理规程                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  检测到矛盾时：                                              │
│  1. 在新内容中标记 [[CONFLICT: 与 xxx 矛盾]]                 │
│  2. 在被矛盾的现有页面添加注释 [[NOTE: 被 xxx 新来源质疑]]    │
│  3. 在 log.md 记录矛盾条目                                   │
│  4. 在文件头部添加 conflict_notes 字段                        │
│  5. 由派蒙后续决策保留哪个                                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### S2.3 增量更新 vs 全量重写

| 场景 | 策略 |
|------|------|
| 新来源补充现有 Topic | **增量更新**：只追加新内容段，不重写全文 |
| 新来源显著扩展 Topic | **扩展更新**：添加新的子章节 |
| 新来源颠覆性认知 | **新建 Insight**：独立文件，与原 Topic 关联 |
| 原页面严重过时 | **全量重写**：保留原内容到 sources/archived/ |

### S3. Query 操作规程

> 当被问到问题，需要基于 Wiki 回答时执行。

#### S3.1 查询流程

```
┌─────────────────────────────────────────────────────────────┐
│                    Query 执行流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 1: 读取 index.md                                       │
│  └── 快速定位相关页面                                        │
│                                                              │
│  Step 2: 读取相关页面                                        │
│  └── 获取完整上下文                                          │
│                                                              │
│  Step 3: 合成答案                                            │
│  └── 基于 Wiki 内容回答，引用来源                            │
│                                                              │
│  Step 4: 评估答案价值                                         │
│  ├── ⭐⭐⭐⭐⭐ 卓越答案 → 存回 Wiki 作为新 Insight           │
│  ├── ⭐⭐⭐⭐ 高价值答案 → 存回 Wiki 作为备注                │
│  └── ⭐⭐⭐及以下 → 直接回答                                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### S3.2 Query 答案沉淀规则

**好答案存回 Wiki 的判断标准**：

| 答案类型 | 示例 | 处理方式 |
|----------|------|----------|
| 独到洞察 | "XX行业的核心矛盾是..." | → 新建 `insight-{date}-query-xxx.md` |
| 对比分析 | "A vs B 的核心差异..." | → 新建 `topics/analysis/xxx-compare.md` |
| 实用总结 | "XX的实施框架..." | → 更新相关 Topic，标记来源 |
| 发现连接 | "XX和XX其实是一体两面" | → 新建 Insight，建立双向链接 |

**沉淀格式**：

```markdown
---
title: {答案主题}
type: insight
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: active
source: query      # ← 标记为 Query 产出
source_question: "{原始问题}"
maintainer: nick-fury
summary: {一句话概括}
---

## 💡 核心洞察

{答案内容}

## 📋 来源

原始问题：{原始问题}
回答时间：{时间}
参考知识：{相关 Insights/Topics}

## 🔗 关联

- [[关联知识1]]
- [[关联知识2]]
```

### S4. Lint 操作规程

> 当被触发 Lint 任务时执行。

#### S4.1 检查项目

| 检查项 | 方法 | 处理 |
|--------|------|------|
| 孤立页面 | 检查无 inbound links | 添加到相关 Topic 或标记孤立 |
| 过时内容 | 检查 updated > 3个月 | 标记 `[STALE]` |
| 矛盾信息 | 检查 `[[CONFLICT:...]]` | 报告派蒙决策 |
| 缺失链接 | 检查链接有效性 | 修复或删除死链 |
| 孤儿概念 | 检查 Concepts 未被引用 | 添加到相关 Topics |

#### S4.2 Lint 报告格式

```markdown
## 🩺 Wiki Lint 报告

执行时间：{YYYY-MM-DD HH:mm}
检查范围：{范围}

### 发现问题

| 问题类型 | 数量 | 详情 |
|----------|------|------|
| 孤立页面 | N | [列表] |
| 过时内容 | N | [列表] |
| 矛盾标记 | N | [列表] |
| 死链 | N | [列表] |

### 建议行动

- [ ] 行动1
- [ ] 行动2

### 整体健康度

{健康度评分}%（{评分说明}）
```

### S5. 文件格式规范

#### S5.1 元数据规范

```markdown
---
title: 标题
type: insight | topic | entity | concept | project
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: active | stale | archived | conflict
maintainer: nick-fury | zhongli | paimon | tony
tags: [tag1, tag2]
summary: 一句话概括（必需）
conflict_notes: |        # 当 status=conflict 时填写
  - 与 {文档} 矛盾：{简述}
  > 待派蒙决策
---
```

#### S5.2 来源标记

| 标记 | 类型 | 说明 |
|------|------|------|
| `[RSS]` | RSS简报 | 来自 RSS 情报系统 |
| `[BP]` | 最佳实践 | 来自最佳实践收集 |
| `[Manual]` | 手动输入 | 用户直接提供 |
| `[Query]` | Query产出 | 回答问题时产出 |
| `[Ref]` | 参考资料 | 外部参考资料 |

#### S5.3 双向链接格式

```markdown
## 🔗 关联知识

- [[topics/fintech/risk-management|风控专题]]
- [[entities/companies/ant-group|蚂蚁集团]]
- [[insight-20260416-ai-trend|AI趋势洞察]]
```

### S6. 日志规范

#### S6.1 log.md 格式

```markdown
## [{YYYY-MM-DD}] {操作类型} | {标题}

**操作**：{操作类型}（ingest/query/lint/update/archive）
**来源**：{来源}
**产出**：{产出描述}
**关联**：{相关文件列表}

{详细说明}
```

**操作类型前缀**：
- `## [2026-04-17] ingest | Article Title`
- `## [2026-04-17] query | Question Title`
- `## [2026-04-17] lint | 健康检查`
- `## [2026-04-17] update | Topic Title`
- `## [2026-04-17] archive | 文档 Title`

**快速查询命令**：
```bash
grep "^## \[" log.md | tail -10    # 最近10条
grep "^## \[2026-04" log.md         # 本月全部
grep "ingest" log.md | wc -l        # Ingest 统计
```

#### S6.2 index.md 维护规程

**index.md 位置**：`/Users/wenbo/Documents/project/Wiki/wiki/index.md`

**更新时机**：
- 新建文件 → 添加条目
- 删除文件 → 移除条目
- 重大更新 → 更新摘要

**条目格式**：

```markdown
### {类别}

| 文件 | 摘要 | 来源 | 更新 |
|------|------|------|------|
| [insight-20260417-xxx](insights/insight-20260417-xxx.md) | 一句话概括 | RSS | 2026-04-17 |
```
