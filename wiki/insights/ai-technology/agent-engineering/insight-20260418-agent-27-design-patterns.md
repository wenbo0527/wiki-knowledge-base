## 💡 核心洞察

### 洞察1：设计模式在Agent系统中的核心价值

**设计模式的作用边界**：
- 设计模式无法直接生成大型复杂系统
- 但可以在编码过程中逐步指导思维
- 帮助拆解大型系统，识别可复用的设计节点

**核心定位**：
> 🚩**Harness比基础模型更工程师可控**
> 
> 我们无法管控大模型本身的精度，但可以通过设计Harness系统约束Agent的运行环境，获得更稳定的运行结果。

### 洞察2：Harness系统的核心设计模式分类

#### Agent循环核心模式
- **`Agent Take Loop`** 是Agent系统最重要的环节
- 必须包含**生成-评判模式**才能保证循环正常终止
- 所有Agent循环逻辑都遵循一致的模式

#### 工具与上下文模式
- **工具系统** → 对应**技能包分发模式**
- **上下文管理** → 对应**上下文分诊、传递、层级记忆模式**
  - 用于解决上下文过长污染问题
  - 区分长中短期不同权重的上下文做层级化管理

#### 治理（Governance）模式
- 权限控制
- **爆炸半径控制** ← *Agent时代新增的重要治理模式*
- 审批门户
- 状态持久化
- 失败日志
- 护栏三明治

### 洞察3：双轴正交设计框架

通过两个问题即可定位需要使用的设计模式：

```
1. 当前组件的功能分类：
   ├── 记忆(Memory)
   ├── 感知(Perception)
   ├── 推理(Reasoning)
   ├── 行动(Action)
   └── 治理(Governance)

2. 所需拓扑结构：
   控制数据流向的拓扑类型
```

定位后即可在双轴正交坐标中找到对应的设计模式。

## 🏗️ 主流Agent框架的模式拆解

### Cloud Code框架（深度交互型）

**定位**：面向深度人机交互场景，单线程一来一往的深度切磋，适合做深做透任务

**核心模块模式**：

| 模块 | 采用的设计模式 |
|------|---------------|
| **记忆模块** | 层级记忆模式（工作记忆→会话记忆→用户记忆→企业记忆→外部记忆） |
| **感知模块** | 上下文分诊路由模式 |
| **反思模块** | **生成-评判迭代循环模式**（修改代码→运行测试→迭代反馈→直到任务完成） |
| **治理模块** | **审批门模式**（自动执行/询问/拒绝三层审批） |

### Open Cloud框架（自动广度型）

**定位**：面向自动化自动运营场景，适合广度扩展的自动任务

**核心模块模式**：

| 模块 | 采用的设计模式 |
|------|---------------|
| **记忆模块** | 双层架构记忆 + 多层向量检索支持长期记忆 |
| **治理模块** | 会话级隔离实现爆炸半径控制 |
| **核心架构** | 多渠道Harness设计（网关路由/会话隔离/声明式配置） |

> 🚩**重点**：设计模式是跨平台通用的，不同框架只是实现细节有差异，底层设计模式完全一致。

## ⚙️ Agent系统的核心工程原理

### 1. 上下文隔离
类比分布式系统的进程隔离：
- 主Agent调用子Agent时，需将子Agent上下文与主上下文完全隔离
- 减少噪声干扰
- 降低主Agent负担
- 支持真正的并行执行

### 2. 声明式配置优先
- 所有Agent系统配置规则都需要预定义声明
- 避免动态prompt带来的结果不一致性
- 主流框架（Cloud Code、Deep Agents、Open Cloud）都采用此设计

### 3. 权限最小化原则
- 采用工具白名单配置机制
- 只开放允许Agent使用的工具
- 而非禁用不允许的工具

### 4. 其他通用设计原则
- 工具轻量化优先，通过组合涌现能力
- 限制循环次数避免Token爆炸
- 状态需要重新注入
- 可观测性、审核、成本控制都极为重要

## 🔮 Harness系统的未来演进路径

### 当前分类（2026 Q2）

| 框架 | 类型 | 特点 |
|------|------|------|
| **Cloud Code** | 纵深交互型 | 深度人机交互 |
| **Magic Agent** | 云端服务型 | 云端托管 |
| **Open Cloud** | 用户自管个人化 | 面向终端用户的个人开发者工具 |
| **Hermes** | 自演化Harness | 支持程序化记忆、动态快照、自进化系统 |

### 未来方向

**从Harness工程化 → Harness自演化**

- 未来让系统学会自我调教
- 人类只做幕后监督控制
- 核心仍然离不开设计模式
- 自演化必须基于反思模式才能成立

### 类比云计算架构

> 即使是云托管的Agent平台（Harness as a Service），工程师仍然需要理解底层架构模式才能用好，就像云计算需要懂弹性伸缩、备份容灾才能成为架构师。
