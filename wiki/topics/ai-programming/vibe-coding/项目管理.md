## 零、跑通的最佳实践（核心要点）

> 如果你只想记住一件事：规划驱动 + 小步迭代 + 人类终审。

### 0.1 成功的关键公式

```
Vibe Coding 成功 = 清晰的上下文 × 具体的约束 × 及时验证 × 人类终审
```

### 0.2 五步跑通流程

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 定义上下文（5分钟）                                │
│  ├── 写 CLAUDE.md / README.md                               │
│  ├── 明确项目目标、技术栈、约束                              │
│  └── 不要省略！这是AI理解项目的唯一途径                       │
├─────────────────────────────────────────────────────────────┤
│  Step 2: 拆解任务（10分钟）                                │
│  ├── AI辅助拆解："把这个功能拆成2-5分钟的任务"             │
│  ├── 按模块/层次分解                                        │
│  └── 明确依赖关系                                          │
├─────────────────────────────────────────────────────────────┤
│  Step 3: 小步执行（15-30分钟/任务）                        │
│  ├── 一次只做一个任务                                       │
│  ├── AI生成 → 即时验证 → 快速调整                           │
│  └── 代码多了就新建会话（避免上下文膨胀）                    │
├─────────────────────────────────────────────────────────────┤
│  Step 4: 测试驱动（自动化）                                 │
│  ├── 让AI写测试用例                                        │
│  ├── 人类审查断言                                          │
│  └── 自动化回归验证                                        │
├─────────────────────────────────────────────────────────────┤
│  Step 5: 人类终审（不可省略）                              │
│  ├── 代码风格检查                                          │
│  ├── 架构合规检查                                          │
│  └── 安全/业务逻辑审查                                     │
└─────────────────────────────────────────────────────────────┘
```

### 0.3 真实案例：跑通的工作流

#### 案例1：横版营销画布（文博最满意的Demo）

```
流程：
1. 明确需求："做一个横版营销画布，支持拖拽、缩放、导出"
2. AI生成初始版本：约2小时
3. 发现问题：导出功能不符合预期
4. 调整Prompt，重新生成
5. 人工优化细节（交互体验）
6. 最终交付：客户满意

关键：Demo先行，需求变更记录在案，再更新
```

#### 案例2：Cursor-Windsurf项目管理的PM工具

```
分析竞品：Asana, Monday.com, ClickUp
找到空白：Agency-focused PM tool（客户门户、计时、发票）
功能优先级矩阵：
- DO FIRST: 客户门户、计时功能
- DO LATER: 高级报表
- NEVER: 移动端（ROI低）
```

#### 案例3：Superpowers工作流

```
Brainstorming → Writing Plans → 子代理执行 → TDD验证 → 人类终审

关键：
- 子代理隔离（干净上下文）
- 测试先行（TDD）
- 不合规不合并
```

### 0.4 避坑清单

| 坑 | 表现 | 解决方案 |
|----|------|----------|
| **上下文丢失** | AI忘记之前的约束 | 写CLAUDE.md，频繁更新 |
| **上下文膨胀** | AI开始胡言乱语 | 切会话，保留关键上下文 |
| **AI自主规划** | 代码库变成乱麻 | 规划驱动，人工审批 |

| **跳过审查** | 安全漏洞/架构违规 | 人类终审不可省略 |

## 一、核心差异：人类PM vs AI PM

### 1.1 传统人类项目管理

| 维度 | 人类PM | Vibe Coding中的角色 |
|------|--------|---------------------|
| **任务分配** | 分配给人，追踪进度 | 定义目标和约束，AI执行 |
| **沟通成本** | 高（会议、文档、对齐） | 低（自然语言对话） |
| **执行监控** | 依赖人肉checklist | 依赖测试和代码审查 |
| **风险管理** | 人工识别和跟踪 | 人类定义风险边界 |
| **变更管理** | 正式流程（需求变更单） | 快速迭代，即时调整 |

### 1.2 关键理念转变

```
传统PM： "我安排任务，你执行"  →  "我定义目标，你实现"
AI PM：   "我定义约束，AI执行" →  "我审查结果，AI优化"
```

## 二、工作拆解：Vibe Coding式分解

### 2.1 三层分解结构

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：Epic（史诗）                                       │
│  ├── 一句话目标 + 非目标                                    │
│  └── 例如："做一个用户管理系统"                             │
├─────────────────────────────────────────────────────────────┤
│  第二层：Feature（功能）                                   │
│  ├── 按职责拆模块                                          │
│  ├── 接口先行，实现后补                                     │
│  └── 例如：用户注册、用户登录、权限管理                      │
├─────────────────────────────────────────────────────────────┤
│  第三层：Task（任务）                                      │
│  ├── 2-5分钟可完成                                         │
│  ├── 一次只改一个模块                                      │
│  └── 例如："实现UserRepository的findById接口"              │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 AI驱动的工作分解流程

```markdown
## 工作分解提示词

"请将以下功能拆解为可执行的开发任务：

功能：用户注册模块
要求：
- 每个任务2-5分钟可完成
- 按模块/层次分解
- 明确输入输出
- 标注依赖关系

输出格式：
1. [任务名称] - [具体描述] (依赖: xxx)
2. [任务名称] - [具体描述] (依赖: xxx)
...
```

### 2.3 优先级矩阵

```markdown
## Feature Risk-Value Matrix

Product: [你的产品]
Proposed features: [功能列表]

评估维度：
- **Value Score** (1-10):
  - 客户需求
  - 收入影响
  - 竞争差异化

- **Risk Score** (1-10):
  - 技术复杂度
  - 开发时间
  - 维护成本
  - 安全风险

输出矩阵：
┌─────────────────────────────────────┐
│ High Value, Low Risk  (DO FIRST) ✓ │
│ High Value, High Risk  (DO LATER)  │
│ Low Value, Low Risk   (BACKLOG)    │
│ Low Value, High Risk  (NEVER DO) ✗ │
└─────────────────────────────────────┘
```

## 三、迭代周期：短频快模式

### 3.1 Vibe Coding迭代 vs 传统Sprint

| 维度 | 传统Sprint（2周） | Vibe Coding迭代 |
|------|-------------------|-----------------|
| **周期** | 1-2周 | 30分钟-2小时 |
| **任务粒度** | 1-3天 | 2-5分钟 |
| **验证方式** | Sprint Review | 即时运行测试 |
| **反馈速度** | 慢 | 快 |
| **调整成本** | 高 | 低 |

### 3.2 快速迭代工作流

```
┌─────────────────────────────────────────────────────────────┐
│  Vibe Coding 快速迭代                                       │
├─────────────────────────────────────────────────────────────┤
│  1. 定义目标（5分钟）                                        │
│     - 明确要做什么、不做什么                                  │
│     - 定义验收标准                                           │
│                                                              │
│  2. AI生成（10-30分钟）                                     │
│     - AI按任务清单生成代码                                   │
│     - 人工监控方向，不干预细节                                │
│                                                              │
│  3. 即时验证（5分钟）                                       │
│     - 运行测试                                               │
│     - 手动验证核心功能                                       │
│                                                              │
│  4. 快速调整（5分钟）                                       │
│     - 发现问题立即纠正                                        │
│     - 调整Prompt或直接修改                                   │
│                                                              │
│  5. 循环迭代                                                │
│     - 返回步骤2处理下一个任务                                │
└─────────────────────────────────────────────────────────────┘
```

## 四、规划驱动：核心原则

### 4.1 规划先行原则

> **核心理念**: "规划就是一切。谨慎让AI自主规划，否则你的代码库会变成一团无法管理的乱麻。"

```markdown
## 规划检查清单

开始AI生成前，必须确认：
- [ ] 一句话目标 + 非目标 已定义
- [ ] 功能模块已拆分
- [ ] 接口已定义（接口先行）
- [ ] 技术栈已确认
- [ ] 约束条件已说明（.cursorrules）
- [ ] 验收标准已明确
```

### 4.2 上下文边界定义

```bash
# 定义AI能看到什么
echo "src/modules/payments/**" > .ai-context
echo "!src/modules/payments/secrets/" >> .ai-context

# 定义AI不能修改什么
echo "src/core/auth/**" > .ai-readonly
echo "database/migrations/**" >> .ai-readonly
```

### 4.3 .cursorrules约束模板

```markdown
## .cursorrules 约束模板

### 项目范围
- 只做xxx，不做xxx
- 技术栈：xxx

### 安全边界
- 🚫 Never generate authentication logic
- 🚫 Never write SQL without parameterization
- 🚫 Never expose internal IDs in API responses
- 🚫 Never modify .ai-readonly files

### 架构约束
- 所有服务必须实现 IService 接口
- 所有数据库访问通过 Repository 模式
- 所有外部调用通过熔断器

### 代码规范
- 函数不超过50行
- 单一职责
- 依赖注入
```

## 五、Superpowers式工作拆解

### 5.1 五阶段工作流

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: Brainstorming（头脑风暴）                          │
│  ├── 输入：需求描述、用户故事                                 │
│  └── 输出：功能列表、边界条件、验收标准草稿                   │
├─────────────────────────────────────────────────────────────┤
│  Phase 2: Writing-Plans（规划撰写）                         │
│  ├── 输入：Brainstorming输出                                │
│  └── 输出：详细实施计划、测试策略、数据流设计                 │
├─────────────────────────────────────────────────────────────┤
│  Phase 3: 子代理驱动（Sub-agent Orchestration）             │
│  ├── 输入：Writing-Plans                                   │
│  ├── 子代理A：生成测试用例（红阶段）                         │
│  ├── 子代理B：实现代码（绿阶段）                             │
│  └── 子代理C：重构优化（重构阶段）                           │
├─────────────────────────────────────────────────────────────┤
│  Phase 4: TDD执行（自动化验证）                             │
│  ├── 自动运行测试套件                                       │
│  ├── 红/绿/重构状态追踪                                     │
│  └── 失败时自动触发修复迭代                                 │
├─────────────────────────────────────────────────────────────┤
│  Phase 5: Code-Review（人类审查）                           │
│  ├── 两阶段审查：AI自审 + 人类终审                           │
│  ├── 按严重程度报告问题                                     │
│  └── 通过后合并，失败则返回Phase 3                           │
└─────────────────────────────────────────────────────────────┘
```

## 六、与人类PM的异同

### 6.1 相同点

| 维度 | 说明 |
|------|------|
| **目标导向** | 都需要明确的目标和验收标准 |
| **分层次管理** | 都需要Epic→Feature→Task分解 |
| **优先级排序** | 都需要价值-风险矩阵 |
| **持续迭代** | 都需要快速反馈和调整 |
| **质量把关** | 都需要人工终审 |

### 6.2 核心差异

| 维度 | 人类PM | AI辅助PM |
|------|--------|----------|
| **执行者** | 人执行任务 | AI执行，人监控 |
| **变更成本** | 高（重新分配、会议） | 低（改Prompt） |
| **上下文** | 人能理解复杂上下文 | 需要显式定义 |
| **创造力** | 高（业务洞察） | 受限（依赖训练数据） |
| **沟通** | 面对面、会议 | 自然语言对话 |
| **风险识别** | 经验判断 | 依赖人类定义边界 |

### 6.3 各自适用场景

| 场景 | 推荐方式 |
|------|----------|
| **需求明确、技术成熟** | Vibe Coding为主 |
| **需求模糊、需要探索** | 人类PM + 快速原型 |
| **复杂业务逻辑** | 人类PM设计 + AI实现 |
| **安全关键系统** | 人类PM + 严格审查 |
| **创新项目** | 人类PM + AI辅助 |

## 八、常见问题

### Q1: AI生成代码质量不稳定怎么办？
**解决**：明确定义约束（.cursorrules）+ 强制代码审查 + 测试驱动

### Q2: 上下文太长导致AI混乱怎么办？
**解决**：切会话（代码多了就切）+ 保持简洁的函数 + 显式传递关键上下文

### Q3: 如何管理大型项目？
**解决**：模块化 + 接口先行 + 一次只改一个模块 + 定期架构验证

### Q4: AI不听指挥怎么办？
**解决**：重新开窗口 + 明确约束条件 + 使用.cursorrules
