## 一、变量命名规范

### 基本原则
- **一致性**：同类变量命名风格统一
- **可读性**：名称能清晰表达意图
- **简洁性**：避免冗长，但也不过度缩写

### 命名风格参考

| 类型 | 风格 | 示例 |
|------|------|------|
| 变量 | snake_case | `user_name`, `total_count` |
| 常量 | UPPER_SNAKE | `MAX_RETRY_COUNT` |
| 类名 | PascalCase | `UserService`, `OrderManager` |
| 函数 | snake_case | `get_user_by_id()` |
| 布尔值 | is/has/can前缀 | `is_active`, `has_permission` |

## 二、文件结构规范

### 标准项目结构

```
project/
├── src/                    # 源代码
│   ├── modules/           # 功能模块
│   ├── utils/            # 工具函数
│   └── main.py           # 入口文件
├── tests/                 # 测试代码
├── docs/                  # 文档
├── configs/              # 配置文件
├── scripts/              # 脚本
├── memory-bank/          # 记忆库（Vibe Coding核心）
│   ├── design-doc.md     # 设计文档
│   ├── tech-stack.md     # 技术栈
│   ├── plan.md          # 实施计划
│   └── progress.md      # 进度记录
├── CLAUDE.md            # AI行为准则
└── README.md            # 项目说明
```

### 模块拆分原则

1. **单一职责**：每个模块只做一件事
2. **高内聚低耦合**：模块内部紧密，模块间松散
3. **接口清晰**：模块间通过接口通信

## 四、架构原则

### 核心原则

| 原则 | 说明 | 适用场景 |
|------|------|----------|
| KISS | Keep It Simple, Stupid | 所有场景 |
| YAGNI | You Aren't Gonna Need It | 避免过度设计 |
| DRY | Don't Repeat Yourself | 重复代码 |
| SOC | Separation of Concerns | 复杂系统 |

### 分层架构建议

```
表现层 → 业务逻辑层 → 数据访问层 → 基础设施层
```

## 五、上下文管理（Vibe Coding核心）

### 为什么重要

上下文是Vibe Coding的第一性要素。垃圾进，垃圾出。

### 实践技巧

1. **定期总结**：每个里程碑后更新上下文文档
2. **清晰结构**：用清晰的目录结构帮助AI理解
3. **版本记录**：记录重要决策和变更原因
4. **切会话**：代码一多就开新会话，避免上下文膨胀

### Memory Bank维护

```markdown
# 每完成一个里程碑后更新

## 当前进度
- [x] 完成用户认证模块
- [ ] 完成订单管理模块
- [ ] 完成支付模块

## 已知问题
- 暂无

## 下一步计划
- 实现订单查询功能
```

## 六、提示词编写技巧

### 1. 明确边界

```
✅ 好：
"实现用户登录功能，只能改 auth/login.py，不能改其他文件"

❌ 差：
"实现用户登录功能"
```

### 2. 指定输出格式

```
✅ 好：
"用以下格式输出：
1. 方案A：...
2. 方案B：...
每个方案要包含：优点、缺点、推荐场景"

❌ 差：
"给出登录功能的实现方案"
```

### 3. 要求验证

```
✅ 好：
"实现后，运行 pytest tests/test_login.py -v 验证"

❌ 差：
"实现登录功能"
```

## 七、项目案例参考

### 案例：空战游戏开发

1. **Game Design Document**：描述游戏结构和意图
2. **技术栈选择**：推荐最简单但最健壮的方案
3. **实施计划**：分步小任务，每步有验证
4. **增量开发**：先基础游戏，再加细节功能

### 关键经验

- 不要一次让AI做太多
- 每个功能都要有测试验证
- 经常提交Git，保留可回溯版本
- 卡住时，换提示词或回退版本

## 八、常见问题处理

| 问题 | 解决方案 |
|------|----------|
| AI跑偏 | 重新明确任务，使用 `/rewind` 回退 |
| 代码混乱 | 切新会话，给清晰上下文 |
| 测试失败 | 提供「预期 vs 实际 + 最小复现」 |
| 上下文膨胀 | `/clear` 或 `/compact` 清理 |
