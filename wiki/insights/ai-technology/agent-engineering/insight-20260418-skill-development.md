## 🎯 Skill描述三板框架

### 核心价值

Skill描述的核心作用是让大模型知道**何时调用**以及**如何做好**。

### 三层结构

```
┌─────────────────────────────────────────────────────────────┐
│                 Skill描述三板框架                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1️⃣ 核心价值层                                             │
│     └── 定义技能能解决的核心问题                              │
│     └── 明确技能的定位                                       │
│     └── 示例："分析Figma设计稿并生成可交付的开发代码"        │
│                                                              │
│  2️⃣ 能力拆解层                                             │
│     └── 罗列式清晰描述技能包含的具体子能力                    │
│     └── 示例：Figma技能包含                                  │
│         • 提取设计规范                                       │
│         • 生成组件文档                                       │
│         • 导出标注                                           │
│                                                              │
│  3️⃣ 激活条件层 ← 必须保留的核心部分                          │
│     └── 明确说明技能应该被触发的具体场景                      │
│     └── 补充明确的关键字                                     │
│     └── 补充禁止触发的反向约束                                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

> 🚩**激活条件是现阶段大模型能力不足的情况下必须保留的核心部分**，该过渡阶段会持续一年以上。

### 反例实践分析

| 错误类型 | 问题 | 后果 |
|----------|------|------|
| **只写能力不写触发** | 仅写"创建Git分支" | 大模型认为自己可以直接生成指令，不需要调用技能，最终完全不触发 |
| **触发条件模糊** | 仅写"CSV数据分析" | 即使只有三行简单数据，大模型也会错误触发该技能，生成不必要复杂流程，浪费Token和成本 |
| **缺少反向约束** | 未说明禁止触发的场景 | 技能会错误触发到语义相近但不匹配的场景 |

## 📝 Skill内容（Body）编写原则

### 两类Skill内容

| 类型 | 定义 | 适用场景 |
|------|------|----------|
| **知识补充型** | 补充大模型不具备的信息，如企业内部规范、API写法、UI编码风格等 | 不限制固定流程 |
| **工作流型** | 定义明确的步骤流程 | API调用、复杂数据流分析、规模化代码迁移等 |

### 核心编写原则

> 🚩**只需要告诉模型「做到什么程度是好的」，不要过度限制步骤**

| 过度限定 | 更优做法 |
|----------|----------|
| 限定死每个步骤 | 明确目标，让模型自由选择路径 |
| 提高任务下限 | 不锁死效果上限 |
| 带来安全感 | 在需要创造力的复杂任务中表现更好 |

**权衡**：过度限定步骤虽然会提高任务下限，但会锁死效果上限。

## 🔄 Skill迭代方法论

### 分层评测体系三原则

```
┌─────────────────────────────────────────────────────────────┐
│                 分层评测三原则                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1️⃣ 分层评估                                               │
│     └── 描述（决定激活）和内容（决定激活后效果）分开评估     │
│     └── 避免问题归因时互相甩锅                               │
│                                                              │
│  2️⃣ 对照实验                                               │
│     └── 只看修改前后的相对效果对比                           │
│     └── 不追求绝对量化分数                                   │
│     └── 确保修改带来正向增量，不导致效果倒退                 │
│                                                              │
│  3️⃣ 人类参与                                               │
│     └── 不存在全自动无人工的评测方案                         │
│     └── 必须由人类专家把控效果                               │
│     └── 自动化方法仅可用于补充修复小漏洞                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 描述评测方法

**测试Query准备**：
- 10-20条测试Query
- 一半正样本（应该触发）
- 一半负样本（不应该触发）
- 只检测激活结果，不需要执行后续内容（节省成本）

> 🚩**评测的核心精力要放在负样本上**
> 
> **误触发的危害远大于不触发**：不触发可以手动补救，误触发会直接导致任务出错、浪费成本。

**核心评估指标**：

| 指标 | 定义 | 目标 |
|------|------|------|
| **召回率** | 应该触发时成功触发的比例 | 理想100% |
| **精准率** | 所有触发中正确触发的比例 | 复杂技能70%-80%是极限 |

### 描述优化技巧

| 问题 | 解决方案 |
|------|----------|
| **欠触发** | 补充常见同义词/中文关键词，即使术语是英文，也要补充常用中文叫法 |
| **误触发** | 先定位根因，无法从根因解决时补充负向约束，明确说明禁止触发的场景 |

### Body评测方法

1. 在真实可运行的沙箱环境中测试
2. 包含完整代码库、可编译代码、完整工具链
3. 通过对照实验对比修改前后效果
4. 人工结合程序化标准评分
5. 决策是否保留修改

### 迭代修改四大原则

```
┌─────────────────────────────────────────────────────────────┐
│                 迭代修改四大原则                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1️⃣ 精简                                                  │
│     └── 不要教模型已经知道的常识                              │
│     └── 删除冗余内容                                        │
│     └── 精简描述后效果通常会提升                             │
│                                                              │
│  2️⃣ 谨慎使用限定词                                         │
│     └── must/critical/mandatory是强限定词                    │
│     └── 必须逐一检查                                        │
│     └── 每个强限定都要补充原因，让模型可以泛化行为            │
│                                                              │
│  3️⃣ 重复劳动脚本化                                         │
│     └── 模型重复做的工作全部改为自动化脚本解决                │
│                                                              │
│  4️⃣ 提炼通用规律                                           │
│     └── 不要针对单个失败Case打补丁                          │
│     └── 从大量失败Case中提炼通用规则                        │
│     └── 彻底重构提示词才能获得持续效果提升                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 🏃 长程复杂任务落地实践

### 长程任务定义

**特征**：
- 步骤多（十步以上的长链路）
- 耗时长（几十分钟到数小时）
- 容易因外部中断出错

**典型场景**：
- 百个代码仓库统一升级依赖
- 批量爬取五十个网站数据生成结构化报告
- 自动化排查修复复杂现场Bug

### 长程任务三大核心问题

| 问题 | 描述 | 后果 |
|------|------|------|
| **错误积累** | 小错误逐步累积 | 最终发现结果不可用时，无法定位错误来源，只能全部重来 |
| **状态丢失** | 长任务必然超出上下文窗口 | 容易丢失当前任务进度 |
| **无限死循环** | 要求必须解决所有错误才能继续 | 遇到无法解决的错误会陷入永久死循环 |

### 长程任务四大设计原则

```
┌─────────────────────────────────────────────────────────────┐
│                 长程任务四大设计原则                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1️⃣ 任务边界清晰                                           │
│     └── 明确说明具体要做什么                                 │
│     └── 什么程度算完成                                      │
│     └── 不要模糊需求                                        │
│     └── ✅ 好："干掉项目所有TypeScript类型中的any"           │
│     └── ❌ 差："帮我优化项目"                               │
│                                                              │
│  2️⃣ 错误隔离在最小范围                                     │
│     └── 子任务错误不能扩散到全局                             │
│     └── 单个文件处理失败要就地重试/记录                      │
│     └── 不能导致整个任务失败                                 │
│                                                              │
│  3️⃣ 步骤间强制校验                                         │
│     └── 不能轻信模型"我完成了"的自述                        │
│     └── 必须通过程序化方式校验结果                          │
│         • 要求输出结构化数据                                │
│         • 约定结束密语                                      │
│         • 自动化测试                                        │
│         • 交叉验证上一步结果                                │
│     └── 确认完成后才能进入下一步                            │
│                                                              │
│  4️⃣ 允许局部失败                                           │
│     └── 允许部分任务失败后跳过，继续推进剩下的任务           │
│     └── 所有失败必须记录到状态文件中                        │
│     └── 最后统一处理，提高整体任务的可用性                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 落地实用技巧

| 技巧 | 说明 | 效果 |
|------|------|------|
| **外部文件维护进度** | 不要在大模型上下文中维护进度，将待办和进度写入外部文件 | 避免上下文丢失 |
| **实时更新状态** | 每个任务标记阶段状态（未开始/进行中/成功/失败），支持回滚恢复 | 中断后可从最近正确状态继续 |
| **CI命令行调度子任务** | 不要让Agent自身做并发调度，通过CI脚本从命令行调度子任务 | 支持上百路高并发，子任务上下文不互相污染 |
| **分层重试策略** | 正常执行出错→允许2-3次修复重试→仍失败则回滚 | 平衡成功率和成本 |
