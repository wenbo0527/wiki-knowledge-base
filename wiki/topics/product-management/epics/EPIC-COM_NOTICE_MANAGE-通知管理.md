## 1. 产品总览

### 1.1 一句话定位
**企业数字社区的智能消息中心，实现多渠道统一推送、精准触达和用户偏好管理，确保"对的内容在对的时间送达对的人"。**

### 1.2 产品定位 & 核心价值

| 维度 | 描述 |
|:---|:---|
| **目标用户** | 系统管理员、运营人员、全体员工（消息接收者） |
| **核心痛点** | 消息分散难以管理、推送频繁造成打扰、重要消息被淹没 |
| **解决方案** | 多渠道统一 + 智能推送 + 偏好管理 + 分级推送 |
| **核心价值** | 消息到达率提升至 99%，用户满意度提升 50%，信息传递效率提升 60% |

## 2. 功能结构（按模块归档）

### 2.1 FEATURE-COM_NOTICE_SEND（消息发送）

#### 功能入口 & 作用

**入口位置**：消息中心 → 发消息

**核心作用**：
- 支持多种消息类型的创建和发送
- 配置消息的目标受众和发送策略
- 管理和使用消息模板
- 支持消息的定时发送和批量发送

#### 核心逻辑

```
消息发送流程：

Step 1: 选择消息类型
  - 系统通知（全员或指定范围）
  - 提醒消息（待办、会议、日程）
  - 公告消息（重要信息，需确认阅读）
  - 审批结果（流程类消息）
  - 营销消息（活动推广）

Step 2: 编辑消息内容
  - 填写标题和正文
  - 选择消息模板（可选）
  - 添加附件（图片、文件）
  - 配置跳转链接

Step 3: 配置发送策略
  - 选择目标受众（全员/部门/标签/个人）
  - 设置发送时间（即时/定时）
  - 选择发送渠道（多选）
  - 设置优先级（普通/重要/紧急）

Step 4: 发送执行
  - 权限校验和内容审核
  - 消息入队（消息队列）
  - 分发到各渠道
  - 发送状态跟踪
```

#### 关键配置项

| 配置项 | 说明 | 示例 |
|:---|:---|:---|
| `message.type` | 消息类型 | `NOTICE` / `REMINDER` / `ANNOUNCEMENT` |
| `message.priority` | 消息优先级 | `NORMAL` / `IMPORTANT` / `URGENT` |
| `message.audience` | 目标受众 | `{type: "ALL"}` / `{type: "DEPT", ids: [...]}` |
| `message.channels` | 发送渠道 | `["in_app", "email", "sms"]` |
| `message.send_time` | 发送时间 | `IMMEDIATE` / `SCHEDULED` |
| `message.template_id` | 消息模板 ID | `tmpl_approval_result` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **消息发送延迟** | 批量发送时队列堆积 | 扩容消费者、优化批量处理逻辑 |
| **渠道发送失败** | 部分渠道发送失败 | 重试机制、失败告警、手动补偿 |
| **消息内容过长** | 超过渠道限制 | 自动截断、分页发送、摘要+链接 |
| **重复发送** | 网络问题导致重复投递 | 幂等设计、消息去重 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 消息模板 | 依赖 | 消息格式来源 |
| 用户服务 | 上游 | 获取用户信息和偏好 |
| 渠道服务 | 依赖 | 实际消息发送 |
| 内容审核 | 依赖 | 消息内容合规检查 |

### 2.2 FEATURE-COM_NOTICE_DELIVERY（消息投递）

#### 功能入口 & 作用

**入口位置**：消息中心 → 投递管理 / 系统管理 → 通道配置

**核心作用**：
- 多渠道消息投递（应用内、短信、邮件、企微/飞书）
- 智能路由（根据用户偏好和设备选择最优渠道）
- 发送策略配置（频率限制、防骚扰）
- 投递状态跟踪和监控

#### 核心逻辑

```
消息投递流程：

Step 1: 路由决策
  - 获取用户消息偏好
  - 获取用户设备信息
  - 判断消息优先级
  - 选择最优渠道组合

Step 2: 消息封装
  - 格式转换（适配各渠道）
  - 内容裁剪（适应渠道限制）
  - 添加追踪参数（已读回执）
  - 签名和加密（如需要）

Step 3: 渠道投递
  - 并行发送到各渠道
  - 重试机制（失败自动重试）
  - 限流控制（遵守渠道限制）
  - 状态回传

Step 4: 状态更新
  - 投递成功 → 更新消息状态
  - 投递失败 → 记录失败原因
  - 已读确认 → 更新已读状态
```

#### 关键配置项

| 配置项 | 说明 | 示例 |
|:---|:---|:---|
| `delivery.retry_times` | 最大重试次数 | `3` |
| `delivery.retry_interval` | 重试间隔 | `60s` |
| `delivery.batch_size` | 批量发送大小 | `100` |
| `delivery.rate_limit` | 发送限流 | `1000/分钟` |
| `delivery.smart_routing` | 智能路由开关 | `true` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **短信发送失败** | 运营商限流或号码问题 | 退订处理、号码验证、备用通道 |
| **邮件进垃圾箱** | 邮件被识别为垃圾邮件 | 域名 SPF/DKIM 配置、邮件内容优化 |
| **推送被忽略** | 用户关闭通知权限 | 引导开启、多种渠道兜底 |
| **消息丢失** | 网络波动导致消息未送达 | 消息持久化、重试机制、补偿查询 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 消息发送服务 | 上游 | 待投递的消息来源 |
| 通道服务 | 依赖 | 短信、邮件等通道 |
| 用户服务 | 上游 | 用户偏好和设备信息 |
| 消息存储 | 依赖 | 消息持久化 |

### 2.3 FEATURE-COM_NOTICE_PREFERENCE（用户偏好管理）

#### 功能入口 & 作用

**入口位置**：个人设置 → 消息设置

**核心作用**：
- 管理用户的通知渠道偏好
- 配置免打扰时间段
- 设置消息分类接收规则
- 黑名单管理（屏蔽不想收到的消息）

#### 核心逻辑

```
用户偏好管理流程：

Step 1: 偏好采集
  - 引导用户设置偏好
  - 分析用户行为（活跃渠道）
  - 智能推荐最优配置

Step 2: 规则配置
  - 渠道开关（开启/关闭各渠道）
  - 免打扰时段（不推送的时间）
  - 消息分类过滤（只接收指定类型）
  - 关键词过滤（屏蔽含关键词的消息）

Step 3: 规则执行
  - 发送前检查用户偏好
  - 符合规则 → 正常发送
  - 违反规则 → 跳过或延迟

Step 4: 规则优化
  - 基于用户反馈优化推荐
  - 定期提醒用户更新偏好
  - 沉默用户引导激活
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---|
| `preference.in_app_enabled` | 应用内通知开关 | `true` |
| `preference.email_enabled` | 邮件通知开关 | `true` |
| `preference.sms_enabled` | 短信通知开关 | `false` |
| `preference.dnd_start` | 免打扰开始时间 | `22:00` |
| `preference.dnd_end` | 免打扰结束时间 | `08:00` |
| `preference.dnd_enabled` | 免打扰开关 | `false` |
| `preference.message_types` | 接收的消息类型 | `ALL` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **偏好设置复杂** | 用户不知道如何设置 | 提供智能推荐和一键优化 |
| **偏好不同步** | 多个端偏好不一致 | 云端同步、优先使用最新配置 |
| **紧急消息被屏蔽** | 用户设置导致紧急消息被过滤 | 紧急消息强制推送、例外规则 |
| **偏好设置丢失** | 清除缓存导致设置丢失 | 服务端持久化、登录时恢复 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 消息投递服务 | 下游 | 投递时读取偏好 |
| 用户服务 | 上游 | 用户登录和标识 |
| 通道服务 | 下游 | 渠道偏好控制 |

## 5. 规则与配置中心

### 5.1 消息分类规则

| 消息类型 | 说明 | 优先级 | 默认渠道 |
|:---|:---|:---|:---|
| **系统通知** | 系统级重要通知 | HIGH | 应用内 + 短信 |
| **审批通知** | 审批流程结果 | HIGH | 应用内 + 短信 |
| **待办提醒** | 任务和待办提醒 | NORMAL | 应用内 |
| **公告** | 重要公告需确认 | HIGH | 应用内 + 邮件 |
| **营销消息** | 活动推广 | LOW | 应用内 |

### 5.2 推送频率规则

| 规则类型 | 规则内容 |
|:---|:---|
| **单用户日上限** | 单用户每天最多收到 50 条推送 |
| **同类型消息合并** | 同一类型的消息在 1 小时内合并为一条 |
| **免打扰时段** | 用户设置的免打扰时段不推送（非紧急）|
| **静默期** | 周末降低推送频率 50% |

### 5.3 紧急消息规则

```
紧急消息定义：
- 消息级别为 URGENT
- 或消息类型为 SYSTEM_CRITICAL

紧急消息处理：
1. 忽略用户免打扰设置
2. 使用所有可用渠道并行推送
3. 未送达持续重试直到确认
4. 5 分钟未送达触发告警
```

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: 消息发送后用户没收到怎么办？

**排查步骤**：
1. 检查消息发送状态
2. 确认目标受众配置是否正确
3. 检查用户偏好设置（是否关闭了通知）
4. 查看投递失败原因

**解决方案**：
- 等待重试（网络问题会自动重试）
- 手动触发重新发送
- 联系用户检查通知权限

#### Q: 如何撤回已发送的消息？

**排查步骤**：
1. 确认消息是否在撤回时限内（24h）
2. 检查撤回权限是否有

**解决方案**：
- 在消息管理页面点击撤回
- 撤回后用户端消息会同步消失

#### Q: 用户反馈消息太多太烦怎么办？

**排查步骤**：
1. 查看该用户的推送频率
2. 检查是否有消息合并
3. 确认用户偏好设置

**解决方案**：
- 引导用户调整偏好设置
- 开启免打扰时段
- 关闭不重要的消息类型

## 8. 方法论沉淀

### 8.1 消息推送方法论

**消息价值公式**：
```
消息价值 = 时效性 × 重要性 × 相关性 / 打扰度
```

**推送策略最佳实践**：
1. **精准触达**：只发给需要的人，避免过度打扰
2. **时机合适**：在用户活跃时段推送
3. **内容精简**：消息内容要简洁明了
4. **渠道适配**：根据内容选择合适渠道

### 8.2 消息降噪方法论

**消息分级策略**：
| 级别 | 定义 | 推送策略 |
|:---|:---|:---|
| **P0 紧急** | 必须立即知晓 | 全量推送、强制提醒 |
| **P1 重要** | 需要尽快知晓 | 应用内推送、声音提醒 |
| **P2 普通** | 常规信息 | 应用内推送、静默 |
| **P3 低价值** | 可延迟消费 | 不主动推送、用户查看 |

**消息合并策略**：
- 同类型消息 1 小时内合并
- 合并后发送摘要 + 链接
- 用户点击后查看完整列表

### 8.3 行业对比

| 对比维度 | 传统短信通知 | 现代推送（极光）| 数字社区消息（我们）|
|:---|:---|:---|:---|
| **渠道覆盖** | 仅短信 | 应用推送 | 多渠道融合 |
| **触达率** | 较高 | 中等（受权限影响）| 高（多渠道兜底）|
| **用户控制** | 无 | 有限 | 完整偏好管理 |
| **智能化** | 无 | 有限 | 智能路由、合并、降噪 |
| **数据分析** | 简单 | 中等 | 完整漏斗分析 |

### 8.4 工具流程

**消息运营工具链**：
- **消息编辑**：模板管理工具、在线预览
- **投递监控**：Grafana 实时监控
- **数据分析**：埋点 + ClickHouse + BI
- **用户反馈**：反馈收集 + 情感分析

**自动化工具**：
- **定时任务**：Cron 调度待办提醒、会议提醒
- **消息合并**：定时任务合并低价值消息
- **A/B 测试**：推送策略效果对比
