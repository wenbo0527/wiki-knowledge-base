# 1. 产品总览

## 1.1 一句话定位

**构建企业级产品需求管理体系，实现从业务需求到技术实现的智能拆解、结构化管理和全流程追踪。**

## 1.2 产品定位 & 核心价值

| 维度 | 描述 |
|:---|:---|
| **目标用户** | 产品经理、技术负责人、项目经理、开发人员、测试人员 |
| **核心痛点** | 需求拆解不系统、协作不透明、知识难沉淀、进度难追踪 |
| **解决方案** | 四层架构（业务入口→AI拆解→本体语义→图向量层）+ 双存储（Neo4j+飞书） |
| **核心价值** | 需求可追踪、协作可视化、知识可沉淀、决策智能化 |

## 1.5 关键约束

| 约束类型 | 具体约束 | 应对策略 | 责任方 |
|:---|:---|:---|:---:|
| **数据安全** | Epic/Story 涉及业务敏感信息 | 飞书表格权限控制、Neo4j 本地部署、敏感数据脱敏 | 技术负责人 |
| **性能要求** | 飞书 API 调用频率限制（100次/秒） | 批量同步、本地缓存、增量更新、异步处理 | 开发团队 |
| **数据一致性** | Neo4j ↔ 飞书表格双向同步 | 以 Neo4j 为准、定时同步、冲突检测、人工介入 | 产品经理 |
| **可扩展性** | 支持多产品域、多 Epic 并行（目标 100+ Epic） | 模块化设计、标准化流程、模板复用、自动化工具 | 架构师 |
| **合规要求** | 数据血缘、审计追踪、变更日志 | 变更日志、操作审计、版本管理、定期审计 | 项目经理 |
| **用户体验** | 飞书表格响应 < 2s、Neo4j 查询 < 500ms | 索引优化、缓存策略、分页加载、异步渲染 | 开发团队 |

## 1.6 术语词典（本体论级别）

| 术语 | 英文 | 定义 | 示例 | 层级 |
|:---|:---|:---|:---|:---:|
| **产品域** | Product Domain | 产品的高层级分类，代表一个完整的业务领域，包含多个 Epic | PD-DFD（数据发现）、PD-MKT（数字营销） | L1 |
| **Epic** | Epic | 产品域下的业务模块，代表一个可独立交付的价值单元，包含多个 Feature | EPIC-DFD_DATA_ASSET（数据资产字典） | L2 |
| **Feature** | Feature | Epic 下的功能模块，代表一个内聚的功能单元，包含多个 Story | FEATURE-DFD_ASSET_OPERATE_TOOL（资产运营工具） | L3 |
| **Story** | User Story | 用户可感知的完整功能点，代表一个可独立交付的用户价值，可在一个迭代内完成 | STORY-ASSET_LIST_PAGE（资产列表页） | L4 |
| **故事点** | Story Point | 相对估算单位，表示完成 Story 所需的工作量，用于评估团队产能 | 1, 2, 3, 5, 8, 13 | - |
| **验收标准** | Acceptance Criteria | Story 完成必须满足的条件，用于测试验证，通常包含功能、操作、数据、交互 4 项 | 功能正常、操作正常、数据准确、交互符合预期 | - |
| **飞书表格** | Feishu Bitable | 飞书多维表格，用于需求状态同步和团队协作，支持多视图、权限控制、自动化 | Epic拆解确认-数据资产字典 | - |
| **四层架构** | Four-Layer Architecture | 产品管理体系的架构分层：业务需求入口 → AI 需求拆解 → 本体论语义 → 图与向量层 | - | - |
| **双存储** | Dual Storage | Neo4j（核心数据）+ 飞书表格（协作同步）的存储架构 | - | - |

- 数据资源申请流程复杂 → 简化流程、提供自助服务、优化体验

# 5. 规则与配置中心

## 5.1 Story 验收标准模板（强制）

**每个 Story 必须包含以下 4 项验收标准**：

| 标准 | 说明 | 示例 |
|:---|:---|:---|
| **功能可正常使用** | 核心功能完整 | 页面可正常加载，数据可正常展示 |
| **相关操作正常** | CRUD 操作无误 | 新增/编辑/删除/查询操作正常 |
| **数据准确完整** | 数据流转正确 | 字段显示准确，数据计算正确 |
| **交互符合预期** | UI/UX 符合设计 | 布局、样式、交互符合 PRD |

## 5.2 Epic 拆解流程标准（SOP）

**必须按以下 5 步执行**：

```
Step 1: 范围确认
  - 阅读 PRD + 查看代码 + 确认边界

Step 2: Feature 重建
  - 清理旧数据 + 重建结构 + 创建节点

Step 3: Story 拆解
  - 拆解 Story + 生成验收标准 + 创建节点

Step 4: 飞书同步
  - 创建表格 + 写入数据 + 生成链接

Step 5: 验证复盘
  - 对比 PRD + 对比代码 + 更新文档
```

## 5.3 Feature 粒度标准

| 标准 | 值 | 说明 |
|:---|:---:|:---|
| 最小 Story 数 | 3 | 避免过度拆分 |
| 最大 Story 数 | 10 | 避免粒度过大 |
| 最佳 Story 数 | 5-7 | 推荐范围 |

## 5.4 优先级定义

| 优先级 | 定义 | 处理时效 |
|:---|:---|:---:|
| **P0** | 阻塞后续工作，必须立即处理 | 24小时内 |
| **P1** | 重要需求，正常迭代完成 | 当前 Sprint |
| **P2** | 一般需求，有空时处理 | 下个 Sprint |
| **P3** | 优化需求，长期规划 | 按需排期 |

# 7. 问题库 & FAQ

## 7.1 常见问题排查

### Q: 飞书表格同步失败怎么办？

**排查步骤**：
1. 检查 App Token 是否有效
2. 检查 Bot 是否有表格编辑权限
3. 检查 API 调用频率是否超限
4. 查看错误日志定位具体原因

**解决方案**：
- 重新授权 Bot 权限
- 分批写入数据，避免超限
- 检查数据格式是否符合要求

### Q: Neo4j 查询性能慢怎么办？

**排查步骤**：
1. 检查是否为常用查询字段创建索引
2. 检查查询是否返回大量数据
3. 检查是否存在全表扫描

**解决方案**：
- 为 id, label, status 字段创建索引
- 使用 LIMIT 限制返回数量
- 优化 Cypher 查询语句

### Q: Story 验收标准不明确怎么办？

**解决方案**：
- 强制使用 4 项标准模板
- 与产品经理确认具体业务场景
- 参考类似 Story 的验收标准

# 8. 方法论沉淀

## 8.1 需求拆解方法论

**四层拆解法**：
```
产品域 (Domain) → Epic → Feature → Story
```

**拆解原则**：
1. **自顶向下**：先理解整体，再拆解细节
2. **业务驱动**：以用户价值为导向
3. **可交付性**：每个层级都能独立交付价值
4. **可测试性**：每个 Story 都有明确的验收标准
5. **粒度适中**：避免过度拆分，也避免粒度过大

## 8.2 产品设计原则

- **简单性**：能用简单方案不用复杂方案
- **一致性**：保持术语、流程、标准的一致性
- **可追踪性**：每个需求都有完整的追踪链路
- **数据驱动**：用数据说话，但也要有直觉
- **快速迭代**：先做出来，再优化

## 8.3 本体论使用范式

**定义原则**：
- 一个名词只有一个含义
- 层级关系清晰（Domain → Epic → Feature → Story）
- 属性定义标准化（id, label, description, status, priority）

**使用场景**：
- Neo4j 节点类型定义
- 飞书表格字段定义
- 团队术语统一

## 8.4 工具流程

**日常流程**：
1. 收到产品 PRD
2. Neo4j 拆解 Epic/Feature/Story
3. 飞书表格同步
4. 技术排期
5. 开发实现
6. 验收确认
7. 更新状态

**工具链**：
- Neo4j：产品知识存储
- 飞书表格：协作同步
- OpenClaw：Agent 自动化
- Python：脚本工具
