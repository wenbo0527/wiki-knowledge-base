## 洞察一：Harness会过期，Build to Delete（Anthropic官方）

> **来源**: Claude官方博客 + Agent Engineering专题
> **适用**: 所有AI编程/Agent工程场景

### 核心洞察

Harness工程化的本质是**编码"模型做不到什么"的假设**。但模型在变强，假设会过期，Harness会变成dead weight。解决方案：定期审查，该拆就拆。

```
问题出现 ──→ 搭建Harness解决 ──→ 模型变强 ──→ Harness成为dead weight
     ↑                                                    ↓
     └────────────── 应该拆掉，而不是永久保留 ──────────────┘
```

### 关键数据
| 模型版本 | BrowseComp准确率 | 关键变化 |
|:---|:---:|:---|
| Sonnet 3.5 | ~40% | 基础能力 |
| Sonnet 4.5 | 43% | 需要context重置机制 |
| Opus 4.5 | 68% | 自主编排能力出现 |
| Opus 4.6 | 84% | Subagent + Memory Folder |

### 三条核心原则

| 原则 | 核心要点 |
|:---|:---|
| **用Claude已经会的** | 底层工具越通用，Claude发挥空间越大；专用工具是在替Claude做决策 |
| **什么时候该停手** | 定期问"我还能停掉什么？"；把控制权还给模型 |
| **边界要谨慎** | 缓存策略、安全边界、专用工具——该设的设，但要精准 |

### 行动建议
- **立即**：为每个Harness设置"到期日"，每月审查一次
- **长期**：建立deprecation流程，该拆就拆

## 洞察二：银行AI本质是"岗位能力数字化"（非"工具化"）

> **来源**: 智能小致公众号 + 银行AI落地专题
> **适用**: 金融/企业AI转型场景

### 核心洞察

> 当大多数银行还在纠结"上了多少模型"时，先行者已经开始思考：**有多少岗位能力，真正变成了可复用的数字资产？**

**现状困境**：
- 智能客服"答非所问"
- 智能风控实际使用率不到30%
- 各部门AI助手像孤岛，难以复用

**根本原因**：陷入了**"工具化"误区**——把AI助手等同于聊天工具，忽视了**"岗位能力数字化"**这个核心。

### 思维转变

```
传统思维：AI = 上线多少模型
                 ↓
新思维：AI = 转化了多少岗位能力为Skill资产
```

### Skill的四大模块

| 模块 | 作用 | 银行场景举例 |
|:---|:---|:---|
| **Prompt模板** | 定义任务逻辑 | "分析客户信用风险"的指令模板 |
| **工具调用(API)** | 连接业务系统 | 调用信贷系统、风控接口 |
| **数据接口** | 获取必要数据 | 客户信息、交易记录 |
| **决策逻辑** | 输出结果规则 | 风险等级判定标准 |

### 四大岗位Skill画像

| 岗位类型 | 核心需求 | 典型Skill示例 |
|:---|:---|:---|
| **管理层** | 战略决策、风险预判 | 经营分析、风险趋势预测 |
| **中层管理者** | 资源配置、绩效分析 | 部门绩效分析、任务分配 |
| **前线员工** | 高频业务处理 | 客户画像、营销话术生成 |
| **中后台员工** | 流程支撑 | 报表生成、合规检查 |

### 行动建议
- 目标：核心Skill≥200个，岗位覆盖≥80%，效率提升≥30%
- 三大难点：隐性经验萃取、安全可控、"上线即闲置"预防

## 洞察三：本体论 = 认知地图，大模型是失忆的天才

> **来源**: Palantir Ontology专题 + DataFunTalk
> **适用**: 企业AI/Knowledge管理场景

### 核心洞察

> 没有本体的大模型是"失忆的天才"——聪明但盲目；有大模型+本体才是"有地图的探险家"——既有智慧又有方向。

**数据层** → **本体层** → **大模型层**，三层架构缺一不可。

### 金融风控本体论验证

```
业务本体五层模型：
第一层：实体模型（客户、授信申请、授信额度）
第二层：流程模型（IBM五级建模：Domain→Value Chain→Activity→Task→Step）
第三层：行为模型（客户.submitApplication()、系统.evaluateCredit()）
第四层：规则模型（信用评分规则、额度计算规则、反欺诈规则）
第五层：数据模型（映射到核心系统API、数据仓库表结构）
```

### 关键洞察

| 范式 | 传统数据驱动 | 本体论驱动 |
|:---|:---|:---|
| 逻辑 | 数据→分析→洞察 | 本体→理解→行动 |
| 响应 | 被动响应（后视镜） | 主动决策（导航仪） |
| 问题 | "发生了什么？" | "应该做什么？" |

### 实践要点
- **必做**：选最痛的场景，建最小模型，跑通MVP闭环
- **必不做**：不要试图一步到位大而全，不要"为本体而本体"
- **7:3原则**：本体建设70%工作量在业务口，业务人员必须深度参与

## 洞察四：模型可解释性 = 从合规成本到竞争壁垒

> **来源**: 模型可解释性专题
> **适用**: 消费金融/风控/合规场景

### 核心洞察

消费金融场景中，模型可解释性的价值不在于技术层面的SHAP值输出，而在于**将专业解释转化为用户可理解、合规、不歧视的自然语言拒贷理由**。能做好这点的公司，把监管义务转化为了用户体验优势和品牌信任。

### 合规强制要求

| 要求 | 说明 |
|:---|:---|
| **用户告知义务** | 拒贷必须在1个工作日内告知具体理由，不能用"综合评分不足" |
| **解释留存义务** | 解释必须留存至少5年，包含用户ID、决策时间、模型版本 |
| **歧视防范义务** | 不能包含性别、民族、地域、宗教等禁止类特征 |
| **监管检查义务** | 必须能提供模型可解释性说明供检查 |

### 落地门槛

```
SHAP技术解释（TreeExplainer）
    ↓
规则引擎"理由模板映射"
    ↓
自然语言拒贷理由（Top3因素）
```

**1个工程师2周可完成基础版，满足中小消费金融公司监管要求。**

### 更大的机会

头部的可解释性能力（如自动生成个性化建议而非单纯拒贷）将成为用户运营差异化竞争点。

## 洞察五：PSI监控ROI>10倍，不监控每年坏账率+0.5-1%

> **来源**: 模型稳定性评估专题
> **适用**: 消费金融/风控模型管理场景

### 核心洞察

不监控的模型，每年效果下降20-30%，坏账率上升0.5-1%。在年放款100亿规模下，这意味着数千万到上亿的额外损失。

```
监控投入100-200万/年
    ↓
避免数千万损失
    ↓
ROI超过10倍
```

### 关键指标

| 指标 | 阈值 | 含义 |
|:---|:---:|:---|
| PSI<0.1 | ✅ 非常稳定 | 继续使用 |
| PSI 0.1-0.25 | ⚠️ 一般稳定 | 关注分析 |
| PSI>0.25 | ❌ 严重不稳定 | 必须修复 |
| PSI>0.5 | 🔴 完全失效 | 必须下线切换 |

| 指标 | 阈值 | 含义 |
|:---|:---:|:---|
| KS<0.2 | 🔴 基本失效 | 必须替换 |
| KS下降>15% | ❌ 明显下降 | 需要修复 |

### 根因分布

| 根因 | 占比 | 说明 |
|:---|:---:|:---|
| **数据源漂移** | 80% | 接口变更、服务不稳定 |
| **客群变化** | 15% | 推广渠道、季节性因素 |
| **特征逻辑错误** | 5% | 计算bug、ETL失败 |

### 行动建议
- **每天**：检查输入PSI均值、输出PSI
- **每周**：检查KS变化
- **必须有**：备用模型，效果稳定可用，定期验证

## 洞察六：OpenSpec管What，Superpowers管How

> **来源**: OpenSpec + Superpowers专题
> **适用**: AI编程/工程化场景

### 核心洞察

```
OpenSpec（上游）    →  管「要做什么」（What）
    ↓                      ↓
需求规约（结构化Markdown）
    ↓                      ↓
Superpowers（下游）  →  管「怎么做」（How）
    ↓                      ↓
标准化工程流程（7个Skill）
    ↓                      ↓
可运行代码
```

**关键解耦**：通过文件系统中的Markdown文件完成握手，不依赖API调用或网络通信。

### AI时代组织重构

| 变化 | 说明 |
|:---|:---|
| **角色坍缩** | 前后端的技能分工在消亡，但思维分工还在 |
| **3人组→2人组** | 架构师 + AI工具操作者，可能替代传统前端+后端+架构师三人组 |
| **核心资产转移** | 从"代码"到"规约"——能定义清楚问题的人永远比只会执行的人稀缺 |

### 7个Superpowers核心工作流

| 工作流 | 做什么 |
|:---|:---|
| brainstorming | 苏格拉底式对话，写代码之前先理清需求 |
| using-git-worktrees | 创建隔离的Git工作空间，并行开发不冲突 |
| writing-plans | 把工作拆成2-5分钟可完成的小任务 |
| subagent-driven-development | 用子代理逐任务执行 |
| test-driven-development | 强制RED-GREEN-REFACTOR循环 |
| requesting-code-review | 自动触发代码审查 |
| finishing-a-development-branch | 完成分支，验证测试 |

## 洞察七：Skill评估需要Gen4混合架构

> **来源**: OpenAI Skill评估专题
> **适用**: 企业AI/Agent/Skill管理场景

### 范式转移

| 代际 | 特征 | 代表技术 | 局限 |
|:---:|:---|:---|:---|
| Gen 1 | 规则匹配 | Regex/Schema验证 | 无法理解语义 |
| Gen 2 | 指标计算 | BLEU/ROUGE/F1 | 忽视上下文 |
| Gen 3 | 模型评判 | LLM-as-a-Judge | 成本高、有bias |
| **Gen 4** | **混合架构** | **规则+模型+人工** | **复杂度高** |

### 四维评分模型（Gen4）

| 维度 | 权重 | 说明 |
|:---|:---:|:---|
| **准确性** | 40% | 事实正确、数据准确 |
| **完整性** | 25% | 覆盖全面、无遗漏 |
| **合规性** | 20% | 符合规范、无风险 |
| **表达性** | 15% | 清晰易懂、结构合理 |

### 混合架构流水线

```
Phase 1: 规则层（30%）→ 低成本过滤
Phase 2: 模型层（50%）→ 语义评估
Phase 3: 人工层（20%）→ 校准反馈
```
