## 1. 产品总览

**企业级元数据的统一管理平台，涵盖数据血缘、成本分摊、技术和业务元信息的全链路治理。**

| 维度 | 描述 |
|:---|:---|
| **目标用户** | 数据工程师、数据管理员、数据使用者、数据架构师 |
| **核心痛点** | 元数据分散难以统一、血缘关系不清晰、数据成本不透明 |
| **解决方案** | 统一元数据中心 + 自动血缘采集 + 成本分析 + 全链路追踪 |

## 2. 功能结构（按模块归档）

### 2.1 FEATURE-DMT_METADATA_CATALOG（统一元数据目录）

#### 功能入口 & 作用

**入口位置**：数据管理 → 元数据中心

**核心作用**：
- 统一管理所有数据资产的元信息
- 提供多维度元数据检索
- 展示元数据详情和关联关系
- 支持元数据版本对比

#### 核心逻辑

```
元数据目录结构：

数据源
├── MySQL
│   ├── 业务库
│   │   ├── ods_orders（订单原始表）
│   │   ├── dwd_user_info（用户维度表）
│   │   └── dws_sales_daily（日销售汇总表）
│   └── 配置库
├── ClickHouse
│   └── 分析库
│       ├── dml_report_sales（月报数据）
│       └── dml_dashboard_xxx（仪表盘数据）
└── Hive
    └── 数据仓库
        ├── ods/（原始层）
        ├── dwd/（维度层）
        └── dws/（汇总层）
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `catalog.show_technical` | 是否展示技术字段 | `true` |
| `catalog.default_page_size` | 默认分页大小 | `20` |
| `catalog.cache_ttl` | 缓存有效期（秒）| `300` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **元数据陈旧** | 实际表结构变更但目录未更新 | 自动同步机制 + 变更提醒 |
| **信息不完整** | 字段缺少业务注释难以理解 | 建立注释必填机制、批量补充工具 |
| **分类混乱** | 元数据分类标准不统一 | 统一分类规范、定期治理 |
| **搜索不准确** | 搜索结果不相关 | 优化搜索算法、增加同义词 |

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 数据源管理 | 上游 | 元数据来源 |
| 血缘管理 | 关联 | 血缘关系展示 |

### 2.2 FEATURE-DMT_LINEAGE_MANAGE（数据血缘管理）

#### 功能入口 & 作用

**入口位置**：数据管理 → 血缘中心

**核心作用**：
- 自动采集数据血缘关系
- 展示多层级血缘图谱
- 支持影响分析和回溯分析
- 血缘变更监控和告警

#### 核心逻辑

```
血缘采集流程：

Step 1: 数据源识别
  - 识别 ETL 任务（Spark SQL、DataX、Shell 等）
  - 解析 SQL 中的表引用
  - 识别调度依赖关系

Step 2: 血缘解析
  - 解析输入表和输出表
  - 解析字段级血缘（Select A.col1 as B.col1）
  - 构建节点和边关系

Step 3: 血缘存储
  - 存储为图结构（Neo4j）
  - 支持多层级展开
  - 支持横向和纵向追溯

Step 4: 血缘展示
  - 图形化血缘图
  - 表格化血缘列表
  - 支持导出和分享
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `lineage.max_depth` | 最大血缘层级 | `10` |
| `lineage.auto_sync` | 自动同步开关 | `true` |
| `lineage.field_level` | 是否采集字段级血缘 | `false` |
| `lineage.sync_interval` | 同步间隔（分钟）| `60` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **血缘缺失** | 复杂 ETL 无法解析血缘 | 标准化 ETL 写法、提供手动维护入口 |
| **血缘环** | 循环依赖导致死循环 | 增加环检测、告警通知 |
| **性能问题** | 深层血缘查询慢 | 缓存预热、分层加载 |
| **误解析** | 注释中的表名被误识别为血缘 | 优化解析规则、忽略注释 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 元数据目录 | 依赖 | 血缘节点信息 |
| 调度中心 | 依赖 | 任务依赖关系 |
| Neo4j | 依赖 | 血缘存储 |

### 2.3 FEATURE-DMT_COST_ALLOCATION（成本分摊）

#### 功能入口 & 作用

**入口位置**：数据管理 → 成本中心

**核心作用**：
- 统计各数据资产的存储和计算成本
- 按部门、项目、数据源维度分摊
- 成本趋势分析和预警
- 优化建议生成

#### 核心逻辑

```
成本分摊流程：

Step 1: 成本采集
  - 存储成本：表大小 × 单价
  - 计算成本：任务耗时 × 单价
  - 传输成本：数据量 × 单价

Step 2: 维度配置
  - 按部门分摊（谁创建谁承担）
  - 按项目分摊（项目维度绑定）
  - 按数据源分摊（物理资源占用）

Step 3: 分摊计算
  - 读取任务执行记录
  - 计算资源占用
  - 按规则分摊到维度

Step 4: 成本展示
  - 成本排行榜
  - 趋势分析图
  - 优化建议
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `cost.storage_price` | 存储单价（元/GB/天）| `0.1` |
| `cost.compute_price` | 计算单价（元/GB）| `0.01` |
| `cost.aggregation_period` | 聚合周期 | `DAILY` |
| `cost.alert_threshold` | 告警阈值（元）| `1000` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **数据归属不清** | 无法确定数据属于哪个部门 | 强制元信息填写部门维度 |
| **任务嵌套** | 子任务成本难以归因 | 按任务树逐层分摊 |
| **价格波动** | 不同时期单价不同 | 记录历史价格、按时段计算 |
| **优化难以执行** | 建议落地困难 | 提供具体优化操作指导 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 调度中心 | 依赖 | 任务执行数据 |
| 数据源管理 | 依赖 | 存储容量数据 |
| 计费系统 | 依赖 | 价格配置 |

## 5. 规则与配置中心

### 5.1 元数据采集规则

| 数据源类型 | 采集方式 | 采集频率 | 说明 |
|:---|:---|:---|:---|
| MySQL | JDBC 连接 | 每日全量 | 支持-schema 级别 |
| ClickHouse | JDBC 连接 | 每日全量 | 采集建表 DDL |
| Hive | HMS API | 实时 + 每日 | 实时监听变更 |
| Kafka | 消费元数据 Topic | 实时 | 采集 Topic 和 Schema |
| API | 主动注册 | 手动/定时 | 提供 OpenAPI |

### 5.2 血缘解析规则

```
血缘解析优先级：

1. Spark SQL
   - 支持 SELECT、INSERT、CREATE TABLE AS SELECT
   - 支持 WITH CTE
   - 字段级血缘可选

2. DataX
   - 解析 reader 和 writer 配置
   - 支持表级血缘

3. Shell 脚本
   - 解析调用的 SQL 文件
   - 支持依赖声明注释

4. 手动维护
   - UI 界面手动添加血缘
   - Excel 批量导入
```

### 5.3 成本分摊规则

```
分摊维度优先级：

1. 直接归属（优先级最高）
   - 表/任务的 owner 部门
   - 项目绑定关系

2. 数据源归属
   - 物理集群归属部门

3. 默认归属
   - 平台运营部门兜底

分摊计算公式：

存储成本 = Σ(表大小 × 存储单价 × 归属系数)
计算成本 = Σ(任务执行时长 × 计算单价 × 归属系数)
```

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: 元数据没有自动同步怎么办？

**排查步骤**：
1. 检查数据源连接是否正常
2. 查看采集任务执行状态
3. 检查同步配置是否启用

**解决方案**：
- 重启采集任务
- 手动触发同步
- 检查数据源权限

#### Q: 血缘显示不完整怎么办？

**排查步骤**：
1. 检查 ETL 任务是否被采集
2. 查看血缘解析日志
3. 确认 SQL 语法是否支持

**解决方案**：
- 手动添加缺失血缘
- 标准化 SQL 写法
- 反馈不支持的语法

#### Q: 成本数据不准确怎么办？

**排查步骤**：
1. 检查任务执行数据是否完整
2. 核对价格配置是否正确
3. 确认分摊规则是否合理

**解决方案**：
- 核对计费明细
- 调整分摊规则
- 联系管理员排查

## 8. 方法论沉淀

### 8.1 元数据治理成熟度模型

```
成熟度等级：

L1 初始级
- 元数据手工维护
- 无统一管理
- 依赖个人经验

L2 流程级
- 关键系统元数据自动采集
- 有流程规范
- 部分标准化

L3 规范级
- 全面自动采集
- 统一分类和标准
- 质量有保障

L4 量化级
- 全面量化管理
- 成本精确分摊
- 优化持续进行

L5 智能级
- AI 辅助元数据管理
- 自动发现问题
- 智能优化建议
```

### 8.2 血缘管理最佳实践

**血缘质量保障**：
```
采集质量：
- 代码规范：SQL 必须符合规范才能解析
- 变更管控：ETL 变更必须关联血缘更新
- 定期巡检：自动检测血缘异常

展示质量：
- 按需加载：深层血缘异步加载
- 可交互：支持缩放、筛选、搜索
- 可追溯：血缘可导出、可分享
```

### 8.3 工具流程

**元数据管理工具链**：
```
采集 → 存储 → 治理 → 应用

工具：
- 采集：Apache Atlas、DataHub、自研采集器
- 存储：MySQL + Elasticsearch + Neo4j
- 治理：元数据质量检测工具
- 应用：血缘图工具、成本分析工具
```
