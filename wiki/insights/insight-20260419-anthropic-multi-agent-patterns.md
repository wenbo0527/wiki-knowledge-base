## 执行摘要 (Executive Summary)

**Anthropic官方发布的《多智能体协作模式》是Agent工程领域的里程碑式文档**，系统性地总结了5种多智能体架构及其适用场景，为正在构建或规划多Agent系统的团队提供了**权威的架构选型指南**。

**核心观点**:
- **从最简单模式开始**: 不是按智能体能干什么活来分工，而是按它们需要什么上下文来分工
- **观察瓶颈再进化**: 先用编排器跑起来，仔细观察哪里卡脖子了，再向其他模式进化
- **混合架构是常态**: 工业级系统往往是混合流派，主干流程用编排器，深度协作环节切成共享状态

**五种模式速查**:

| 模式 | 核心特征 | 最佳场景 | 致命局限 |
|:---|:---|:---|:---|
| **生成器-验证器** | 生成→验证→循环 | 客服回复、代码审查 | 循环卡死 |
| **编排器-子智能体** | 组长派活、短任务 | 独立子任务清晰 | 组长瓶颈 |
| **智能体团队** | 并行长期任务 | 大规模并行持久战 | 缺乏协调 |
| **消息总线** | 事件驱动、动态扩展 | 流水线、生态扩展 | 调试困难 |
| **共享状态** | 高度协作、实时互通 | 研究型任务 | 无限循环 |

## 1. 模式详解

### 1.1 模式一：生成器与验证器 (Generator-Verifier)

**工作机制**:
```
生成器接到任务 → 输出初稿 → 验证器评估
    ↑________________________________↓
         打回修改 ← 不通过 ← 检查标准
              通过 → 输出终稿
```

**核心特点**:
- **循环迭代**: 生成→验证→修改，直到满足标准或达到最大重试次数
- **明确标准**: 验证器必须基于清晰、可量化的评估标准
- **质量优先**: 适合输出错误代价远高于计算成本的场景

**最佳场景**:
- **客服邮件回复系统**: 生成器起草回复，验证器核对知识库、评估语气、检查完整性
- **代码生成与测试**: 生成器写代码，验证器运行测试、检查覆盖率、评估复杂度
- **事实核查系统**: 生成器生成内容，验证器交叉验证多个来源、标注置信度
- **合规性审查**: 生成器生成合同/政策文档，验证器检查法律条款、合规要求

**致命局限**:

| 问题 | 表现 | 解决方案 |
|:---|:---|:---|
| **标准模糊** | 验证器只检查"内容好不好"，缺乏具体标准 | 定义详细的检查清单和评分规则 |
| **循环卡死** | 生成器怎么都改不到验证器想要的程度 | 设置最大迭代次数 + 人工降级方案 |
| **技能重叠** | 评估创意的难度和想出一个好创意一样高 | 不适合创意类任务，改用其他模式 |

**代码示例**:

```python
from deerflow.patterns import GeneratorVerifierPattern

# 初始化生成器-验证器模式
customer_service = GeneratorVerifierPattern(
    generator_config={
        "model": "gpt-4o",
        "temperature": 0.7,
        "system_prompt": """你是客服助手，根据知识库信息起草专业、友好的客户回复。
要求：
- 语气亲切专业
- 准确回答客户问题
- 提及相关产品信息"""
    },
    verifier_config={
        "model": "claude-3-opus",
        "temperature": 0.1,
        "system_prompt": """你是客服质量检查员，严格评估客服回复质量。
检查清单：
[准确性] 回答是否准确（是/否，置信度0-100）
[完整性] 是否回答了所有问题（是/否，遗漏点）
[语气] 是否符合品牌调性（评分1-5，说明）
[安全] 是否有敏感信息泄露（是/否，具体问题）

如果任一项不通过，详细说明问题并要求修改。"""
    },
    max_iterations=5,
    fallback_strategy="human_review"  # 超过最大迭代转人工
)

# 使用
result = await customer_service.run(
    query="我买的手机充电很慢，是不是质量问题？",
    context={
        "product": "XPhone Pro",
        "purchase_date": "2026-01-15",
        "knowledge_base": [...]  # 产品FAQ
    }
)

print(result.final_output)  # 最终审核通过的回复
print(result.iterations)    # 实际迭代次数
print(result.quality_score) # 质量评分
```

### 1.2 模式二：编排器与子智能体 (Orchestrator-Subagents)

**工作机制**:
```
编排器接到任务 → 分析并拆解 → 分发给子智能体
    ↑________________________________↓
         接收结果 ← 子智能体处理 ← 分配子任务
              汇总整合 → 输出终稿
```

**核心特点**:
- **层级制**: 一个智能体当组长负责统筹规划和汇总结果，其他子智能体各自领走特定任务
- **清晰边界**: 子任务之间边界分明，相互独立
- **短平快**: 子任务生命周期短，完成后即销毁

**最佳场景**:
- **代码审查系统**: 安全漏洞检查、测试覆盖率、代码风格、架构一致性分别由不同子Agent处理
- **数据分析流程**: 数据清洗、特征工程、模型训练、结果可视化分别由不同子Agent处理
- **内容审核流程**: 涉黄检测、涉暴检测、广告检测、版权检测分别由不同子Agent处理
- **文档生成流程**: 大纲生成、内容撰写、格式排版、质量检查分别由不同子Agent处理

**与智能体团队模式的区别**:

| 维度 | 编排器-子智能体 | 智能体团队 |
|:---|:---|:---|
| **记忆保留** | 子任务完成后即销毁 | 长期在线，保留上下文 |
| **适用任务** | 短平快、输出明确 | 长期、多步骤、需积累 |
| **典型场景** | 代码审查、单次分析 | 代码迁移、长期监控 |
| **编排复杂度** | 低（组长统一调度） | 中（需协调员派活收作业） |

**致命局限**:

| 问题 | 表现 | 解决方案 |
|:---|:---|:---|
| **信息瓶颈** | 关键信息必须先传给组长再转发，细节丢失 | 关键信息直接广播，或使用共享状态 |
| **顺序执行** | 默认子智能体排队执行，无法并行 | 显式编写并行执行逻辑 |
| **上下文丢失** | 子智能体无法继承父Agent的完整上下文 | 设计良好的上下文传递机制 |

**代码示例**:

```python
from deerflow.patterns import OrchestratorSubagentPattern

# 初始化编排器-子智能体模式
code_review = OrchestratorSubagentPattern(
    orchestrator_config={
        "model": "claude-3-opus",
        "system_prompt": """你是代码审查协调员，负责分配审查任务并整合结果。
任务分配原则：
1. 安全审查：涉及用户输入、数据库操作、网络请求
2. 质量审查：测试覆盖率、代码复杂度、注释完整性
3. 风格审查：命名规范、代码格式、架构一致性

整合要求：
- 按严重程度排序（Critical > Warning > Info）
- 标注每个问题的文件位置和行号
- 提供具体的修改建议
"""
    },
    subagent_configs=[
        {
            "name": "security_reviewer",
            "model": "claude-3-sonnet",
            "system_prompt": """你是安全审查专家，专注于发现安全漏洞。
检查清单：
- [ ] SQL注入风险
- [ ] XSS攻击向量
- [ ] 敏感数据泄露
- [ ] 不安全的反序列化
- [ ] 权限绕过漏洞

输出格式：
[CRITICAL/WARNING/INFO] 文件:行号 - 问题描述 - 修复建议"""
        },
        {
            "name": "quality_reviewer",
            "model": "claude-3-sonnet",
            "system_prompt": """你是代码质量专家，专注于测试和架构。
检查清单：
- [ ] 单元测试覆盖率
- [ ] 集成测试完整性
- [ ] 代码复杂度（圈复杂度）
- [ ] 注释和文档
- [ ] 设计模式应用

输出格式：
[CRITICAL/WARNING/INFO] 文件:行号 - 问题描述 - 改进建议"""
        },
        {
            "name": "style_reviewer",
            "model": "gpt-4o",
            "system_prompt": """你是代码风格专家，专注于命名和格式。
检查清单：
- [ ] 变量命名规范
- [ ] 函数命名一致性
- [ ] 代码格式（缩进、空格）
- [ ] 导入排序
- [ ] 类型注解

输出格式：
[WARNING/INFO] 文件:行号 - 问题描述 - 规范参考"""
        }
    ],
    execution_mode="parallel",  # 并行执行所有子审查
    aggregation_strategy="merge_and_rank"  # 合并结果并按严重程度排序
)

# 使用
result = await code_review.run(
    pr_details={
        "pr_number": 1234,
        "files_changed": [
            "src/auth/login.py",
            "src/api/users.py",
            "tests/test_auth.py"
        ],
        "diff": "..."  # PR的完整diff
    }
)

print(result.combined_report)  # 整合后的审查报告
print(result.critical_issues)  # 严重问题列表
print(result.suggestions)      # 改进建议
```

## 7. 架构选型决策树

基于Anthropic官方指南和DeerFlow实践，我们总结出以下决策树：

```
开始构建多Agent系统
    ↓
任务是否对质量要求极高且标准明确？
    ├── 是 → 使用【生成器-验证器】模式
    │         例：客服回复、代码审查、事实核查
    │
    └── 否 → 子任务是否清晰且边界分明？
              ├── 是 → 使用【编排器-子智能体】模式
              │         例：代码审查、数据分析、内容审核
              │         ⚠️ 观察：组长是否成为瓶颈？
              │
              └── 否 → 是否需要并行且长期运行？
                        ├── 是 → 使用【智能体团队】模式
                        │         例：代码迁移、长期监控
                        │         ⚠️ 观察：是否需要更多协调？
                        │
                        └── 否 → 工作流是否事件驱动且会扩展？
                                  ├── 是 → 使用【消息总线】模式
                                  │         例：安全运营、流水线
                                  │         ⚠️ 观察：调试是否困难？
                                  │
                                  └── 是 → 是否需要高度协作实时互通？
                                            ├── 是 → 使用【共享状态】模式
                                            │         例：科研综述、知识挖掘
                                            │         ⚠️ 设置强终止条件防循环
                                            │
                                            └── 不确定 → 从【编排器-子智能体】开始
                                                      观察瓶颈，逐步进化
```

## 8. 总结

Anthropic官方发布的《多智能体协作模式》为Agent工程领域提供了**权威的架构选型指南**，系统性地总结了5种多智能体架构及其适用场景。

**核心启示**:

3. **混合架构是常态**: 工业级系统往往是混合流派，主干流程用编排器，深度协作环节切成共享状态

**五种模式速查**:

| 模式 | 核心特征 | 最佳场景 | 致命局限 |
|:---|:---|:---|:---|
| **生成器-验证器** | 生成→验证→循环 | 客服回复、代码审查 | 循环卡死 |
| **编排器-子智能体** | 组长派活、短任务 | 独立子任务清晰 | 组长瓶颈 |

**给新手的建议**:
