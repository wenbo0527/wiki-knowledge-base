## 二、容器化与Kubernetes实践

### 2.1 金融容器化架构

```yaml
# Kubernetes集群架构示例
apiVersion: v1
kind: Namespace
metadata:
  name: fintech-prod
  labels:
    environment: production
    compliance: PCI-DSS
---
apiVersion: v1
kind: ResourceQuota
metadata:
  name: prod-quota
  namespace: fintech-prod
spec:
  hard:
    pods: "500"
    requests.cpu: "200"
    requests.memory: 500Gi
    limits.cpu: "400"
    limits.memory: 1Ti
---
apiVersion: v1
kind: LimitRange
metadata:
  name: prod-limits
  namespace: fintech-prod
spec:
  limits:
  - max:
      cpu: "32"
      memory: 64Gi
    min:
      cpu: "100m"
      memory: 128Mi
    default:
      cpu: 500m
      memory: 512Mi
    defaultRequest:
      cpu: 200m
      memory: 256Mi
    type: Container
```

### 2.2 金融级容器平台要求

| 维度 | 要求 | 解决方案 |
|------|------|---------|
| **高可用** | 99.99% SLA | 多AZ部署、Pod高可用 |
| **安全合规** | PCI-DSS、等保三级 | 网络策略、安全扫描 |
| **性能隔离** | 金融交易低延迟 | 资源配额、优先级调度 |
| **灾备** | RPO<1min, RTO<5min | 跨Region复制、快速恢复 |
| **可观测** | 全链路追踪 | Jaeger、Prometheus、Grafana |

### 2.3 实际应用案例

**案例1：招商银行云原生平台**
- 规模：5000+容器，200+微服务
- 架构：双Region多AZ，Kubernetes集群
- 成效：部署频率从每周1次提升到每天50次

**案例2：中国平安容器云**
- 规模：10000+容器，300+微服务
- 架构：混合云，跨云编排
- 成效：资源利用率提升60%，成本降低35%

### 3.2 Istio在金融场景的应用

```yaml
# 金金融交易服务的流量管理示例
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: payment-service
  namespace: fintech-prod
spec:
  hosts:
  - payment-service
  http:
  - match:
    - headers:
        x-risk-level:
          exact: "high"
    route:
    - destination:
        host: payment-service-v2    # 高风险走新版本
        subset: stable
      weight: 100
  - route:
    - destination:
        host: payment-service-v1    # 正常走稳定版本
        subset: stable
      weight: 90
    - destination:
        host: payment-service-v2
        subset: canary
      weight: 10                    # 10%金丝雀
---
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: payment-service
spec:
  host: payment-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        h2UpgradePolicy: UPGRADE
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
```
