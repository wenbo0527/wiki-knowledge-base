## 1. 产品总览

### 1.1 一句话定位
**让业务人员无需编写 SQL，通过可视化界面拖拽配置，即可快速完成数据查询和分析。**

## 2. 功能结构（按模块归档）

### 2.1 FEATURE-DEX_QUERY_BUILDER（可视化查询构建器）

#### 功能入口 & 作用

**入口位置**：数据探索 → 即席查询 → 新建查询

**核心作用**：
- 提供直观的拖拽式查询配置界面
- 支持多数据源联合查询
- 自动生成 SQL 语句并优化
- 提供查询条件智能推荐

#### 核心逻辑

```
可视化查询构建流程：

Step 1: 选择数据源和表
  - 数据源列表（MySQL、ClickHouse、Hive 等）
  - 表分类浏览（业务表、维度表、汇总表）
  - 表收藏和最近使用
  - 表元信息查看（字段、注释）

Step 2: 字段选择
  - 勾选需要查询的字段
  - 支持全字段快捷选择
  - 字段别名设置
  - 聚合计算配置（求和、计数、平均）

Step 3: 条件配置
  - 拖拽添加筛选条件
  - 条件组合（AND/OR）
  - 运算符选择（等于、包含、范围等）
  - 日期条件快捷设置

Step 4: 分组和排序
  - 分组字段选择
  - 聚合函数配置
  - 排序规则设置
  - Top N 限制

Step 5: SQL 预览和执行
  - 实时生成 SQL 语句
  - SQL 语法高亮
  - 手动编辑和调整
  - 执行查询
```

#### 关键配置项

| 配置项 | 说明 | 示例 |
|:---|:---|:---|
| `querybuilder.default_limit` | 默认查询条数限制 | `1000` |
| `querybuilder.max_export_rows` | 最大导出行数 | `100000` |
| `querybuilder.auto_complete` | SQL 自动补全开关 | `true` |
| `querybuilder.syntax_check` | 语法检查开关 | `true` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **跨库查询性能差** | 多数据源 JOIN 性能问题 | 提供数据源关联配置、优化执行计划 |
| **条件组合复杂** | 多条件组合后 SQL 难以理解 | 提供条件预览、语法解释 |
| **字段选择过多** | 全量字段导致结果集过大 | 默认限制字段数量、提示用户筛选 |
| **日期条件歧义** | "上个月"等语义条件配置模糊 | 提供标准日期快捷选项 |
| **聚合查询慢** | 大表 COUNT DISTINCT 等操作慢 | 提供近似算法选项、结果缓存 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| EPIC-DMT_METADATA_MANAGE | 上游 | 获取表和字段元信息 |
| EPIC-DMT_DATA_QUALITY | 上游 | 获取数据质量信息辅助查询 |
| 数据权限中心 | 依赖 | 查询权限校验和数据脱敏 |
| 调度引擎 | 下游 | 定时执行查询任务 |

### 2.2 FEATURE-DEX_NATURAL_LANGUAGE_QUERY（自然语言查询）

#### 功能入口 & 作用

**入口位置**：数据探索 → 自然语言查询

**核心作用**：
- 支持用户用自然语言描述查询需求
- 自动转换为 SQL 或可视化配置
- 提供查询结果解读和建议
- 支持多轮对话式查询优化

#### 核心逻辑

```
自然语言处理流程：

Step 1: 意图识别
  - 查询类型识别（简单查询、聚合查询、对比分析）
  - 时间范围提取
  - 筛选条件提取
  - 聚合维度提取

Step 2: 实体链接
  - 表名识别（"用户" → user_table）
  - 字段识别（"销售额" → order_amount）
  - 指标识别（"日活" → daily_active_users）
  - 时间实体识别（"上周" → 2026-04-01 至 2026-04-07）

Step 3: SQL 生成
  - 查询语句构建
  - 条件转换
  - 聚合函数映射
  - 排序和限制处理

Step 4: 结果解读
  - 自然语言结果描述
  - 关键发现提取
  - 追问建议生成
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:---|
| `nlq.embedding_model` | 向量模型 | `text2vec-base-chinese` |
| `nlq.confidence_threshold` | 意图置信度阈值 | `0.7` |
| `nlq.max_suggestions` | 最大追问建议数 | `3` |
| `nlq.sql_fallback` | 识别失败时回退到可视化 | `true` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **意图识别不准** | "销售好的"被理解为销售额高 | 增加领域词典、用户反馈优化 |
| **表字段映射错误** | "用户"映射到错误的表 | 提供表/字段推荐确认环节 |
| **复杂查询无法理解** | 超过 3 层嵌套的查询无法生成 | 引导用户拆分为简单查询 |
| **方言表达差异** | 不同用户对同一概念表达不同 | 收集用户反馈持续优化模型 |
| **识别延迟高** | LLM 推理时间过长 | 简化模型、增加缓存、异步处理 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| FEATURE-DEX_QUERY_BUILDER | 依赖 | 生成的 SQL 交由查询构建器执行 |
| LLM 服务 | 依赖 | 意图识别和 SQL 生成 |
| 元数据中心 | 依赖 | 表字段信息获取 |

### 2.3 FEATURE-DEX_QUERY_HISTORY（查询历史管理）

#### 功能入口 & 作用

**入口位置**：数据探索 → 我的查询

**核心作用**：
- 记录用户所有查询操作
- 支持查询结果快速复用
- 查询版本管理和对比
- 查询分享和协作

#### 核心逻辑

```
查询历史管理流程：

Step 1: 查询记录
  - 自动记录每次执行的查询
  - 记录执行时间、耗时、结果数
  - 记录查询参数和条件

Step 2: 历史分类
  - 按时间分类（今天/本周/更早）
  - 按数据源分类
  - 按查询类型分类
  - 自定义收藏夹

Step 3: 查询复用
  - 一键重新执行
  - 修改条件后执行
  - 导出为模板
  - 分享给其他人

Step 4: 查询分析
  - 执行频率统计
  - 耗时分布分析
  - 热门查询排行
  - 优化建议
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---:|
| `history.retention_days` | 历史保留天数 | `90` |
| `history.max_records` | 最大记录数 | `1000` |
| `history.auto_save` | 自动保存草稿 | `true` |
| `history.share_enabled` | 分享功能开关 | `true` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **历史数据过多** | 查询记录太多难以查找 | 提供搜索、筛选、批量删除 |
| **历史数据泄露** | 敏感查询条件被其他人看到 | 区分私有/共享历史、权限控制 |
| **查询无法复现** | 底层表结构变更导致历史查询失败 | 提供查询兼容性检查和升级提示 |
| **存储成本高** | 大量查询语句占用存储 | 定期归档和压缩历史数据 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| FEATURE-DEX_QUERY_BUILDER | 依赖 | 查询历史的来源 |
| 用户中心 | 依赖 | 用户身份识别 |

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: 查询超时怎么办？

**排查步骤**：
1. 检查查询条件是否过于宽泛
2. 查看是否涉及大表全表扫描
3. 检查数据库连接是否正常

**解决方案**：
- 添加更精确的筛选条件
- 使用索引字段进行筛选
- 联系管理员优化查询性能

#### Q: 查询结果为空怎么办？

**排查步骤**：
1. 检查筛选条件是否正确
2. 确认数据是否已同步
3. 检查是否有权限访问该数据

**解决方案**：
- 放宽筛选条件
- 等待数据同步后重试
- 申请数据访问权限

#### Q: 自然语言识别错误怎么办？

**排查步骤**：
1. 检查输入是否包含明确的表和字段
2. 确认表述是否清晰无歧义
3. 查看历史识别记录

**解决方案**：
- 使用更明确的表达
- 切换到可视化查询模式
- 提交纠错反馈

## 8. 方法论沉淀

### 8.1 查询性能优化方法论

**查询优化检查清单**：
```
1. 索引使用检查
   - WHERE 条件字段是否有索引
   - JOIN 条件字段是否有索引
   - 避免全表扫描

2. 查询范围检查
   - 是否限定了时间范围
   - 是否使用了 TOP/LIMIT
   - 聚合查询是否必要

3. SQL 质量检查
   - 避免 SELECT *
   - 减少子查询嵌套
   - 优化 JOIN 顺序

4. 执行计划分析
   - 识别慢查询
   - 分析执行路径
   - 优化执行计划
```
