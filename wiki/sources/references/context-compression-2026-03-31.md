# 聊聊各家 AI 产品的上下文压缩方法论

> 来源: 微信公众号 - 三元同学 | 发布: 2026-03-31

---

## 第一性原理：为什么上下文不是越长越好

上下文压缩要解决的核心问题是：

> **在有限的注意力预算内，让模型看到最关键的信息。**

因为 Agent 是自主执行的，一个复杂任务跑下来，可能要几十次工具调用，每次工具结果少则几百 token、多则几万 token。一个文件读取就是几千 token，一个 shell 命令的输出可能上万。

**成本问题**：即使你的模型支持 1M 的上下文，每次推理都喂 1M 进去，token 费用是很恐怖的。做过 Agent 产品的人都知道，这东西的 token 消耗是普通对话的几十倍。不做压缩的话，烧钱速度能让你心跳加速。

所以上下文压缩这个事，不是一个"有了就更好"的优化，而是一个"不做就跑不动"的基础设施。

## Manus 的方法论：三板斧

Manus 把上下文工程拆成了三板斧：**Reduce（缩减）、Offload（卸载）、Isolate（隔离）**。

### 缩减：两阶段逐级加码

**第一阶段是 Compaction**：
Manus 给每个工具调用维护了两个版本：完整版和紧凑版。完整版就是原始工具输出，存在 sandbox 文件系统里；紧凑版只保留一个引用，比如文件路径。随着对话推进，老的工具结果从完整版切换到紧凑版，新的保持完整。

关键的设计理念——**所有压缩都是可恢复的**。即使网页内容移出上下文了，URL 还在，拉回来就行。

**第二阶段是 Summarization**：
当大部分老结果都 compact 完了，窗口还是太大，就切换到轨迹摘要。但 Manus 不是让模型自由发挥写个总结，而是用**预定义的 schema** 来约束摘要格式。这样 summary 的质量和可靠性会高不少。

### 卸载：把文件系统当"外挂大脑"

上下文窗口有 token 限制、有性能衰减、费钱。但文件系统呢？大小不受限，天然持久化，Agent 可以直接读写。

**Manus 让模型学会了把文件系统当外部记忆用**。信息用完写到文件里，需要的时候再读回来。

工具层面也是这个思路。Manus 只有**不到 20 个**原子化工具函数——Bash、文件系统、代码执行这些基础的。其他功能不绑定成 LLM 的 tool，而是**通过 CLI 暴露在 sandbox 里**，Agent 用 Bash 去调用。这叫做 **Progressive Disclosure**（渐进式披露）。

### 隔离：多 Agent 不是为了分工，是为了分上下文

Manus 的多 Agent 架构有一个特别清醒的认知：

> "人类按角色组织是因为认知限制。LLM 不一定有这些限制。"

**子 Agent 的主要目标不是按角色分工（设计师、工程师、PM 那套），而是隔离上下文。**

架构上分三层：Planner 分配任务，Knowledge Manager 决定什么信息要持久化到文件系统，Executor 干具体的活。

## Claude Code 的三层防线

**第一层：Tool Result Clearing**。最轻量的手段——对话深处的工具调用结果，模型不再需要了，直接清掉。几乎没有信息损失，因为那些老结果早就被消化过了。

**第二层：Compaction**。对话快到上限了，让模型读一遍完整历史，生成结构化摘要。这个摘要会保留任务进度（"5/17 个文件改完"）、当前在做什么、已做的决策和原因、未解决的问题。压缩完带着摘要 + 最近 5 个文件继续干活。

**第三层：子 Agent**。复杂任务拆给子 Agent，各自在干净上下文里工作。探索了几万 token 的内容，只返回 1-2K 的摘要。**10-20 倍的压缩比**。

这三层的优先级是递进的——能用轻量手段解决就不上重量级的。

## OpenClaw 的工程细节

OpenClaw 设计了一个叫 **ContextEngine** 的可插拔接口——ingest（摄入）、assemble（组装）、compact（压缩），整个上下文生命周期管理得很清晰。

工具结果截断有个巧妙的设计：如果尾部有错误信息或 JSON 结构，用 **head+tail** 策略截断，中间省略但保留首尾。你想想，shell 命令的报错通常在最后面，直接截尾部就丢了最关键的信息。

分段摘要也做得很细。消息按 token 份额切段，分段生成摘要再合并。有 fallback：某条消息太大导致摘要失败，跳过它标个 `[Large message omitted]`，不至于整个流程挂掉。

**关键参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| 单工具结果上限 | 上下文窗口30% | 硬性限制 |
| 安全边距 | 1.2 (20%) | 补偿中文等多字节字符低估 |
| 最近对话 | 保留3轮不压缩 | 保持连续性 |
| 摘要触发点 | 体量180K左右 | 预留20K给下一轮 |
| 标识符保护 | UUID/路径原文保留 | 摘要指令强制要求 |

## Cursor 的思路

Cursor 的 **Dynamic Context Discovery**——与其压缩，不如一开始就少放东西。工具描述按需加载，大输出写文件，Agent 用 `grep`/`tail` 按需读取。

A/B 测试结果：**token 消耗减少了 46.9%，而且模型表现反而更好了**。

## LangChain 的自主压缩

LangChain 的 Deep Agents SDK 加了一个"自主压缩工具"——不是固定阈值触发，而是 Agent 自己判断"现在是个好时机"然后触发压缩。

| 触发时机 | 说明 |
|----------|------|
| 刚完成一个子任务 | 中间状态已经转化为结果 |
| 从大量数据提取完结论 | 原始数据不再需要 |
| 准备开始消费大量新信息前 | 腾出空间 |

**核心假设**：Agent 比固定规则更懂"现在是不是个好时机"。

## 管理优先级框架

```
优先级从轻到重：
┌─────────────────────────────────────────────────────────────┐
│  1. 管入口 ─── 少放不必要信息 (Cursor按需加载、Manus小工具集)  │
│  2. 管工具 ─── 工具结果最大token源 (截断、compact、清除)       │
│  3. 对话压缩 ─── Compaction / Summarization                 │
│  4. 拆Agent ─── 上下文隔离                                  │
│  5. 文件系统 ─── 外部记忆兜底                               │
└─────────────────────────────────────────────────────────────┘
```

## 工程坑提醒

| 坑 | 说明 |
|-----|------|
| 别在系统提示开头放时间戳 | 精确到秒会把KV Cache命中率干到地板 |
| 保留错误，别擦除 | 移除证据=移除学习机会 |
| 小心few-shot陷阱 | 重复处理类似任务会让模型陷入"节奏" |
| 引入结构化随机性 | 不同序列化模板、替代措辞、格式变化 |

## 核心认知

> **上下文压缩的本质是注意力管理**
> 不是在做数据压缩，是在帮模型"聚焦"——确保模型当前决策所需要的信息在它的注意力范围内。

> **KV Cache 命中率是生产阶段最重要的单一指标**。10 倍的成本差异，不是开玩笑的。

> **Karpathy 定义**：context engineering is the delicate art and science of filling the context window
