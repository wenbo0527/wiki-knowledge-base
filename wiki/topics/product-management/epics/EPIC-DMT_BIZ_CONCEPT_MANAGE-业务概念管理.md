## 1. 产品总览

### 1.1 一句话定位
**统一管理企业业务概念和数据口径，消除数据歧义，实现业务与技术的一致性对齐。**

### 1.2 产品定位 & 核心价值

| 维度 | 描述 |
|:---|:---|
| **目标用户** | 数据分析师、业务人员、数据管理员、数据工程师 |
| **核心痛点** | 同一指标有多个口径、数据定义不统一、业务与技术沟通成本高 |
| **解决方案** | 业务概念中心 + 指标管理 + 口径定义 + 冲突检测 |
| **核心价值** | 口径统一率 100%、数据信任度提升、分析效率提升 50% |

### 1.6 术语词典（本体论级别）

| 术语 | 英文 | 定义 | 示例 |
|:---|:---|:---|:---|
| **业务概念** | Business Concept | 业务领域中具有明确含义的实体或术语 | 用户、订单、商品 |
| **指标** | Metric / KPI | 可量化的业务测量值 | 日活用户数（DAU）、销售额 |
| **口径** | Business Definition | 指标的计算规则和数据范围 | "销售额 = SUM(实付金额)，不含退款" |
| **维度** | Dimension | 分析指标的分类角度 | 时间、地区、渠道 |
| **衍生指标** | Derived Metric | 基于基础指标计算的指标 | 利润率 = 利润 / 销售额 |
| **原子指标** | Atomic Metric | 不可再分的基础指标 | 订单数、支付金额 |
| **指标集市** | Metric Store | 指标的统一管理和查询平台 | 一站式指标查询和获取 |
| **口径冲突** | Definition Conflict | 同一指标存在多个不同口径 | "活跃用户"有两个不同的定义 |

### 2.1 FEATURE-DMT_BIZ_GLOSSARY（业务术语表）

#### 功能入口 & 作用

**入口位置**：数据管理 → 业务概念 → 术语表

**核心作用**：
- 统一管理企业业务术语
- 建立术语标准化定义
- 维护术语间的关系
- 支持多语言和同义词管理

#### 核心逻辑

```
业务术语管理流程：

Step 1: 术语录入
  - 录入标准术语名称
  - 填写业务定义和说明
  - 添加英文名称和同义词
  - 关联相关术语

Step 2: 术语分类
  - 一级分类（业务域）
  - 二级分类（业务模块）
  - 标签管理

Step 3: 术语审核
  - 提交审核申请
  - 业务负责人审批
  - 审核通过后发布

Step 4: 术语应用
  - 术语检索和查询
  - 口径引用
  - 数据字段映射
```

#### 关键配置项

| 配置项 | 说明 | 默认值 |
|:---|:---|:---|
| `glossary.approval_required` | 是否需要审批 | `true` |
| `glossary.synonym_enabled` | 同义词开关 | `true` |
| `glossary.max_hierarchy` | 最大层级深度 | `3` |

#### 常见坑 & 注意点

| 坑点 | 问题描述 | 解决方案 |
|:---|:---|:---|
| **术语定义模糊** | 不同人对同一术语理解不同 | 提供详细定义模板、示例说明 |
| **同义词泛滥** | 大量同义词难以管理 | 限制同义词数量、审核机制 |
| **分类不统一** | 术语归属分类争议 | 建立分类标准、仲裁机制 |
| **英文翻译不一致** | 中英翻译存在多个版本 | 统一翻译标准、版本管理 |

#### 关联页面 / 依赖

| 关联项 | 类型 | 说明 |
|:---|:---:|:---|
| 元数据中心 | 依赖 | 术语元信息存储 |
| 指标管理 | 下游 | 术语被指标引用 |

## 5. 规则与配置中心

### 5.1 指标分类体系

| 一级分类 | 二级分类 | 说明 |
|:---|:---|:---|
| **用户指标** | 活跃、留存、转化 | 用户相关指标 |
| **交易指标** | 订单、支付、退款 | 交易相关指标 |
| **财务指标** | 收入、成本、利润 | 财务相关指标 |
| **流量指标** | UV、PV、访问时长 | 流量相关指标 |
| **产品指标** | 功能使用、体验 | 产品相关指标 |

### 5.2 指标命名规范

```
命名规范：

原子指标：{业务修饰}_{原子指标}
  示例：日活用户数、月付费用户数

衍生指标：{衍生方式}_{原子指标}
  示例：毛利率、环比增长率

汇总指标：{汇总维度}_{原子指标}
  示例：各省份日活用户数、各渠道周订单数

命名原则：
- 使用业务语言，避免技术术语
- 保持简洁，2-6 个字
- 统一时间修饰词（日/周/月/年）
- 避免歧义和重复
```

### 5.3 口径冲突检测规则

```
冲突类型：

1. 完全冲突
   - 同名指标，但 SQL 完全不同
   
2. 部分冲突
   - 同名指标，SQL 大部分相同但有细微差异

3. 语义冲突
   - SQL 相同但业务定义不同

检测方法：
- 指标名称相似度匹配
- SQL 结构相似度分析
- 口径描述语义分析

处理流程：
1. 发现冲突 → 发送告警
2. 协调确认 → 确定保留口径
3. 合并/废弃 → 更新指标
4. 通知下游 → 口径变更同步
```

## 7. 问题库 & FAQ

### 7.1 常见问题排查

#### Q: 找不到需要的指标怎么办？

**排查步骤**：
1. 使用关键词搜索
2. 按分类浏览查找
3. 查看热门指标推荐

**解决方案**：
- 使用更通用的关键词
- 联系数据管理员协助查找
- 申请创建新指标

#### Q: 发现口径冲突怎么办？

**排查步骤**：
1. 查看冲突详情，确认是否真正冲突
2. 联系相关指标负责人
3. 提交口径仲裁申请

**解决方案**：
- 协调统一口径
- 保留争议指标，标注差异
- 下线重复指标

#### Q: 指标数据与预期不符怎么办？

**排查步骤**：
1. 查看指标口径定义
2. 对比数据源实际值
3. 检查是否有数据延迟

**解决方案**：
- 确认口径理解是否正确
- 联系指标负责人排查
- 提交数据质量问题反馈

## 8. 方法论沉淀

### 8.1 指标体系建设方法论

```
指标体系构建步骤：

1. 确定分析框架
   - 商业模式画布
   - 用户旅程地图
   - 业务流程梳理

2. 识别核心指标
   - 北极星指标（唯一核心）
   - 关键结果指标（KR）
   - 运营过程指标

3. 建立指标分层
   - 战略层：CEO/CPO 关注的宏观指标
   - 战术层：业务负责人关注的中观指标
   - 执行层：运营人员关注的微观指标

4. 定义指标口径
   - 明确计算公式
   - 确定数据范围
   - 说明业务含义
   - 提供使用示例
```

### 8.2 口径治理最佳实践

**口径标准化原则**：
```
1. 唯一性：一个指标只能有一个权威口径
2. 明确性：口径定义清晰无歧义
3. 可追溯：能追溯到数据来源
4. 可验证：计算结果可验证
5. 版本化：变更需要版本记录

口径文档模板：
- 指标名称：日活跃用户数（DAU）
- 业务定义：当日有任意行为的去重用户数
- 计算口径：COUNT(DISTINCT user_id) WHERE dt = '${bizdate}'
- 数据范围：2020-01-01 至今
- 业务备注：包含 Web、App、小程序所有端
- 常见问题：Q: 为什么和 A 系统不一致？A: A 系统仅包含 App 端
```

### 8.3 工具流程

**指标管理工具链**：
```
定义 → 审核 → 发布 → 使用 → 监控 → 优化

工具：
- 定义：指标管理平台
- 审核：审批流程引擎
- 发布：指标集市
- 使用：BI 工具、分析平台
- 监控：指标监控大盘
- 优化：使用分析工具
```
